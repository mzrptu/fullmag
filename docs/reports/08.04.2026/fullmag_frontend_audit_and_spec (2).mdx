# Fullmag frontend audit i specyfikacja docelowa

## 0. Kontekst

Ten raport jest **auditem wstępnym** opartym o aktualne publiczne repo oraz bieżące zachowanie widoczne w kodzie. Nie jest to jeszcze audyt „ostateczny plik po pliku” według naszego protokołu, bo do tego potrzebuję od Ciebie **aktualny zip frontendu albo konkretne pliki po refaktorze**, które faktycznie chcesz audytować. Publiczne repo mówi dużo, ale nie musi być 1:1 tym, co masz lokalnie. Taka jest uroda świata i branch driftu.

Ten dokument robi trzy rzeczy naraz:

1. porównuje aktualną implementację do wcześniejszego planu refaktoru,
2. wskazuje co nadal jest źle semantycznie,
3. definiuje **docelowy kontrakt UI** dla ribbonów, drzewa modelu, inspectorów i focus/isolate w 3D.

---

## 1. Executive verdict

### 1.1. Co zostało zrobione

W aktualnej implementacji widać realny postęp:

- są już bardziej rozbudowane grupy akcji w ribbonie,
- są akcje study-stage typu relax / run / eigen / hysteresis,
- istnieją podstawowe mechanizmy `Focus`, `Context`, `Isolate` w widoku 3D,
- istnieje rozdzielenie na model tree + contextual settings panel,
- pojawiły się bardziej konkretne panele dla Universe, Mesh, Study, Results.

### 1.2. Co nie zostało zrealizowane z wcześniejszego planu

Najważniejsze: **duża refaktoryzacja architektury informacji nie została jeszcze domknięta**.

Nadal obecne są stare problemy:

- nadal istnieje **3-warstwowa góra** (`TopHeader + StageBar + RibbonBar`),
- nadal istnieje osobna oś `build / study / analyze`,
- ribbon nadal jest **stage-dependent**, a nie **workflow-driven**,
- `Geometry`, `Materials` i `Physics` nadal nie są prawdziwie osobnymi domenami autora modelu,
- kliknięcie tabów ribbonu nadal potrafi mutować selekcję drzewa,
- drzewo modelu nadal nie odzwierciedla poprawnie semantyki obiektu, materiału, physics stack i study,
- inspector nadal pokazuje sporo informacji zbędnych, zdublowanych lub nieadekwatnych,
- selection/focus/highlight w 3D istnieją, ale nie są jeszcze **spójnym kontraktem całego UI**.

### 1.3. Krótki werdykt

**Nie, refaktor nie został jeszcze zrealizowany zgodnie z pełnym planem.**

To nie jest katastrofa. To jest raczej stan „połowa rusztowania stoi, ale budynek dalej ma stare schody w środku”. Czyli klasyczne przejście między „już coś działa” a „to wreszcie ma sens”.

## 2. Audit delta względem planu

## 2.1. Górna nawigacja

### Stan obecny

Aktualny shell nadal logicznie składa się z trzech warstw:

- `TopHeader`
- `StageBar`
- `RibbonBar`

To oznacza, że użytkownik dalej musi rozumieć:

1. menu aplikacji,
2. tryb/perspektywę (`Model Builder / Study / Analyze`),
3. dodatkowy zestaw tabów ribbonu zależny od powyższego.

### Ocena

To jest nadal za ciężkie poznawczo i zbyt mało semantyczne.

### Docelowy stan

Na górze powinny zostać **tylko dwa poziomy**:

1. **App Bar**
2. **Ribbon**

#### App Bar

App Bar powinien zawierać:

- menu aplikacji,
- quick actions: save / undo / redo / sync script,
- nazwę projektu + status dirty/synced,
- backend status,
- runtime status,
- solver controls: `Relax`, `Run`, `Pause`, `Stop`,
- command search,
- ewentualnie przełącznik layoutu/perspektywy, ale nie jako osobny poziom workflow.

#### Ribbon

Ribbon powinien być **jedyną główną nawigacją roboczą**.

Docelowe core tabs:

- `Home`
- `Definitions`
- `Geometry`
- `Materials`
- `Physics`
- `Mesh`
- `Study`
- `Results`
- `Automation`

Dodatkowo contextual tab sets:

- `Selected Object`
- `Selected Material`
- `Selected Interaction`
- `Selected Mesh Part`
- `Selected Stage`
- `Selected Plot`
- `Universe`

### Wniosek implementacyjny

`StageBar` powinien **zniknąć z widocznej, głównej nawigacji**. Jeśli zostaje jakaś perspektywa layoutu, ma być niskiego rzędu i nie może sterować ontologią modelu.

## 2.2. Ribbon semantics

### Stan obecny

Tabs ribbonu są nadal powiązane z `workspaceMode`, a nie z logiczną strukturą modelu.

Obecnie mamy zestawy typu:

- build: `Home / Geometry / Materials / Physics / Mesh / StudyBuilder`
- study: `Home / LiveView / Runtime / Charts / Diagnostics`
- analyze: `Home / Spectrum / Modes / Compare / Export`

Dodatkowo kilka tabów nadal mapuje się na te same group builders, więc część tabów jest semantycznie pozorna.

### Co jest złe

1. `LiveView`, `Runtime`, `Charts`, `Diagnostics` nie są dobrymi top-level workflow tabs.
2. `Spectrum`, `Modes`, `Compare`, `Export` to raczej widoki wyników albo operacje, nie domeny modelowania.
3. `Geometry`, `Materials`, `Physics` nadal nie są w pełni niezależnymi warstwami authoringu.
4. Klikanie tabów ribbonu nadal nie powinno zmieniać selekcji drzewa „za plecami użytkownika”.

### Docelowa zasada

Ribbon ma robić dwie rzeczy:

- pokazywać komendy relewantne do aktualnego kontekstu,
- przełączać **zakres narzędzi**, ale nie semantykę selekcji.

### Twarde reguły

1. **Tree selection jest źródłem kontekstu.**
2. Ribbon dobiera się do selekcji.
3. Kliknięcie ribbon tab **nie wybiera samo z siebie innego node w drzewie**, chyba że użytkownik użyje jawnej komendy `Go to ...`.
4. Każda ikonka ribbonu musi odpowiadać **realnemu commandowi** z command registry.
5. Każdy command musi mieć:
   - `id`
   - `label`
   - `scope`
   - `selection requirements`
   - `effect on model`
   - `backend binding`
   - `undo/redo contract`

### Stan obecny

Drzewo jest nadal zbyt mocno zorganizowane wokół jednej gałęzi `study-root / Simulation`, a potem pod to wpadają różne warstwy: Universe, Study Domain Mesh, Objects, Physics, Study, Outputs.

To jest bliżej „run tree” niż „model builder tree”.

1. Brakuje jawnego `Definitions`.

## 2.4. Inspector / Settings Panel

### Stan obecny

Routing w `SettingsPanel` jest nadal mocno ad hoc. Dla niektórych node’ów pokazuje kombinacje paneli, które nie mają sensu semantycznego.

Najgorszy przykład:

- kliknięcie `obj-*` nadal potrafi dawać jednocześnie `GeometryPanel` + `ObjectMeshPanel`.

Do tego dochodzi jeszcze globalny blok typu `Capability Coverage`, który nie powinien być renderowany prawie wszędzie.

### Docelowa zasada

Każdy node w drzewie ma mieć **jeden dominujący inspector purpose**.

Nie „trochę geometrii, trochę meshu, trochę statusów”, tylko:

- Overview
- Editor
- Runtime state
- Results view

Zależnie od typu węzła.

### Reguły inspectorów

1. **Object root** pokazuje overview obiektu, a nie geometrię + mesh.
2. **Geometry node** pokazuje tylko geometrię.
3. **Material node** pokazuje tylko materiał.
4. **Magnetic Parameters root** pokazuje stack interakcji + add/remove/reorder + summary.
5. **Interaction child node** pokazuje tylko formularz konkretnego oddziaływania.
6. **Object Mesh node** pokazuje tylko parametry meshu i statystyki preview/build.
7. **Study stage node** pokazuje overview stage.
8. **Results nodes** pokazują tylko wyniki, wykresy, eksport albo I/O.
9. `Capability Coverage` nie może być globalnym blokiem w połowie aplikacji.

## 2.5. 3D focus / isolate / highlight

### Stan obecny

Jest już część mechaniki:

- `selectedObjectId`
- `selectedEntityId`
- `focusedEntityId`
- `objectViewMode`
- `focusObjectRequest`

Są też przyciski `Focus`, `Context`, `Isolate`.

To znaczy, że fundament jest. Ale zachowanie nadal nie jest rozpisane jako jeden spójny kontrakt.

### Docelowy kontrakt zachowania 3D

#### A. Kliknięcie obiektu

- obiekt dostaje stan `selected`
- obiekt jest lekko podświetlony
- ribbon przełącza się na odpowiedni contextual set `Selected Object`
- inspector przechodzi na `Object Overview`
- kamera nie musi się ruszać automatycznie, chyba że to pierwsza selekcja albo obiekt jest poza kadrem

#### B. Kliknięcie child node obiektu

Każdy child node nadal odnosi się do **tego samego owning object**, więc:

- owning object ma być podświetlony,
- jeśli child reprezentuje konkretną warstwę logiczną, highlight ma się różnić.

Przykład:

- `Geometry` -> highlight powierzchni / obrysu geometrii
- `Material` -> highlight całego ciała delikatnym tintem materiałowym
- `Exchange` / `Demag` / `DMI` -> highlight całego ciała + badge/legend overlay w inspectorze
- `Object Mesh` -> przejście na mesh overlay albo mesh-only hybrid view
- `Initial Magnetization` -> przejście na magnetization field preview

#### C. Focus

`Focus` powinien:

- ustawić kamerę na obiekt/region,
- nie zmieniać visibility innych obiektów,
- zachować current context mode.

#### D. Isolate

`Isolate` powinien:

- ukryć lub mocno wygasić wszystkie pozostałe obiekty,
- pokazać tylko aktualny obiekt i powiązane warstwy.

#### E. Context

`Context` powinien:

- zostawić pozostałe obiekty widoczne,
- ale ghosted/desaturated,
- z wyraźnym akcentem na selection owner.

#### F. Hover

Hover powinien być lokalny i tymczasowy:

- outline albo jaśniejszy edge,
- bez zmiany globalnej selekcji,
- z możliwością kliknięcia w obiekt albo jego mesh partition.

## 3. Status wobec poprzedniego planu: checklist

## 3.1. Flatten top navigation

**Status:** NIE

### Co miało być

- AppBar + Ribbon
- brak widocznego StageBar jako osobnej osi

### Co jest

- TopHeader + StageBar + RibbonBar

### Działanie

- usunąć StageBar z głównej góry,
- w store zastąpić `currentStage` czymś w rodzaju `currentPerspective`, ale bez wpływu na semantykę modelu.

## 3.2. Workflow-driven ribbon

**Status:** NIE

### Co miało być

- core tabs zgodne z workflow
- contextual tabs oparte o selekcję

### Co jest

- tabs zależne od `build/study/analyze`
- część tabów to tylko różne etykiety do tych samych group builders

### Działanie

- przepisać `RibbonBar` na command registry + semantic tab registry.

## 3.3. Tabs do not change tree selection

**Status:** NIE

### Co miało być

- tab zmienia narzędzia, nie wybór node’a

### Co jest

- kliknięcia tabów nadal potrafią wymuszać `universe`, `objects`, `study`, `results`, `mesh`

### Działanie

- usunąć side effects `onSelectModelNode(...)` z click handlerów tabów
- wprowadzić osobne jawne komendy `Go to ...`.

## 3.4. Per-object Magnetic Parameters subtree

**Status:** NIE / CZĘŚCIOWO

### Co miało być

- węzeł `Magnetic Parameters` pod obiektem
- child nodes: exchange, demag, dmi, ku itd.

### Co jest

- tree nadal nie pokazuje takiego subtree jako first-class elementu
- physics jest nadal w dużej mierze globalne

### Działanie

- przebudować `ModelTree.tsx`
- dodać node descriptors dla interaction stack.

## 3.5. Proper object inspector

**Status:** NIE

### Co miało być

- object overview + actions + links do child editors

### Co jest

- object root potrafi nadal zwracać geometrię i mesh w jednym panelu

### Działanie

- wprowadzić `ObjectOverviewPanel.tsx`
- przenieść geometrię/material/mesh do child nodes.

## 3.6. Focus/isolate semantics across subtree

**Status:** CZĘŚCIOWO

### Co miało być

- każdy child node obiektu powinien wpływać logicznie na 3D highlight/focus

### Co jest
n
- fundament jest,
- kontrakt zachowania nadal niespójny i nieopisany per node.

### Działanie

- dodać `selection presentation model` między tree a viewportem.

## 6. Tree click -> Ribbon switching contract

To jest ważne i trzeba to naprawić świadomie.

## 6.1. Zasada główna

**Kliknięcie node’a drzewa ma ustawiać aktywny ribbon kontekstowo.**

Ale odwrotnie:

**kliknięcie ribbon tab nie ma samoczynnie zmieniać wybranego node’a**.

## 6.2. Mapowanie

### Root / model nodes

- `Model` -> `Home`
- `Definitions*` -> `Definitions`
- `Universe*` -> `Geometry` + contextual `Universe`
- `Objects` -> `Geometry`
- `Object root` -> `Home` + contextual `Selected Object`
- `Object / Geometry` -> `Geometry`
- `Object / Material` -> `Materials`
- `Object / Magnetic Parameters*` -> `Physics`
- `Object / Object Mesh` -> `Mesh`
- `Interfaces*` -> `Physics`
- `Excitations*` -> `Physics`
- `Study*` -> `Study`
- `Results*` -> `Results`

## 6.3. Dodatkowa reguła

Jeśli user kliknie child interaction node, ribbon powinien aktywować:

- core tab `Physics`
- contextual set `Selected Interaction`

Jeśli user kliknie stage root:

- core tab `Study`
- contextual set `Selected Stage`

Jeśli user kliknie plot/result node:

- core tab `Results`
- contextual set `Selected Plot` albo `Selected Result Asset`

## 7. Selection presentation model dla 3D

## 7.1. Potrzebny nowy obiekt stanu

Między tree/store a viewportem powinien powstać prosty, jawny model prezentacji selekcji:

```ts
interface SelectionPresentationModel {
  selectionOwnerType:
    | "model"
    | "universe"
    | "object"
    | "interface"
    | "stage"
    | "result";
  selectionOwnerId: string | null;
  selectedNodeId: string | null;
  highlightMode:
    | "none"
    | "object"
    | "geometry"
    | "mesh"
    | "field"
    | "interface";
  visibilityMode:
    | "normal"
    | "context"
    | "isolate";
  cameraHint:
    | "none"
    | "focus-owner"
    | "fit-all";
  meshOverlayMode:
    | "off"
    | "selected-object"
    | "domain"
    | "hybrid";
}
```

## 7.2. Przykłady

### Object root

- `selectionOwnerType = object`
- `highlightMode = object`
- `visibilityMode = current`
- `cameraHint = none`

### Object / Geometry

- `selectionOwnerType = object`
- `highlightMode = geometry`
- `meshOverlayMode = off`

### Object / Object Mesh

- `selectionOwnerType = object`
- `highlightMode = mesh`
- `meshOverlayMode = selected-object`

### Universe / Domain Mesh

- `selectionOwnerType = universe`
- `highlightMode = mesh`
- `meshOverlayMode = domain`

### Results / Field Data

- `selectionOwnerType = result`
- `highlightMode = field`
- `meshOverlayMode = hybrid`

## 11. Finalny werdykt

Aktualny frontend jest już dalej niż na początku, ale nadal nie jest jeszcze spójny jako profesjonalny GUI do budowy symulacji „jak MuMax + COMSOL”. Największy brak to nie pojedyncze ikonki, tylko to, że nadal **architektura informacji** i **semantyka selekcji** nie zostały doprowadzone do końca.

W praktyce:

- ribbon nadal jest częściowo dekoracją zamiast prawdziwym command surface,
- drzewo nadal jest zbyt run-centric i zbyt mało object-centric,
- inspector nadal bywa zlepkiem paneli zamiast czystym edytorem konkretnego node’a,
- 3D focus/highlight istnieją, ale nie są jeszcze częścią jednego, jawnego kontraktu.

Docelowy kierunek jest jednoznaczny:

1. **tree selection drives context**,
2. **ribbon reflects context**,
3. **inspector edits exactly one semantic node**,
4. **viewport presents the semantic owner of that node**.

To jest dokładnie ten moment, w którym trzeba przestać doklejać ikonki do starej architektury i dokończyć refaktor porządnie.
