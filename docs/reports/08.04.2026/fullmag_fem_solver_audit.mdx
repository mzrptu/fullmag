# Audyt solvera FEM w Fullmag

> Ten raport koncentruje się na tym, o co prosiłeś najbardziej: **czy implementacja FEM, relaksacji i oddziaływań jest poprawna naukowo oraz czy można ją traktować jako bezkompromisową podstawę do weryfikacji solvera mikromagnetycznego**.

> Główny wniosek, bez pudrowania: **jeszcze nie**. W repozytorium jest sporo dobrej roboty i kilka fragmentów naprawdę sensownej fizyki, ale są też bootstrapowe szwy, ukryte heurystyki, twarde stałe i kilka błędów spójności/API, które dyskwalifikują obecną całość jako „strict reference FEM solver” bez dalszych napraw.

## 1. Zakres audytu

Przeanalizowałem przede wszystkim te części kodu, które realnie decydują o fizyce i dynamice solvera:

- `native/backends/fem/src/mfem_bridge.cpp`
- `native/backends/fem/src/context.cpp`
- `native/backends/fem/src/api.cpp`
- `native/backends/fem/include/context.hpp`
- `native/include/fullmag_fem.h`
- `crates/fullmag-fem-sys/src/lib.rs`
- `crates/fullmag-engine/src/fem.rs`

Zakres merytoryczny:
- metoda FEM i sposób dyskretyzacji exchange/demag oraz pozostałych termów
- poprawność implementacji integratorów i relaksacji
- czy oddziaływania są liczone bez ukrytych kompromisów, heurystyk i hardkodowanych decyzji
- czy ścieżki bootstrap/reference są uczciwie nazwane i czy nie są mylone z pełnym solverem natywnym
- czy warstwa FFI i CPU reference nadaje się do walidacji backendu natywnego
- jak zautomatyzować testy tak, żeby solver był walidowany, a nie „wydawał się działać”

## 2. Werdykt końcowy

### 2.1. Werdykt w jednym zdaniu

**Obecna implementacja nie spełnia jeszcze standardu „bez uproszczeń i kompromisów” wymaganego do ścisłej weryfikacji solvera mikromagnetycznego FEM.**

### 2.2. Co jest dobre

- Natywny exchange w MFEM wygląda fizycznie sensownie i ma standardową strukturę weak-FEM z operatorem dyfuzji oraz projekcją przez zlumptowaną masę.
- Uniaxial anisotropy, cubic anisotropy i bulk DMI są zaimplementowane w sposób, który po czytaniu kodu wygląda merytorycznie solidnie.
- Poisson/airbox demag jest dużo bliższy prawdziwej natywnej ścieżce FEM niż transfer-grid bootstrap.
- Explicit LLG integrators są obecne i technicznie rozbudowane: Heun, RK4, BS23, DP54, adaptacyjny krok, FSAL.
- Kod zawiera już wiele śladów świadomego dojrzewania: komentarze o bootstrapie, poprawki F-xx/FEM-xx, walidacje markerów, ortogonalności osi, itp.

### 2.3. Co uniemożliwia dziś uznanie solvera za strict reference

- Transfer-grid demag nie jest czystym FEM demag, tylko hybrydą FEM→FDM→FEM.
- Rasteryzacja transfer-grid ma błąd nadpisywania komórek i nie jest konserwatywna objętościowo.
- Po uploadzie magnetyzacji `H_eff` może być niepełne.
- Step/snapshot błędnie zakładają, że musi istnieć exchange albo demag.
- Dirichlet Poisson ma fallback `pin DOF 0`, co jest naukowo nie do obrony w trybie referencyjnym.
- Termika jest liczona z jednego globalnego sigma, czyli niepoprawnie dla materiałów niejednorodnych i nieregularnych siatek.
- Interfacial DMI jest w praktyce ograniczone do przypadku z normalną interfejsu wzdłuż globalnego +z.
- Rustowa ścieżka CPU reference nie ma pełnej parzystości z backendem natywnym i ma własne heurystyki/hardkody.
- Rust FFI nie jest w pełni zgodne z nagłówkiem C dla observables.

### 2.4. Macierz oceny oddziaływań

| Oddziaływanie / mechanizm | Stan implementacji | Czy nadaje się do strict reference? | Główne zastrzeżenia | Werdykt |
| --- | --- | --- | --- | --- |

## 3. Mapa repo i faktyczny tor wykonania

### 3.1. Co jest czym

- `native/backends/fem/src/*` to natywny backend C++/CUDA/MFEM.
- `native/include/fullmag_fem.h` to publiczny kontrakt C.
- `crates/fullmag-fem-sys/src/lib.rs` to ręcznie utrzymywana warstwa Rust FFI.
- `crates/fullmag-engine/src/fem.rs` to CPU reference / bootstrapowa ścieżka FEM po stronie Rust.

### 3.2. Dlaczego to rozróżnienie jest kluczowe

Dla walidacji solvera nie wolno wrzucać do jednego worka następujących rzeczy:

- pełnej natywnej ścieżki mesh-native FEM
- bootstrapu, który rzutuje stan FEM na grid FDM i z powrotem
- CPU reference z uproszczonym zakresem termów
- IR/study-level deklaracji algorytmów, które jeszcze nie mają pełnej realizacji runtime

W Fullmag te warstwy są już dość blisko siebie, ale nadal nie są jeszcze w 100% rozdzielone semantycznie. To jest niebezpieczne właśnie wtedy, gdy chcesz solver **weryfikować**, a nie tylko odpalać demonstracje.

### 3.3. Faktyczny przepływ fizyki w backendzie natywnym

W uproszczeniu obecny backend natywny działa tak:

1. `context_from_plan(...)` wczytuje mesh, materiały, pola per-node i konfiguracje termów.
2. `context_initialize_mfem(...)` buduje przestrzenie FE, formy exchange i masę, oraz przygotowuje pola materiałowe.
3. Jeśli demag ma realizację airbox, `context_initialize_poisson(...)` buduje operator potencjału skalarnego na pełnej siatce.
4. W czasie `snapshot`/`step` całość przechodzi przez `compute_effective_fields_for_magnetization(...)`, która dokłada exchange, demag, anisotropy, DMI, bulk DMI, cubic anisotropy, Oersted, thermal i magnetoelastic.
5. Integrator LLG korzysta z gotowego `H_eff`.

Problem polega nie na tym, że kod nic nie liczy. Problem polega na tym, **jak dokładnie** to liczy, jak wiele rzeczy robi po drodze „po cichu”, oraz które ścieżki nadal są bootstrapowe.

## 4. Audyt metody FEM i mapowania równań na kod

### 4.1. Exchange

Teoria mikromagnetyczna (w wersji klasycznej) daje energię exchange w postaci:

```text
E_ex = ∫_Ω A |∇m|² dV
H_ex = -(1 / (μ0 Ms)) δE_ex / δm
```

W implementacji natywnej odpowiada temu operator dyfuzji zbudowany na polu `A`, a następnie projekcja przez diagonalną przybliżoną masę. W `apply_exchange_component_device(...)` pole składowej jest liczone jako:

```text
h_component = -(2 / (μ0 Ms_i)) * (K_A m_component) * inv_lumped_mass_i
```

To jest sensowna, standardowa dyskretyzacja weak-FEM z lumped-mass inversion. Jest kompromisem praktycznym, ale nie jest fizycznym absurdem. Wprost przeciwnie: **exchange jest jedną z mocniejszych części backendu natywnego**.

Jednocześnie trzeba uczciwie powiedzieć, że:
- to nie jest jeszcze finalny matrix-free / libCEED path
- kod sam deklaruje, że partial assembly w obecnym stosie MFEM 4.7 potrafi się wywrócić i dlatego użyto assembled operator
- projekcja przez masę jest zlumptowana, a nie consistent-mass

Wniosek: **exchange uznaję za naukowo sensowny, ale jeszcze nie „bezkompromisowy”**.

### 4.2. Demag transfer-grid

Tu zaczynają się prawdziwe kłopoty, bo to właśnie ten fragment najbardziej psuje status solvera jako referencyjnego FEM.

Aktualna ścieżka transfer-grid robi następujące rzeczy:

- buduje bounding box części magnetycznej i z niego dedukuje siatkę FDM transfer-grid
- dobiera krok komórki z `demag_transfer_cell_size` albo z `hmax`, z dolną podłogą `1e-12`
- rasteryzuję FEM magnetyzację do komórek FDM według testu środka komórki i barycentrycznych wag
- tworzy backend FDM i każe mu policzyć demag exact tensor/Newell na gridzie
- kopiuje `H_demag` z powrotem i trójliniowo próbuje na węzłach FEM
- energię liczy z `-0.5 μ0 Ms_i m·H * lumped_mass_i`

Sama koncepcja bootstrapu może być uzasadniona jako etap przejściowy. Nie to jest problemem. Problemem jest to, że **w tej ścieżce jest zbyt dużo ukrytych kompromisów na raz**:

- projekcja do gridu jest niekonserwatywna i zależna od kolejności elementów
- używany jest test środka komórki zamiast objętości przecięcia tetraedr-komórka
- próbkowanie z powrotem do FEM jest trójliniowe, a nie realizowane jako porządna projekcja między przestrzeniami
- przekazane widma jądra demag są ignorowane
- dobór siatki transfer-grid ma ukryte heurystyki

Wniosek: **tej ścieżki nie wolno przedstawiać jako strict reference FEM demag.** Można ją utrzymywać jako bootstrap/parity seam albo szybki tryb porównawczy, ale nie jako złoty standard.

### 4.3. Demag airbox Poisson / Robin / Dirichlet

Ta ścieżka jest dużo ciekawsza i dużo bardziej obiecująca.

Po stronie natywnej kod buduje skalarne H1 na pełnej siatce (materiał + airbox), składa operator Laplace’a i prawą stronę pochodzącą od `M = Ms m`. To już jest realna ścieżka natywnego FEM demag.

Co jest tu dobrze zrobione:
- RHS korzysta z interpolowanego `M = Ms m`, a nie z czysto elementowej stałej z powietrza
- mesh airbox jest jawnie częścią problemu, a nie tylko sztucznym marginesem w FDM
- solver ma warm start z poprzedniego kroku
- jest osobna ścieżka hostowego CG i osobna ścieżka Hypre/BoomerAMG
- energia demag i korekta Robin są uwzględniane

Co nadal wymaga naprawy, zanim nazwę to strict reference:
- Robin beta ma heurystykę z `R* = max_extent / 2` i fallback `R*=1`
- Dirichlet potrafi przypiąć DOF 0, jeśli marker granicy nie da żadnych essential DOFs
- parametry solvera liniowego i AMG są częściowo twardo zakodowane
- odzyskiwanie pola `H_demag = -∇u` przez ręczne nodal averaging jest praktyczne, ale nie jest najczystszą projekcją referencyjną

Wniosek: **to jest ścieżka, na której budowałbym walidację referencyjną Fullmag FEM**, ale dopiero po usunięciu fallbacków, ekspozycji heurystyk i poprawie recovery.

### 4.4. Zeeman / external

Ta część jest prosta i nie widać tu wielkich czerwonych flag. Pole zewnętrzne jest po prostu dodawane do `H_eff`. Tu bardziej problemem jest spójność całego `H_eff` niż sam Zeeman.

### 4.5. Uniaxial anisotropy

Implementacja używa spójnej formuły:

```text
H_ani = (2 Ku1 / (μ0 Ms)) (m·u) u + (4 Ku2 / (μ0 Ms)) (m·u)^3 u
E_ani = -Ku1 (m·u)^2 - Ku2 (m·u)^4
```

Kod poprawnie wspiera `Ku`, `Ku2`, osie i pola per-node. W tej części nie widzę krytycznego błędu naukowego.

### 4.6. Cubic anisotropy

Cubic anisotropy wygląda na zaimplementowaną merytorycznie poprawnie: są trzy stałe `Kc1`, `Kc2`, `Kc3`, walidacja osi krystalograficznych, wyprowadzenie pola i energii. To jest dobra część kodu. Nadal wymaga testów regresyjnych, ale nie widzę tu oczywistego sabotażu fizyki.

### 4.7. Interfacial DMI

Tu trzeba być bardzo precyzyjnym. Kod jest poprawny **dla konkretnego przypadku**: interfacial DMI z normalną interfejsu wzdłuż globalnego +z. Wzory używają wtedy tylko pochodnych po x i y.

To oznacza:

- dla cienkiej warstwy lub klasycznego układu filmowego z normalną `z` to może być poprawne
- dla geometrii obróconych, zakrzywionych lub wymagających jawnego `n_hat` implementacja nie jest ogólna
- API nie niesie tego ograniczenia dość głośno

Wniosek: **nie wolno mówić po prostu „interfacial DMI jest zaimplementowane”, bez dopowiedzenia „dla globalnego z-normal”**.

### 4.8. Bulk DMI

Bulk DMI jest policzone przez curl magnetyzacji i wygląda dużo pełniej niż interfacial DMI. Tutaj nie widać tej samej fundamentalnej blokady orientacyjnej. Nadal potrzebne są testy analityczne, ale implementacja wygląda obiecująco.

### 4.9. Oersted

Oersted jest liczony dla analitycznego modelu nieskończonego przewodnika cylindrycznego o arbitralnej osi. To jest uczciwe i całkiem sensowne, o ile model ograniczeń jest jasno opisany. Nie ma tu czerwonej flagi porównywalnej z demag transfer-grid.

### 4.10. Thermal

To jest ważne: nie wystarczy „mieć termikę”. Trzeba jeszcze liczyć ją zgodnie z lokalną fizyką.

Obecny kod używa jednego globalnego `sigma` wyprowadzonego z:

- uśrednionego `alpha`
- uśrednionego `Ms`
- średniej objętości magnetycznego węzła
- aktualnego `dt`

To jest akceptowalny bootstrap dla jednorodnego testu, ale **nie jest poprawnym referencyjnym modelem dla materiałów niejednorodnych ani dla nieregularnej siatki**.

### 4.11. Magnetoelastic

Magnetoelastic jest obecne, ale w modelu prescribed-strain. To trzeba traktować dokładnie tak, jak jest napisane: to nie jest pełne sprzężenie magnetomechaniczne, tylko użycie zadanego tensora odkształceń. W tym zakresie model jest sensowny, poza nim nie.

## 5. Audyt relaksacji i integratorów

### 5.1. Co realnie jest zaimplementowane w backendzie natywnym

W C++ natywnym backendzie widać:
- legacy Heun
- RK4
- Bogacki-Shampine 2(3)
- Dormand-Prince 4(5)
- adaptacyjny krok dla metod z embedded error estimate
- FSAL reuse dla BS23/DP54

To są **integratory czasowe LLG**. To nie jest automatycznie to samo co relaksatory energetyczne w sensie minimizacji.

### 5.2. Problem semantyczny: „relaksacja” vs „krok LLG”

W dokumentach i przykładach pojawia się słownictwo w rodzaju `projected_gradient_bb` czy `nonlinear_cg`, ale w natywnej warstwie C++ nie widać odpowiadających im natywnych solverów FEM. Widać za to dynamiczne integratory LLG.

To oznacza, że trzeba rozdzielić dwie klasy algorytmów:

- **integratory dynamiczne**: rozwiązują równanie ruchu LLG
- **relaksatory energetyczne**: bezpośrednio minimalizują energię, zwykle z projekcją na sferę `|m|=1`

Dla walidacji solvera to rozróżnienie jest kluczowe. Jeśli projekt mówi „FEM relax”, a runtime realnie robi tylko LLG z dużym tłumieniem, to nie wolno tego mieszać pod jedną etykietą.

### 5.3. Najważniejsze problemy relaksacji / integracji

- Brak natywnego FEM minimizera `projected_gradient_bb` / `nonlinear_cg` w warstwie C++.
- Błędny guard exchange/demag blokuje legalne przypadki innych pól efektywnych.
- Po uploadzie `H_eff` może być niespójne, co bezpośrednio wpływa na RHS LLG i testy relaksacji.
- Adaptive PI controller ma ukryte stałe (`pi_alpha`, `pi_beta`) niewystawione przez publiczne API.
- CPU reference ma ABM3, którego nie ma w natywnym API. To kolejny sygnał braku pełnej parzystości.

### 5.4. Wniosek o relaksacji

**Dynamika LLG jest zaimplementowana. Zaawansowana natywna relaksacja FEM jako osobna, wiarygodna klasa algorytmów jeszcze nie jest.**

## 6. Katalog usterek i kompromisów

### FND-001 — Transfer-grid demag w C++ nadpisuje komórki zamiast je konserwatywnie akumulować

**Waga:** KRYTYCZNE

**Lokalizacja:**

- `native/backends/fem/src/mfem_bridge.cpp:349-432`
- `native/backends/fem/src/mfem_bridge.cpp:1522-1637`

**Opis problemu**

Rasteryzacja przechodzi po tetraedrach i po wszystkich komórkach, których środek leży w tetraedrze. Dla każdej takiej komórki zapisuje `cell_magnetization_xyz[out + component] = ...`, czyli ostatni tetraedr, który trafi w komórkę, wygrywa. Nie ma akumulacji wkładów ani normalizacji przez objętość pokrycia komórki. To nie jest konserwatywna projekcja M = Ms·m na siatkę transferową.

**Konsekwencja naukowa / walidacyjna**

Współdzielone komórki między tetraedrami dają wynik zależny od kolejności elementów. Całkowity moment magnetyczny na gridzie może być błędny, szczególnie przy cienkich cechach geometrii, ostrych gradientach, lub siatce, w której kilka elementów przecina jedną komórkę transferową.

**Co trzeba zrobić**

Zastąpić ten krok konserwatywnym rzutem objętościowym: akumulować fizyczne M w każdej komórce z wagą równą objętości przecięcia tetraedru i komórki, a na końcu dzielić przez objętość wypełnioną. Domyślny tryb referencyjny ma zakazać ścieżki cell-center / last-write-wins.

### FND-002 — Ten sam błąd rasteryzacji istnieje w Rustowym CPU reference

**Waga:** KRYTYCZNE

**Lokalizacja:**

- `crates/fullmag-engine/src/fem.rs:1698-1780`

**Opis problemu**

Rustowa implementacja transfer-grid również robi test środka komórki i przypisanie `cell_magnetization[index] = ...` zamiast akumulacji. Dodatkowo nie skaluje magnetyzacji przez lokalne `Ms`, więc przy heterogenicznym materiale nie reprezentuje poprawnie fizycznego M.

**Konsekwencja naukowa / walidacyjna**

CPU reference i natywny backend dzielą ten sam podstawowy błąd projekcji. To psuje walidację FDM↔FEM, bo obie ścieżki mogą być błędne w ten sam sposób.

**Co trzeba zrobić**

Wyrównać CPU reference do poprawnej, konserwatywnej projekcji fizycznego pola M. Dodać test niezmienniczości na permutację elementów i zachowania całkowitego momentu.

### FND-003 — Ścieżka transfer-grid ignoruje przekazane spektra jądra demag i regeneruje własny tensor

**Waga:** WYSOKIE

**Lokalizacja:**

- `native/backends/fem/src/context.cpp:463-495`
- `native/backends/fem/src/mfem_bridge.cpp:1471-1503`

**Opis problemu**

Plan i kontekst przechowują `demag_kernel_*_spectrum`, ale przy budowie backendu FDM wszystkie wskaźniki są zerowane, a długość widma ustawiona na 0. Backend FDM generuje własne spektra Newella dla swojej siatki.

**Konsekwencja naukowa / walidacyjna**

Użytkownik może sądzić, że solver używa dokładnie przekazanego, wcześniej zweryfikowanego jądra, a w rzeczywistości ścieżka bootstrap je ignoruje. To jest zła sytuacja dla reprodukowalności i walidacji solvera.

**Co trzeba zrobić**

Albo przekazywać widma dalej po walidacji rozmiaru siatki, albo usunąć pola z API bootstrap i zwracać jawny błąd, że w tej realizacji nie są używane.

### FND-004 — Po uploadzie magnetyzacji `H_eff` może być niepełne

**Waga:** KRYTYCZNE

**Lokalizacja:**

- `native/backends/fem/src/context.cpp:762-845`
- `native/backends/fem/src/mfem_bridge.cpp:2970-2990`
- `native/backends/fem/src/mfem_bridge.cpp:1639-1797`

**Opis problemu**

Ścieżka uploadu woła pełne odświeżenie pól, ale zaraz potem ręcznie składa `H_eff` tylko z `H_ext + H_ex + H_demag`, a następnie dokłada Oersteda i magnetoelastykę. Nie dokłada uniaxial anisotropy, cubic anisotropy, interfacial DMI, bulk DMI ani termiki.

**Konsekwencja naukowa / walidacyjna**

Odczyt obserwabli zaraz po `upload_magnetization` może dać inny `H_eff` niż snapshot lub step. To jest błąd spójności API i może fałszować testy solvera.

**Co trzeba zrobić**

Usunąć ręczne składanie `H_eff` z uploadu i delegować wszystko do jednej funkcji składającej kompletne pole efektywne.

### FND-005 — Snapshot i step błędnie wymagają włączenia exchange albo demag

**Waga:** WYSOKIE

**Lokalizacja:**

- `native/backends/fem/src/mfem_bridge.cpp:2992-3008`
- `native/backends/fem/src/mfem_bridge.cpp:3075-3082`

**Opis problemu**

## 8. Różnice między backendem natywnym a CPU reference

To jest ważne, bo inaczej łatwo się oszukać i pomyśleć, że „CPU reference potwierdza backend natywny”.

### 8.1. Co potrafi natywny backend, a czego nie wnosi CPU reference

- natywny backend potrafi anisotropy, cubic anisotropy, interfacial DMI, bulk DMI, Oersted, thermal i magnetoelastic
- CPU reference składa `H_eff` tylko z exchange + demag + external
- CPU reference ma własny transfer-grid bootstrap z własnymi heurystykami
- CPU reference ma ABM3, którego nie ma w natywnym FFI API

### 8.2. Co to oznacza dla walidacji

Jeśli chcesz weryfikować backend natywny, CPU reference może dziś służyć tylko częściowo:

- jako sanity check dla exchange
- jako pomocniczy bootstrap dla demag w prostych przypadkach
- jako test integratorów CPU-only w ograniczonym modelu

Nie może natomiast być traktowane jako pełny odpowiednik wszystkich termów backendu natywnego.

## 10. Proponowana architektura po refaktorze

Żeby zapanować nad semantyką solvera, proponuję rozdzielić trzy profile obliczeń:

### 10.1. `strict_reference`

- bez cichych fallbacków
- bez automatycznych heurystyk, o ile użytkownik ich jawnie nie zaakceptuje
- natywny demag tylko przez ścieżkę Poisson/airbox albo inną jawnie mesh-native
- pełna ekspozycja tolerancji, recovery mode i polityk solvera
- testy canonical shapes i zbieżności obowiązkowe

### 10.2. `bootstrap_fast`

- może używać transfer-grid/FDM seam
- może używać cell-center rasterization, ale tylko jeśli nazwa i log mówią to wprost
- może używać uproszczonego recovery i heurystyk cell size
- służy do szybkich smoke testów i porównań z FDM

### 10.3. `production_accelerated`

- docelowy matrix-free / libCEED / Hypre / GPU-native
- agresywna optymalizacja do czasu, ale bez gubienia zgodności z `strict_reference`
- każda optymalizacja musi mieć test równoważności z profilem referencyjnym

To rozdzielenie pozwala wreszcie przestać mieszać fazę badawczą, bootstrap i solver referencyjny pod jednym słowem `fem`.

Dla solvera mikromagnetycznego nie wystarczy sam smoke test typu „program się nie wywalił”. Potrzebujesz golden data.

### 11.7. Kryteria akceptacji merge requesta

- Brak cichych fallbacków w trybie referencyjnym: żadnego `pin DOF 0`, żadnego ignorowania widm, żadnego auto cell-size bez jawnego profilu.
- Transfer-grid conservative projection przechodzi test zachowania całkowitego momentu i nie zależy od kolejności elementów.
- Po `upload_magnetization`, `snapshot` i `step` zwracają spójne `H_eff` dla identycznego stanu.
- Przypadki anisotropy-only, DMI-only, Oersted-only i thermal-only są poprawnie wspierane lub jawnie odrzucane z czytelnym błędem.
- CPU reference ma jasno ograniczony zakres albo pełną parzystość z natywnym backendem.
- Demag airbox przechodzi testy canonical shapes oraz zbieżność względem rafinacji siatki i rozmiaru air-box.
- FFI header parity jest w CI i nie da się przypadkiem rozjechać numeracji enumów.
- Workflow solver-regression publikuje artefakty JSON z energiami, polami i residuami, a comparator pilnuje tolerancji.

## 13. Końcowa ocena naukowa

Gdybym miał dziś odpowiedzieć jednym zdaniem, czy Fullmag FEM już liczy oddziaływania dipolowe, exchange i resztę **„prawidłowo, bez żadnych uproszczeń i kompromisów, hardkodowanych wartości”**, odpowiedź brzmi:

**Nie.**

Ale to nie jest zła wiadomość w sensie technicznym. To nie jest projekt, który stoi na glinianych nogach. To jest projekt, który ma już sensowny rdzeń, ale nadal łączy w jednym runtime:

- rzeczy naukowo solidne
- rzeczy bootstrapowe
- rzeczy niedokończone
- rzeczy niespójne między warstwami API/FFI/reference

Najbardziej obiecująca ścieżka do prawdziwej weryfikacji solvera to:

- oprzeć walidację demag na natywnym airbox/Poisson, nie na transfer-grid bootstrap
- naprawić spójność `H_eff` i API
- usunąć fallbacki i ukryte heurystyki z trybu referencyjnego
- zbudować naprawdę agresywną automatyczną matrycę testów canonical-shapes + convergence + API consistency + FFI parity

Po tych zmianach Fullmag będzie dużo bliżej solvera, któremu da się zaufać nie tylko dlatego, że daje ładne obrazki, ale dlatego, że przechodzi twardą walidację. Rzadki luksus w projektach naukowo-inżynierskich, gdzie zwykle wszyscy wierzą wynikom, dopóki nie zaczną się nie zgadzać.

## 15. Appendix B — checklist przed uznaniem solvera za reference-grade

- [ ] Transfer-grid conservative projection jest zaimplementowane i przetestowane.
- [ ] Tryb referencyjny nie ma cichych fallbacków ani ukrytych heurystyk.
- [ ] Poisson airbox ma jawnie konfigurowalne tolerancje, recovery mode i Robin policy.
- [ ] `H_eff` jest spójne między upload/snapshot/step.
- [ ] Snapshot/step działają dla dowolnej kombinacji legalnych termów.
- [ ] Interfacial DMI ma jawnie określoną normalną lub jest ograniczone i opisane jako z-normal only.
- [ ] Thermal używa lokalnego sigma_i.
- [ ] FFI jest automatycznie synchronizowane z nagłówkiem C.
- [ ] CPU reference ma jawnie określony zakres zgodności.
- [ ] CI wykonuje canonical-shape tests, convergence tests i API consistency tests.
