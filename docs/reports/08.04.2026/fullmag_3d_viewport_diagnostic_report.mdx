## 0. Executive summary

Mój główny werdykt jest prosty:

**obecny viewport 3D Fullmaga nie ma jednego spójnego modelu stanu.**  
Zamiast tego istnieje kilka częściowo niezależnych warstw:

1. globalny stan Control Room,
2. lokalny stan FDM viewportu,
3. lokalny stan FEM viewportu,
4. per-part state dla FEM,
5. stan selekcji wynikający z drzewa modelu,
6. osobne heurystyki dla isolate, focus, dimmingu i renderingu.

To właśnie dlatego użytkownik ma poczucie, że:

- „tryby się wykluczają”,
- „stany się mieszają w zależności od tego, który węzeł wybiorę”,
- „powrót do widoku globalnego nie jest przewidywalny”,
- „kolorowanie nie zawsze działa tak samo”,
- „surface/wire/layers/nodes” nie układają się w jeden logiczny system.

### Najważniejsze problemy produkcyjne

#### P0 — wybór węzła z drzewa wyłącza isolate
Aktualna logika `RunSidebar.tsx` przy wyborze węzła wykonuje side-effect:

```ts
model.setSelectedSidebarNodeId(id)
model.setSelectedObjectId(objectId)
model.setObjectViewMode("context")
```

To oznacza, że **sama nawigacja po drzewie modelu może kasować isolate**.  
Jest to dokładnie ten typ błędu, który użytkownik odbiera jako „GUI jest nieprzewidywalne”.

#### P0 — FDM i FEM nie mają wspólnej semantyki viewportu
FDM używa modelu:

- `glyph | voxel`
- `orientation | x | y | z`

FEM używa modelu:

- `surface | surface+edges | wireframe | points`
- `orientation | x | y | z | magnitude | quality | sicn | none`

Czyli z punktu widzenia użytkownika to wygląda jak jeden viewport 3D, ale pod spodem są to **dwa inne produkty**, z innym językiem renderingu, innym językiem koloru i inną logiką isolate.

#### P0 — w FDM color mode wpływa na geometrię
W `FdmInstances.tsx` próg widoczności voxeli jest liczony z metryki zależnej od aktualnego trybu koloru:

```ts
const metric =
  voxelColorMode === "orientation"
    ? mag
    : Math.abs(componentValue(mx, my, mz, voxelColorMode))

const isVisible = cellActive && sampled && metric >= voxelThreshold
```

To jest bardzo zły kontrakt produktu.  
Zmiana `ORI -> m_x` nie powinna zmieniać tego, **co znika z ekranu**, tylko jak to jest kolorowane.

#### P0 — isolate w FDM nie izoluje właściwego pola tak, jak użytkownik oczekuje
FDM isolate operuje głównie na overlayach obiektów, a nie na właściwej instancjonowanej geometrii pola.  
`FdmObjectOverlayMeshes` ukrywa nieselectowane boxy w isolate, ale `FdmInstances` nie dostaje obiektowego maskowania `selectedObjectId/objectViewMode`.  
W praktyce użytkownik widzi isolate „bardziej jako nakładkę”, a nie jako spójny display scope.

#### P1 — FEM ma potrójne źródło prawdy dla render mode
W aktualnym kodzie istnieją równolegle:

- globalny `meshRenderMode` w `ControlRoomContext`,
- lokalny `internalRenderMode` w `FemMeshView3D`,
- per-part `renderMode` w `meshEntityViewState`.

To oznacza, że render mode może istnieć jednocześnie jako:

- stan globalny,
- stan komponentu,
- stan warstwy/partu.

To jest architektoniczny zapach.  
Nawet jeśli dziś działa w części ścieżek, jest to fundament pod przyszłe regresje.

#### P1 — wybór air/interface/boundary nie jest traktowany jako pełnoprawny obywatel viewportu
W FEM domyślne `visible=false` dla `air` i `outer_boundary`, a logika `selectionKeepsVisible` działa tylko dla `part.role !== "air"`.  
Efekt: wybór air part może prowadzić do stanu „coś jest wybrane, ale nic nie widać”.

#### P1 — UI expose’uje niedojrzałość segmentacji
Sam viewport pokazuje badge typu:

- `Mesh Part`
- `Legacy Segment`
- `Bounds Fallback`
- `Missing exact object segmentation`

To jest uczciwe diagnostycznie, ale produktowo oznacza, że **jakość isolate zależy od źródła segmentacji**, a GUI nie buduje na tym czytelnego kontraktu.

### Najkrótsza recepta

Viewport trzeba rozdzielić na 4 osobne, jawne byty:

1. **Selection state** — co jest wskazane / wybrane,
2. **Visibility state** — co jest pokazywane / ukrywane,
3. **Presentation state** — jak to jest renderowane i kolorowane,
4. **Camera state** — co jest aktualnym framingiem i bookmarkiem widoku.

Dziś te 4 rzeczy są częściowo wymieszane.

## 2. Co jest dobrego w obecnym kierunku

Zanim przejdę do krytyki: w tym kodzie widać kilka **bardzo dobrych intuicji**.

### 2.1. Repo ma poprawną zasadę semantyczną
W README zapisano bardzo dobrą regułę:

- `UI visibility or isolate mode must never change the physical solver domain`

To jest dokładnie poprawna zasada.  
Czyli semantycznie Fullmag chce iść w dobrym kierunku: viewport jest tylko **prezentacją**, a nie zmianą dziedziny fizycznej.

### 2.2. FEM ma już bardziej „profesjonalny” zestaw trybów
`FemMeshView3D` ma dużo dojrzalszy zestaw pojęć niż FDM:

- `surface`
- `surface+edges`
- `wireframe`
- `points`

oraz

- `orientation`
- `x`, `y`, `z`
- `magnitude`
- `quality`
- `sicn`
- `none`

To jest dobry fundament pod produkcyjny viewport.

### 2.3. FDM ma już sensowną orientacyjną koloryzację
`magnetizationColor.ts` implementuje sensowną „orientation sphere”:

- hue z azymutu,
- saturation z promienia w płaszczyźnie,
- value z `m_z`.

To jest dobra baza dla trybu orientacyjnego.

### 2.4. FEM ma już realne jakościowe diagnostyki
`FemGeometry.tsx` ma obsługę:

- `quality`
- `sicn`
- fallback z `sicn -> aspect ratio`

To oznacza, że viewport nie jest tylko „ładną przeglądarką”, ale potrafi być narzędziem diagnostycznym.

### 2.5. Widać świadomość performance
`ViewportPanels.tsx` utrzymuje FDM canvas stale zamontowany, żeby nie niszczyć WebGL contextu i stanu kamery.  
To jest dobra intuicja wydajnościowa.

**Problem nie polega na tym, że kod jest „głupi”.**  
Problem polega na tym, że te dobre lokalne decyzje nie zostały spięte w **jedną architekturę produktu**.

## 3. Jak robią to dojrzałe narzędzia — COMSOL i 3ds Max

Nie chodzi o kopiowanie ich 1:1, ale o wzorce semantyczne.

### 3.1. COMSOL: selekcja, ukrywanie, reset i wireframe są jawnie rozdzielone
W COMSOL Graphics window istnieją jawne operacje typu:

- **Click and Hide**
- **View Unhidden**
- **View Hidden Only**
- **View All**
- **Reset Hiding**
- **Wireframe Rendering**
- **Zoom Selected**

Dodatkowo zaznaczane byty są w Graphics window **kolorystycznie podświetlane**, a selekcję można jawnie blokować / wyłączać.

To daje użytkownikowi trzy rzeczy:

1. osobną kontrolę nad **selection**,
2. osobną kontrolę nad **visibility**,
3. osobną kontrolę nad **display style**.

### 3.2. 3ds Max: isolate jest pełnoprawnym stanem UI
W 3ds Max `Isolate Selection` jest integralną częścią UI, ma dedykowane skróty i działa obok:

- `Hide Selection`
- `Hide Unselected`
- `Unhide All`
- `Selection Lock Toggle`
- `Zoom Extents Selected`

Najważniejsza lekcja z 3ds Max nie brzmi „dodaj Alt+Q”.  
Najważniejsza lekcja brzmi:

> **isolate nie może być tylko lokalnym efektem renderera; musi być pełnoprawnym, zarządzanym stanem sceny i selekcji.**

### 3.3. Wniosek dla Fullmaga
Fullmag powinien mieć taki sam rozdział:

- `selection`
- `isolate`
- `visibility`
- `camera / framing`
- `display style`
- `color mapping`

Dziś te pojęcia są częściowo połączone i częściowo rozproszone.

## 4. Aktualna architektura stanu — dlaczego ona się „rozjeżdża”

## 4.1. Globalny stan w `ControlRoomContext`
`ControlRoomContext.tsx` trzyma równocześnie m.in.:

- `meshRenderMode`
- `meshOpacity`
- `meshClipEnabled`
- `meshClipAxis`
- `meshClipPos`
- `meshShowArrows`
- `selectedObjectId`
- `viewportScope`
- `focusObjectRequest`
- `objectViewMode`
- `airMeshVisible`
- `airMeshOpacity`
- `meshEntityViewState`
- `selectedEntityId`
- `focusedEntityId`

Czyli już na poziomie jednego kontekstu jest bardzo dużo różnych kategorii stanu:

- selection,
- focus,
- visibility,
- isolate,
- per-part display,
- camera requests,
- clip planes,
- render mode.

To samo w sobie nie jest jeszcze błędem.  
Błąd pojawia się później, kiedy **te same kategorie są duplikowane niżej w drzewie komponentów**.

## 4.2. FDM ma własny lokalny stan persystowany globalnie w localStorage
`MagnetizationView3D.tsx` ma własne, lokalne `Settings`, a do tego zapisuje je do globalnych kluczy `localStorage`, np.:

- `preview3d_quality`
- `preview3d_render_mode`
- `preview3d_voxel_color_mode`
- `preview3d_voxel_threshold`
- `preview3d_voxel_gap`
- `preview3d_voxel_opacity`

To daje dwa efekty uboczne:

### 4.2.1. Stan nie jest sesyjny
Użytkownik przełącza sesje / uruchomienia / węzły, a viewport pamięta coś „z poprzedniego świata”.

### 4.2.2. Stan nie jest namespacowany
To nie jest:
- per run,
- per session,
- per backend,
- per quantity.

To jest po prostu globalne `preview3d_*`.

W profesjonalnym narzędziu to powinno być:
- albo stan sesji,
- albo jawny preset użytkownika,
- ale nie ukryty, globalny side effect.

## 4.3. FEM ma jednocześnie controlled props i internal state
`FemMeshView3D.tsx` trzyma lokalnie m.in.:

- `internalRenderMode`
- `field`
- `arrowColorField`
- `internalOpacity`
- `internalClipEnabled`
- `internalClipAxis`
- `internalClipPos`
- `internalShowArrows`
- `internalPreviewMaxPoints`
- `internalShrinkFactor`

a następnie robi:

```ts
const renderMode = controlledRenderMode ?? internalRenderMode
const opacity = controlledOpacity ?? internalOpacity
const clipEnabled = controlledClipEnabled ?? internalClipEnabled
...
```

To znaczy, że komponent jest jednocześnie:

- częściowo sterowany z góry,
- częściowo samoistny,
- a do tego ma jeszcze `meshEntityViewState` per part.

Dla produkcyjnego Control Room to jest zbyt dużo warstw pośrednich.

## 4.4. Dodatkowy trzeci poziom: per-part state
FEM ma jeszcze `meshEntityViewState[part.id]`, które niesie:

- `visible`
- `renderMode`
- `opacity`
- `colorField`

Czyli dla FEM mamy już **trzy poziomy sterowania renderem**:

1. globalny `meshRenderMode`,
2. lokalny `internalRenderMode`,
3. per-part `viewState.renderMode`.

To samo dotyczy opacity i częściowo color field.

### Konsekwencja
Przy takim modelu łatwo otrzymać sytuacje typu:

- wybrałem `surface`, ale selected part pokazuje się jako `surface+edges`,
- część warstw reaguje na globalne ustawienie, część na part-specific,
- jeden popover pokazuje stan A, a realny geometry pass renderuje stan B.

I dokładnie to już dziś w kodzie widać.

## 5. Diagnoza problemów — szczegółowo

## 5.1. Problem P0: kliknięcie w drzewie resetuje isolate

W `RunSidebar.tsx` funkcja `selectModelNode(id)` robi:

```ts
model.setSelectedSidebarNodeId(id)
model.setSelectedObjectId(objectId)
model.setObjectViewMode("context")
```

To jest najważniejszy błąd interakcyjny w całym flow.

### Dlaczego to jest złe
Użytkownik mentalnie robi dwa różne działania:

1. **nawiguję po drzewie / wybieram węzeł**
2. **zmieniam tryb widoku sceny**

Aktualny kod scala te dwa działania.  
Czyli zwykłe „kliknięcie obiektu w drzewie” jest równocześnie:
- selection command,
- focus command,
- exit isolate command.

To jest nieczytelne i niezgodne z intuicją narzędzi typu COMSOL/3ds Max.

### Jak użytkownik to odczuwa
- „wybrałem inny node i isolate mi zniknął”
- „czasem viewport wraca do globalu sam”
- „nie wiem, czy isolate jest aktywny czy nie”

### Co powinno być zamiast tego
`selectModelNode` nie może sam z siebie wymuszać `context`.

Powinno istnieć coś w rodzaju:

```ts
dispatch({
  type: "selection/set",
  target: resolveViewportTargetFromTreeNode(id),
})
```

a dopiero osobna polityka decyduje:

- jeśli `isolate.locked = true` → selection nie rusza isolate,
- jeśli `isolate.locked = false` i tryb = live isolate → selection aktualizuje isolate scope,
- jeśli użytkownik jest w zwykłym context mode → selection tylko highlightuje / fokusuja.

## 5.2. Problem P0: isolate ma inną semantykę w FDM i FEM

## 5.2.1. FDM
W FDM `objectViewMode` jest używany głównie w `FdmObjectOverlayMeshes`, gdzie:

- selected overlay jest highlightowany,
- w `isolate` nieselectowane overlaye są `return null`.

Natomiast właściwy renderer pola `FdmInstances` nie dostaje w propsach:
- `selectedObjectId`,
- `objectViewMode`,
- żadnej object-level maski.

Czyli isolate w FDM jest w praktyce **izolacją overlaya**, a nie pełnym izolowaniem prezentowanej domeny pola.

To jest podstawowa niespójność semantyczna.

## 5.2.2. FEM
W FEM isolate budowane jest przez `visibleLayers`, gdzie wchodzą:

- `selectedEntityId`
- `focusedEntityId`
- `selectedObjectId`
- `meshEntityViewState`
- `part.role`
- `objectViewMode`

Czyli isolate w FEM jest dużo bardziej „rzeczywisty” — operuje na render layers.

## 5.2.3. Wniosek
Użytkownik widzi jeden pill:
- `Context`
- `Isolate`

ale w dwóch backendach ten pill oznacza **coś innego**.

To nie jest problem kosmetyczny.  
To jest problem kontraktu produktu.

### Docelowa zasada
`objectViewMode` nie może być bezpośrednio konsumowany inaczej przez FDM i FEM.  
Musi istnieć wcześniej jedna warstwa:

```ts
DisplayScopeResolver
```

która mówi:

- jaki jest **resolved display scope**,
- jakie maski / index sets / node masks z tego wynikają,
- jak potem FDM albo FEM mapuje to na swoje backendowe render passes.

## 5.3. Problem P0: w FDM zmiana color mode zmienia widoczność geometrii

To jest jeden z najważniejszych błędów projektowych w całym viewportcie.

Obecnie:

```ts
const metric =
  voxelColorMode === "orientation"
    ? mag
    : Math.abs(componentValue(mx, my, mz, voxelColorMode))

const isVisible = cellActive && sampled && metric >= voxelThreshold
```

### Co to oznacza w praktyce
Jeśli użytkownik przełączy:
- `ORI` → `m_x`,
to nie tylko zmienia się kolor,
ale także zmienia się zbiór widocznych voxeli.

### Dlaczego to jest złe
Color mapping powinien odpowiadać na pytanie:
- **jak wizualnie zakodować wartość?**

Threshold / visibility powinny odpowiadać na pytanie:
- **co w ogóle pokazujemy?**

Mieszanie tych dwóch poziomów prowadzi do totalnej nieprzewidywalności.

### Objaw użytkownika
- „po zmianie koloru obiekt jakby się rozpadł”
- „m_x wygląda jak inna geometria”
- „nie wiem czy to kolor, czy clipping, czy threshold”

### Docelowa zasada
Wprowadzić dwa jawne pojęcia:

```ts
type VisibilityMetric =
  | "active-mask"
  | "magnitude"
  | "component-x"
  | "component-y"
  | "component-z"
  | "none"

type ColorField =
  | "orientation"
  | "x"
  | "y"
  | "z"
  | "magnitude"
  | "quality"
  | "sicn"
  | "none"
```

i UI musi je pokazywać osobno.

W trybie produkcyjnym domyślnie:
- `VisibilityMetric = magnitude`
- `ColorField = orientation | x | y | z | magnitude`

a próg widoczności nie powinien zależeć od aktualnie wybranego trybu koloru.

## 5.4. Problem P0/P1: FDM i FEM mają różne słowniki renderingu

## FDM
- `glyph`
- `voxel`

## FEM
- `surface`
- `surface+edges`
- `wireframe`
- `points`

Z punktu widzenia użytkownika nie ma jednego pytania:
> „jak chcę narysować obiekt?”

Zamiast tego ma dwa różne światy:

- w FDM: „arrows vs voxels”
- w FEM: „surface vs wire vs points”

### Dlaczego to boli
Użytkownik myśli w kategoriach:
- surface,
- wireframe,
- edges,
- vectors,
- points,
- layers.

A aplikacja zmienia mu słownik zależnie od backendu.

### Wniosek
Trzeba zdefiniować **kanoniczny model prezentacji**, np.:

```ts
type DisplayStyle =
  | "solid"
  | "solid+edges"
  | "wireframe"
  | "points"
  | "vectors"
  | "voxels"
```

z capability matrix:

```ts
FDM:
  vectors: yes
  voxels: yes
  solid: no (na dziś)
  solid+edges: no
  wireframe: no
  points: no

FEM:
  solid: yes
  solid+edges: yes
  wireframe: yes
  points: yes
  vectors: yes
  voxels: no
```

Wtedy GUI jest wspólne, ale umie:
- pokazać tylko wspierane tryby,
- resztę wyszarzyć z tooltipem,
- nie udawać, że FDM i FEM mają ten sam renderer.

## 5.5. Problem P1: FEM nadpisuje intencję użytkownika dla render mode

W `FemMeshView3D` aktualny `viewState` robi to:

```ts
renderMode:
  isSelected &&
  part.role !== "air" &&
  baseViewState.renderMode === "surface"
    ? "surface+edges"
    : baseViewState.renderMode
```

Czyli jeśli użytkownik wybrał `surface`, a part jest selected, renderer może go automatycznie podnieść do `surface+edges`.

### Dlaczego to jest problem
To jest klasyczny przykład „helpful UI, które łamie intencję użytkownika”.

Użytkownik chciał:
- czysty surface.

A aplikacja mówi:
- „domyślam się, że jednak chcesz edges”.

### Jak to powinno działać
Selection i highlight muszą być realizowane przez:
- obrys,
- halo,
- shell highlight,
- outline pass,
- ewentualnie overlay edges,

ale **nie przez zmianę wybranego display style**.

### Poprawny model
`renderMode` powinien być nienaruszalny, a highlight powinien być osobnym aspektem:

```ts
type HighlightPolicy =
  | "none"
  | "outline"
  | "shell"
  | "wire-overlay"
  | "dim-others"
```

## 5.6. Problem P1: air/interface/boundary nie są pełnoprawnymi targetami isolate i selection

Domyślne state dla partów:

```ts
visible: part.role !== "air" && part.role !== "outer_boundary"
renderMode: part.role === "air" ? "wireframe" : "surface+edges"
opacity: air=28, outer_boundary=46, interface=88, else=100
colorField: magnetic_object ? "orientation" : "none"
```

Dodatkowo:

```ts
const selectionKeepsVisible = isSelected && part.role !== "air"
```

a potem:

```ts
const visibleForMode =
  objectViewMode === "isolate" && hasSelection
    ? isSelected && (viewState.visible || selectionKeepsVisible)
    : (viewState.visible || selectionKeepsVisible)
```

### Efekt
Jeżeli wybrany jest air part, a air ma domyślnie `visible=false`, to selection nie musi go pokazać.

### Co to znaczy produktowo
Użytkownik może kliknąć:
- `Universe Airbox`
- `air mesh`
- `boundary`

i nie dostać jednoznacznej odpowiedzi wizualnej.

### Moja ocena
To może być świadoma decyzja techniczna, ale produktowo jest to zły UX.  
W profesjonalnym narzędziu wybór czegoś powinien dawać jednoznaczny feedback:
- highlight,
- auto focus,
- lub jawny komunikat „element ukryty — kliknij Show”.

### Rekomendacja
Air/interface/boundary muszą być pełnoprawnymi `ViewportTargetKind`, np.:

```ts
type ViewportTarget =
  | { kind: "object"; objectId: string }
  | { kind: "mesh-part"; partId: string }
  | { kind: "air-part"; partId: string }
  | { kind: "interface-part"; partId: string }
  | { kind: "boundary-part"; partId: string }
  | { kind: "node-set"; ... }
  | { kind: "face-set"; ... }
```

i dla każdego targetu trzeba określić:
- jak się highlightuje,
- czy może wejść do isolate,
- jak działa `Zoom Selected`,
- jaki jest fallback, jeśli exact segmentation nie istnieje.

## 5.7. Problem P1: isolate controls nie są równomiernie dostępne

- `orientation`

## 5.10. Problem P1: `magnitude` nie jest spójnie wspierane między backendami

To jest bardzo konkretny przypadek użytkownikowskiego bólu.

## FEM
- `magnitude` jest w `COLOR_OPTIONS`
- `FemGeometry.tsx` naprawdę liczy `sqrt(fx^2 + fy^2 + fz^2)`

## FDM
- `VoxelColorMode = "orientation" | "x" | "y" | "z"`
- brak `magnitude`

Czyli jeśli użytkownik myśli:

Jeśli FDM `magnitude` nie ma, GUI musi to komunikować, a nie tworzyć wrażenie „czasem działa, czasem nie”.

## 5.11. Problem P1: GUI expose’uje fallback segmentacji, ale nie buduje wokół tego polityki

Viewport pokazuje badge:
- `Mesh Part`
- `Legacy Segment`
- `Bounds Fallback`

oraz ostrzeżenie:
- `Missing exact object segmentation`

To jest bardzo cenna informacja developerska.  
Ale z punktu widzenia użytkownika oznacza to, że:
- isolate może być exact,
- albo approximate,
- albo tylko po boundsach.

### Co brakuje
Brakuje polityki UI:

- jeśli exact scope istnieje → isolate działa normalnie,
- jeśli jest tylko legacy segment → isolate działa z badge „approximate”,
- jeśli jest tylko bounds fallback → UI nie powinno udawać pełnego isolate; powinno np. wejść w `focus + dim + bounds-scope`, nie w exact isolate.

### Rekomendacja
Wprowadzić:

```ts
type ScopeQuality =
  | "exact"
  | "segment-approx"
  | "bounds-approx"
  | "unavailable"
```

oraz pokazywać to nie jako przypadkową etykietę, tylko jako element kontraktu isolate.

## 5.12. Problem P1: brak pełnego systemu `Show All / Reset Hiding / Zoom Extents / Zoom Selected`

Porównując do COMSOL i 3ds Max, Fullmag nie ma jeszcze kompletnego zestawu profesjonalnych operacji widoku.

### Dziś w praktyce są:
- focus na selected object,
- Context / Isolate pill,
- ViewCube / camera presets,
- trochę automatycznego camera-fit.

### Brakuje:
- `Show All`
- `Reset Visibility`
- `Reset Hiding`
- `Zoom Extents All`
- `Zoom Selected`
- `Frame Selection` niezależnie od isolate
- `Exit Isolate and Restore Camera`

To powoduje, że „powrót do widoku globalnego” nie jest pierwszoklasową operacją.

### Rekomendacja
Dodać jawny zestaw komend viewportu:

```ts
type ViewportCommand =
  | { type: "camera/zoomExtentsAll" }
  | { type: "camera/zoomSelected" }
  | { type: "camera/resetProjection" }
  | { type: "visibility/showAll" }
  | { type: "visibility/resetBaseline" }
  | { type: "isolate/enter"; target: ViewportTarget }
  | { type: "isolate/exit" }
  | { type: "isolate/toggleLock" }
```

## 5.13. Problem P2: hardcoded wartości i heurystyki są porozrzucane po kodzie

To nie są najgorsze bugi, ale są symptomem niedojrzałości systemu.

### FDM
- `sceneOpacityMultiplier = 1` z komentarzem o sorting artifacts
- `geometry-only` opacity = `0.85`
- `baseScale = max(0.12, step * (1 - voxelGap))`
- dla `nz == 1` depthScale ma osobny hack
- `brightness` range `0.3..3.0`
- `voxelGap` `0.02..0.42`
- `voxelThreshold` `0..0.95`

### FEM
- air opacity = `28`
- outer boundary = `46`
- interface = `88`
- dimmed non-selected = `8` lub `14`
- selected lift = `52` lub `96`

### Dlaczego to boli
Hardcoded values są okej w prototypie, ale produktowo trzeba mieć:
- centralny `ViewportThemeTokens`
- centralny `ViewportDefaultPolicies`
- centralny `SelectionVisualizationPolicy`
- centralny `MeshRoleDefaults`

Inaczej każda mała zmiana stylu lub zachowania wymaga dłubania w kilku plikach.

## 8. Kontrakt isolate — jak to ma działać

## 8.1. Wejście w isolate
Po wejściu w isolate:

1. zapisujemy `camera bookmark`,
2. zapisujemy `visibility snapshot`,
3. resolve’ujemy target do `display scope`,
4. przechodzimy w `isolationState.enabled = true`.

## 8.2. Dwa tryby isolate

### `live isolate`
Selection aktualizuje isolate target.

### `locked isolate`
Selection zmienia tylko highlight, ale nie scope isolate.

To jest dokładnie ten brakujący element, który 3ds Max rozwiązuje selection lockiem.

## 8.3. Wyjście z isolate
`Exit isolate` robi:

1. odtworzenie visibility snapshot,
2. odtworzenie camera bookmark,
3. wyzerowanie isolate state,
4. pozostawienie selection bez zmian.

To jest kluczowe: wyjście z isolate nie może kasować selection, bo selection i isolate to nie to samo.

## 9. Kontrakt kamery

## 9.1. Osobne komendy
Viewport musi mieć osobne, jawne komendy:

- `Zoom Extents All`
- `Zoom Selected`
- `Frame Isolate Scope`
- `Restore Previous View`
- `Fit Active Target`
- `Reset Projection`

## 9.2. Camera bookmarks
Wejście w isolate tworzy bookmark.  
Wyjście z isolate przywraca bookmark.

## 9.3. Global view
„Powrót do widoku globalnego” musi być pierwszoklasową funkcją, nie produktem ubocznym:
- kliknięcia w drzewie,
- wyłączenia isolate,
- albo zresetowania selection.

## 16. Acceptance criteria — kiedy viewport będzie „produkcyjny”

Uważam, że viewport można uznać za produkcyjny dopiero gdy spełni jednocześnie poniższe warunki.

## 16.1. Selection / isolate
- kliknięcie w drzewie **nie kasuje isolate**, chyba że użytkownik jawnie tego chce,
- istnieje `lock isolate`,
- istnieje `exit isolate`,
- istnieje `show all / reset visibility`,
- air/interface/boundary/object są równoprawnymi targetami.

## 16.2. Rendering
- display style nie jest nadpisywany przez selection,
- FDM i FEM używają jednego słownika UI, z capability matrix,
- unsupported style są disabled, nie ukryte „magicznie”.

## 16.3. Color
- active color mode jest zawsze widoczny,
- color field nie wpływa na visibility,
- legenda jest spójna z kolorem,
- `|m|` działa wszędzie tam, gdzie ma sens.

## 16.4. Camera
- działa `Zoom Extents All`,
- działa `Zoom Selected`,
- działa restore previous camera,
- isolate zachowuje camera bookmark.

## 16.5. Persistence
- stan viewportu jest sesyjny lub jawnie presetowy,
- nie ma ukrytych globalnych side effectów localStorage bez namespacowania.

## 19. Mój końcowy werdykt

Problem viewportu 3D w Fullmagu **nie polega na pojedynczym bug-u w isolate**.  
To nie jest „jedna pomylona flaga”.

To jest problem bardziej fundamentalny:

> **aplikacja nie ma jeszcze jednej kanonicznej semantyki widoku 3D.**

Dziś istnieją równolegle:

- model obiektowy,
- model mesh-part,
- model overlayowy,
- model FDM pola,
- model FEM powierzchni,
- model selection z drzewa,
- model isolate z pill buttonu,
- model widoczności per part,
- model koloru zależny od backendu.

Dopóki nie zostanie to rozdzielone na:
- selection,
- isolate,
- visibility,
- presentation,
- camera,

dopóty użytkownik będzie czuł, że:
- tryby się gryzą,
- stany są „magiczne”,
- powrót do globalnego widoku jest niejednoznaczny,
- a viewport jest „niedopracowany”.

### Najkrótsza definicja refaktoru
Nie refaktoruj „kolejnego przycisku”.
Refaktoruj **model stanu viewportu**.

### Najważniejszy pierwszy commit
Jeśli miałbym wskazać jeden pierwszy commit, zrobiłbym dokładnie ten:

1. usunąć `model.setObjectViewMode("context")` z `RunSidebar.tsx`,
2. dodać `ViewportTarget + isolate lock`,
3. rozdzielić `ColorField` od `VisibilityMetric` w FDM.

To nie rozwiąże wszystkiego, ale natychmiast zatrzyma najbardziej irytujące objawy.

## 20. Aneks — propozycja docelowego API wewnętrznego

```ts
export interface UnifiedViewportState {
  selection: ViewportSelectionState
  isolation: ViewportIsolationState
  visibility: ViewportVisibilityState
  presentation: ViewportPresentationState
  camera: ViewportCameraState
}

export interface ViewportCameraState {
  projection: "perspective" | "orthographic"
  navigationMode: "trackball" | "orbit"
  currentBookmarkId: string | null
  stack: CameraBookmark[]
}

export interface CameraBookmark {
  id: string
  label: string
  position: [number, number, number]
  target: [number, number, number]
  up: [number, number, number]
  projection: "perspective" | "orthographic"
}
```

### Adapter do starego UI
```ts
function legacyObjectViewMode(state: UnifiedViewportState): "context" | "isolate" {
  return state.isolation.enabled ? "isolate" : "context"
}
```

### Adapter do rendererów
```ts
function buildFdmRenderPayload(state: UnifiedViewportState, scene: SceneData): FdmRenderPayload
function buildFemRenderPayload(state: UnifiedViewportState, scene: SceneData): FemRenderPayload
```

To jest moim zdaniem właściwy kierunek dojścia do **jednego, profesjonalnego viewportu 3D**, który można potem rozwijać na poziomie COMSOL/3ds Max zamiast ciągle gasić lokalne konflikty.
