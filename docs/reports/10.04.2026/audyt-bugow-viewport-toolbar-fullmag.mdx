# Audyt bugów viewport / toolbar – Fullmag

## Najważniejsze ustalenia

### 1. Slider `Opacity` nie steruje faktyczną opacity w trybach wireframe / surface+edges
- `FemViewportToolbar.tsx` wystawia slider 10–100.
- `FemGeometry.tsx` używa `opacity / 100` tylko dla `meshStandardMaterial` i `pointsMaterial`.
- Linie w `surface+edges` i `wireframe` mają zahardkodowane opacity (`0.12`, `0.22`, `0.58`, `0.95`, `0.09`, `0.16`, `0.32`, `0.72`).
- Efekt: po przełączeniu trybu wygląd i intensywność zmieniają się „same”, a powrót do 100% przez slider nie odtwarza poprzedniego stanu.

### 2. Toolbar render mode / opacity patchuje wszystkie aktualnie widoczne części siatki
- `FemMeshView3D.tsx` buduje `toolbarStylePartIds` jako `visibleLayers.map((layer) => layer.part.id)`.
- `applyToolbarRenderMode()` i `applyToolbarOpacity()` robią patch dla całej tej listy.
- To oznacza, że jeden toolbar działa globalnie na wszystkie widoczne warstwy, mimo że UI sugeruje pojedynczy stan.

### 3. Slider pokazuje opacity tylko pierwszej widocznej warstwy
- `toolbarOpacity` bierze `visibleLayers.find(...)` i zwraca opacity pierwszej znalezionej warstwy.
- Jeśli części mają już różne opacity, UI pokazuje wartość mylącą.

### 4. Strzałki mogą być „włączone” w UI, ale nadal niewidoczne
- Przycisk `Vectors` przełącza tylko boolean `arrowsVisible`.
- Render używa ostrzejszego warunku: `showArrows && !missingMagneticMask && visibleArrowNodeCount > 0`.
- `FemArrows.tsx` dodatkowo zwraca pusty wynik, gdy `visible === false` albo `sampledNodes.length === 0`.
- Efekt: użytkownik klika włączenie strzałek, ale nic nie wraca, bo blokuje je maska albo brak kandydatów do rysowania.

### 5. Podczas interakcji viewport sam degraduje tryb renderu i gęstość strzałek
- `runtimeRenderMode` zamienia `surface+edges` i `points` na `surface`, gdy `interactionActive === true`.
- `runtimeArrowDensity` obcina gęstość do ~45% podczas interakcji.
- To poprawia wydajność, ale bez jasnego sygnału w UI wygląda jak niestabilny stan toolbaru.

### 6. Architektura ma dwa źródła prawdy: globalny stan FEM i stan per-part
- `ControlRoomContext.tsx` przechowuje globalne `meshRenderMode`, `meshOpacity`, `meshShowArrows`.
- Równolegle trzyma `meshEntityViewState` z per-part `renderMode`, `opacity`, `colorField`, `visible`.
- Presety FEM również zapisują oba poziomy.
- To samo w sobie nie jest błędem, ale bez jasnych reguł priorytetu łatwo o rozjazd UI ↔ viewport.

## Priorytet napraw

### P1 — naprawić semantykę opacity
W `FemGeometry.tsx` wszystkie opacity dla linii powinny być mnożone przez `opacityVal`, np.:
- `lineOpacity = baseLineOpacity * opacityVal`
- `hiddenLineOpacity = baseHiddenLineOpacity * opacityVal`

### P2 — zmienić scope toolbaru
Toolbar nie powinien domyślnie targetować `all visible parts`.
Najbezpieczniej:
- jeśli zaznaczony jest konkretny part → patch tylko ten part,
- w braku selekcji → tryb globalny,
- opcjonalnie dodać jawny scope switch: `Selected / Visible / All`.

### P3 — dodać stan `mixed`
Dla render mode / opacity / color field toolbar powinien rozpoznawać, że widoczne części mają mieszane wartości.
Wtedy UI nie powinno udawać jednej konkretnej wartości.

### P4 — rozdzielić `arrows requested` od `arrows renderable`
Dodać diagnostykę:
- `showArrowsRequested`
- `arrowsRenderable`
- przyczyna blokady: `missingMagneticMask`, `noVisibleArrowNodes`, `vectorDomainFilteredOut`

Toolbar powinien pokazać badge lub tooltip typu:
- `Vectors enabled, but no visible nodes match current filter`.

### P5 — ujawnić albo ograniczyć runtime degradation
Są dwie sensowne ścieżki:
1. zostawić degradację, ale oznaczyć ją w UI jako `interactive simplification`,
2. albo wyłączyć downgrade `surface+edges -> surface` i zostawić tylko redukcję density.

## Minimalny test regresyjny
1. Otwórz mesh z co najmniej 2 widocznymi partami.
2. Ustaw `surface+edges`, opacity 100.
3. Przełącz: `surface -> wireframe -> surface+edges -> points -> surface`.
4. Sprawdź, czy po każdym kroku:
   - slider pokazuje prawdziwy stan,
   - opacity może wrócić do 100,
   - kolor/intensywność nie dryfują,
   - strzałki wracają po ponownym włączeniu,
   - brak różnicy między stanem toolbaru a stanem renderu.

## Wniosek
To nie wygląda na pojedynczy bug w samym toolbarze. Główna przyczyna jest mieszana:
- błędna semantyka opacity w rendererze,
- zły scope aktualizacji stanu,
- ukryte warunki wygaszające strzałki,
- dwa poziomy stanu bez jasnej synchronizacji.
