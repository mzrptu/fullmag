# Fullmag Frontend / 3D Viewport — raport diagnostyczny regresji wydajności

**Data:** 2026-04-10  
**Zakres audytu:** `apps/web/components/preview/**`, `apps/web/components/runs/control-room/**`, `apps/web/lib/workspace/**`  
**Tryb pracy:** audyt kodu frontendu, bez bisectu commitów i bez profilera runtime w przeglądarce  
**Cel:** znaleźć najbardziej prawdopodobne źródła tego, że viewport 3D i cały frontend reagują ciężko, z opóźnieniami rzędu sekund, mimo mocnego sprzętu.

---

## TL;DR

To nie wygląda na jeden pojedynczy bug. Regresja jest dziś skutkiem kilku nakładających się decyzji architektonicznych:

1. **Viewport nadal renderuje się od zbyt szerokiego stanu aplikacji.**  
   Rozdzielenie kontekstów zostało częściowo unieważnione przez legacy `useControlRoom()` oraz przez to, że `viewportValue` nadal zawiera dane transportowe i preview, które zmieniają się często.

2. **FDM 3D viewport jest stale zamontowany, nawet gdy jest ukryty.**  
   To miało chronić WebGL context, ale w praktyce powoduje, że ukryty viewport nadal uczestniczy w zmianach propsów, przełączeniach trybów i przebudowach instancji.

3. **FEM renderuje geometrię per-part, a każda warstwa wykonuje własną ciężką przebudowę topologii.**  
   Przy jednym obiekcie FEM nie renderujemy jednej geometrii, tylko zwykle kilka warstw: obiekt magnetyczny, airbox, boundary, interface. Każda z nich odpala własny ciężki `FemGeometry`.

4. **Każdy ruch kamery rozsyła pracę do kilku dodatkowych subsystemów.**  
   Główny canvas, `ViewCube`, `HslSphere`, overlay manager i sync z kontrolkami pracują razem. To samo w sobie nie musi zabijać FPS, ale przy powyższych problemach staje się mocnym multiplikatorem kosztu.

Najmocniejszy wzorzec regresji jest więc taki:

`zbyt szerokie subskrypcje Reacta` + `ukryty ale żywy viewport` + `duplikacja ciężkiej geometrii FEM` + `dodatkowe gizmo canvasy i sync`

To bardzo dobrze tłumaczy objawy:

- scroll / orbit / pan są ciężkie nawet dla małej sceny,
- zmiana zakładki powoduje dziwne „mikro-przesunięcie” widoku 3D,
- frontend muli globalnie, nie tylko sam shader,
- problem mógł narastać przez wiele commitów, bo każda z tych rzeczy osobno wyglądała „rozsądnie”.

## Główne źródła problemu

## 1. Split context istnieje tylko częściowo

W `apps/web/components/runs/control-room/context-hooks.tsx:495` nadal istnieje legacy hook:

```ts
export function useControlRoom(): ControlRoomContextValue {
  const transport = useTransport();
  const viewport = useViewport();
  const command = useCommand();
  const model = useModel();
  return useMemo(
    () => ({ ...transport, ...viewport, ...command, ...model }),
    [transport, viewport, command, model],
  );
}
```

To oznacza, że każdy komponent używający `useControlRoom()`:

- subskrybuje **wszystkie cztery** domeny naraz,
- dostaje nowy obiekt po każdej zmianie w którejkolwiek domenie,
- traci korzyść z rozdzielenia kontekstów.

Problem jest praktyczny, nie teoretyczny:

- `apps/web/components/runs/control-room/ViewportPanels.tsx:406` używa `useControlRoom()`
- `apps/web/components/runs/RunControlRoom.tsx:185` używa `useControlRoom()`
- `apps/web/components/runs/control-room/WorkspaceControlStrip.tsx:63` używa `useControlRoom()`

To są dokładnie komponenty, które otaczają viewport.

### Dodatkowy błąd architektoniczny

W `apps/web/components/runs/control-room/ControlRoomContext.tsx:3748-3783` `viewportValue` nadal zawiera:

- `selectedVectors`
- `fieldStats`
- preview-related state

czyli dane, które nie są czysto „UI viewportu”, tylko są zależne od transportu i preview pipeline.

To rozwala izolację nawet wtedy, gdy komponent przeszedłby z `useControlRoom()` na `useViewport()`.

### Wniosek

Największy problem Reactowy jest taki:

- rozdzieliliśmy konteksty,
- ale viewport dalej czyta je jak jeden wielki store,
- a do „viewport context” nadal wrzucamy dane, które zmieniają się często.

To jest **P0**.

## 2. FDM viewport jest ukryty, ale nie jest wyłączony

W `apps/web/components/runs/control-room/ViewportPanels.tsx:966-1001` FDM viewport jest stale mounted:

```tsx
<div className={cn("absolute inset-0", showFdm3D ? "block" : "hidden")}>
  <ViewportErrorBoundary label="FDM 3D Viewport">
    <MagnetizationView3D
      vectors={isFdm3DActive ? ctx.selectedVectors : null}
      geometryMode={isFdmMeshActive}
      ...
      viewportVisible={showFdm3D}
    />
  </ViewportErrorBoundary>
</div>
```

To ma bardzo konkretną konsekwencję:

- przejście z `3D` do `Analyze` albo `2D` nie odmontowuje viewportu,
- tylko zmienia mu propsy,
- między innymi `vectors` przechodzą z pełnego pola do `null`,
- a potem z powrotem do pełnego pola.

W `apps/web/components/preview/r3f/FdmInstances.tsx:249-360` ten komponent reaguje na to pełnym przebiegiem po siatce i przebudową instancji.

Szczególnie kosztowne są:

- pełna pętla przez `vectors`
- liczenie `maxMagnitude`
- zapis macierzy instancji
- zapis kolorów
- `invalidate()`

czyli dokładnie to, czego nie chcemy wykonywać, gdy viewport jest niewidoczny albo gdy użytkownik tylko zmienia zakładkę.

### Dlaczego to tłumaczy „viewport drgnął po zmianie zakładki”

Zmiana zakładki:

- wywołuje re-render shellu,
- zmienia layout i widoczność paneli,
- hidden canvas dostaje nowe propsy,
- canvas przechodzi przez resize / reflow / invalidate,
- kontrolki i gizma synchronizują się z nowym stanem.

To nie musi oznaczać, że kamera „naprawdę” wykonała orbitę.  
Wystarczy, że doszło do przeliczenia projekcji, targetu, size albo ukrytego re-renderu z remeasure.

To jest bardzo spójne z opisanym przez Ciebie objawem.

## 3. FEM mnoży koszt topologii przez liczbę partów

W `apps/web/components/preview/fem/FemViewportScene.tsx:241-260` każdy widoczny layer renderuje własny `FemGeometry`:

```tsx
{hasMeshParts
  ? visibleLayers.map((layer) => (
      <FemGeometry
        key={layer.part.id}
        meshData={meshData}
        ...
      />
    ))
  : null}
```

To wygląda niewinnie, ale `FemGeometry` jest ciężkie.

W `apps/web/components/preview/r3f/FemGeometry.tsx:245-380` każde `FemGeometry`:

- tworzy nowe `Float32Array(nNodes * 3)`,
- kopiuje wszystkie node positions,
- liczy bounding box,
- filtruje elementy/faces,
- buduje nowe `BufferGeometry`,
- liczy normals,
- opcjonalnie buduje wireframe,
- opcjonalnie buduje points geometry.

A potem w `apps/web/components/preview/r3f/FemGeometry.tsx:570-647` jeszcze robi:

- odświeżanie `color` bufferów,
- kopiowanie kolorów z base array do aktualnej geometrii,
- kolejne `invalidate()`.

### Dlaczego to jest krytyczne nawet dla „jednego okna”

Bo „jedno okno FEM” w aktualnym modelu nie oznacza jednej geometrii renderowej.  
Typowo mamy co najmniej:

- part magnetyczny,
- airbox,
- boundary,
- interface.

Czyli już od razu kilka instancji `FemGeometry`, a każda wykonuje ciężką robotę.

To jest klasyczny przypadek, gdzie:

- scena wizualnie jest mała,
- ale koszt CPU jest mnożony przez liczbę warstw i danych pomocniczych.

### Wniosek

Największy koszt FEM nie siedzi tylko w shaderach.  
Siedzi w tym, że topologia jest budowana **per warstwa**, a nie raz na mesh z lekkimi overlayami / maskami.

To jest **P0/P1**.

## 4. Główna interakcja kamery rozsyła pracę do kilku równoległych podsystemów

Sam główny canvas nie jest jedynym odbiorcą ruchu kamery.

### 4.1 `HslSphere`

`apps/web/components/preview/HslSphere.tsx:73-116`:

- podpina się do `controls.change`,
- kopiuje quaternion kamery,
- woła `invalidate()` na własnym canvasie.

Do tego `apps/web/components/preview/HslSphere.tsx:143-156` pokazuje, że to jest osobny `Canvas`.

Czyli każdy obrót głównej kamery = dodatkowy render osobnego canvasu.

### 4.2 `ViewCube`

`apps/web/components/preview/ViewCube.tsx:104-127`:

- podpina listener pod `controls.change`,
- aktualizuje CSS transform dla cube i axis DOM.

To nie jest najcięższa rzecz w systemie, ale jest kolejnym kosztem od każdego ruchu kamery.

### 4.3 Overlay manager

`ViewportOverlayManager` ma własny `ResizeObserver`, oblicza warianty layoutu i generuje overlay tree zależne od szerokości/wysokości viewportu.  
To nie jest główny winowajca, ale przy częstych resize/reflow dokłada kolejne obliczenia do ścieżki interakcji.

### Wniosek

Wydajność nie pada tylko dlatego, że „mesh jest ciężki”.  
Pada też dlatego, że pojedynczy ruch kamery uruchamia kaskadę:

- render głównego canvasu,
- sync gizma DOM,
- render HSL sphere,
- overlay/layout reactions,
- czasem dodatkowy invalidation chain.

## 5. `BoundsPreview3D` nadal renderuje się w trybie ciągłym

W `apps/web/components/preview/BoundsPreview3D.tsx:357`:

```tsx
<Canvas camera={{ position: [3, 2.4, 3], fov: 45, near: 0.0001, far: 10000 }}>
```

Tu nie ma `frameloop="demand"`.  
Domyślnie oznacza to continuous render loop.

Jeżeli użytkownik jest w tym typie widoku albo jeśli taki viewport pozostaje mounted dłużej w większym shellu, to mamy kolejny canvas renderujący stale.

To nie tłumaczy całej regresji, ale jest ewidentnym długiem wydajnościowym.

## 6. Taby i layout nadal mogą przepychać viewport przez szeroki rerender

`apps/web/components/runs/RunControlRoom.tsx:184-238`:

- `ControlRoomShell` używa `useControlRoom()`,
- używa `useActiveStageLayout()`,
- ma własny `viewportSize` state aktualizowany na `resize`,
- reaguje na dock/tab layout.

Do tego `apps/web/lib/workspace/workspace-store.ts:120-122`:

```ts
export function useActiveStageLayout(): StageLayoutState {
  const state = useWorkspaceStore();
  return state.stageLayouts[state.currentPerspective];
}
```

To subskrybuje cały store, nie tylko potrzebny wycinek.

### Efekt praktyczny

Zmiana zakładki / docka / ribbonu może:

- re-renderować shell,
- re-renderować `ViewportCanvasArea`,
- przepchnąć nowy zestaw propsów do viewportu,
- wywołać resize/reflow overlayów,
- a przy FDM także obudzić hidden canvas.

To jest bardzo silny kandydat na Twoje obserwowane „przy zmianie zakładki 3D minimalnie się rusza”.

## 7. Dodatkowe wtórne koszty

To nie są najpewniej główne źródła sekundowych lagów, ale wzmacniają problem:

### 7.1 Transport ma stały tick 200 ms podczas runu

`apps/web/components/runs/control-room/hooks/useControlRoomTransport.ts:58-64`

Podczas `running` i `materializing_script` timer aktualizuje `now` co 200 ms.  
Sam timer jest niewinny, ale w połączeniu z szerokimi subskrypcjami daje gwarantowany churn Reacta nawet bez nowego payloadu solvera.

### 7.2 FDM tworzy geometrię i materiał bez wyraźnego dispose path

`apps/web/components/preview/r3f/FdmInstances.tsx:191-234` tworzy:

- nowe geometry,
- nowe materials

przy zmianie `mode`, `quality` i opacity profile.  
W tym komponencie nie widać symetrycznej logiki disposal porównywalnej do `FemGeometry`.

To nie tłumaczy samego „scroll jest ciężki od razu”, ale dobrze tłumaczy degradację po dłuższej sesji, wielu zmianach jakości i trybów.

### 7.3 FEM field-data nadal kopiuje wektory node-by-node przy każdej aktualizacji

`apps/web/components/runs/control-room/ControlRoomContext.tsx:3397-3418`

To jest sensowne przy live preview, ale oznacza dodatkowy koszt CPU na każdy tick solvera.  
Jeżeli szeroki rerender nie zostanie odcięty, to ten koszt łatwo zaczyna promieniować dalej po UI.

## Zweryfikowane i odrzucone podejrzenia

W obecnym stanie kodu **nie potwierdziłem**, że główny FDM viewport ma stale włączone `preserveDrawingBuffer: true`.

Wręcz przeciwnie: w `apps/web/components/preview/MagnetizationView3D.tsx:1073-1089` aktualny canvas ma tylko:

```tsx
gl={{
  antialias: settings.quality !== "low",
}}
```

To ważne, bo część starszych raportów lub podejrzeń mogła wskazywać `preserveDrawingBuffer` jako główny winowajca.  
W aktualnym kodzie to **nie jest** potwierdzone źródło problemu dla głównego FDM viewportu.

## Najbardziej prawdopodobny łańcuch przyczyn

Najbardziej prawdopodobny obecny root-cause chain wygląda tak:

1. Solver / preview / command / tab / layout zmienia jakiś fragment globalnego stanu.
2. `useControlRoom()` scala domeny z powrotem, więc viewport host dostaje nowy szeroki obiekt.
3. `ViewportCanvasArea` przelicza dużo logiki i pcha propsy do canvasa.
4. W FDM hidden viewport nadal jest mounted, więc nadal reaguje.
5. W FEM kilka `FemGeometry` przebudowuje własne topologie i kolorowanie.
6. W trakcie interakcji kamery dochodzi jeszcze `ViewCube` i `HslSphere`.

Efekt końcowy:

- input jest ciężki,
- orbit/scroll/pan przestaje być „GPU-bound”, a staje się mieszanym CPU/React/layout bottleneckiem,
- UI sprawia wrażenie, jakby viewport odświeżał się skokowo.

## Kolejność napraw

Jeżeli mamy naprawiać to z maksymalnym zwrotem z inwestycji, kolejność powinna być taka:

1. **Przestać używać `useControlRoom()` w ścieżce viewportu i shella.**  
   `ViewportCanvasArea`, `ViewportBar`, `ControlRoomShell`, `WorkspaceControlStrip` powinny przejść na granular hooks i selektywne propsy.

2. **Oczyścić `viewportValue` z transportowych danych.**  
   `selectedVectors`, `fieldStats` i podobne rzeczy nie powinny siedzieć w viewport context.

3. **Przestać utrzymywać hidden FDM viewport jako aktywne drzewo renderowe.**  
   Jeśli musi przetrwać WebGL context, to potrzebny jest prawdziwy mechanizm „freeze/suspend”, a nie tylko `display:none`.

4. **Zredukować FEM z per-part heavy geometry do jednego topology base + lekkie overlaye.**  
   Dziś każda warstwa ma własny koszt topologii. To trzeba spłaszczyć.

5. **Przełączyć `BoundsPreview3D` na `frameloop="demand"`.**

6. **Ograniczyć auxiliary gizma work.**  
   `HslSphere` nie powinien być osobnym renderującym canvasem na każdą zmianę kamery, jeżeli viewport już ledwo oddycha.

7. **Dodać disposal path do FDM geometry/material switching.**

## Ocena końcowa

Najbardziej prawdopodobne źródło regresji nie leży w pojedynczym shaderze ani w jednym złym parametrze Three.js.  
Problem siedzi głębiej: w tym, że frontend 3D został z czasem połączony zbyt szeroko z globalnym stanem aplikacji, a jednocześnie zaczął utrzymywać coraz więcej żywych podsystemów viewportu naraz.

To jest dobra wiadomość o tyle, że problem da się naprawić bez polowania na jeden historyczny commit.  
Zła wiadomość jest taka, że sama kosmetyka nie wystarczy. Trzeba przeciąć kilka hot-pathów renderowania i reaktywności.

## Najkrótszy werdykt techniczny

**Źródło problemu to przede wszystkim frontendowy render-churn i duplikacja ciężkiej pracy, nie „za słaby komputer” i nie pojedynczy błąd GPU.**

Najmocniejsze winne dziś:

- `useControlRoom()` w ścieżce viewportu,
- `viewportValue` zanieczyszczony danymi transportowymi,
- always-mounted hidden FDM canvas,
- per-layer `FemGeometry` rebuild,
- dodatkowe sync/render subsystemy (`HslSphere`, `ViewCube`, overlays).

## Status wdrożenia napraw

### Zaimplementowane

1. **`viewportValue` został oczyszczony z danych transportowych.**  
   `selectedVectors` i `fieldStats` zostały przeniesione do `TransportContextValue`, więc viewport context nie jest już tykany przez szybkie dane solvera tylko dlatego, że są potrzebne do renderu pola.

2. **Ukryty FDM viewport nie jest już utrzymywany jako stale mounted canvas.**  
   `MagnetizationView3D` jest teraz renderowany tylko wtedy, gdy FDM 3D lub FDM Mesh naprawdę jest aktywny.

3. **Canvasy R3F dostały stabilny host / event source.**  
   Dodano wspólny helper `useCanvasHost`, a `MagnetizationView3D`, `BoundsPreview3D` i `ScientificViewportShell` nie montują już canvasa zanim istnieje realny host DOM.  
   To adresuje błąd typu `Cannot read properties of null (reading 'addEventListener')` w lifecycle R3F.

4. **`BoundsPreview3D` przeszedł na `frameloop="demand"`.**  
   Ten preview nie renderuje się już stale bez potrzeby.

5. **Pomocnicze gizma zostały odchudzone.**  
   `ViewCube` i `HslSphere` aktualizują się przez `requestAnimationFrame` zamiast wykonywać pełną synchronizację na każdy event kontrolera.

6. **Dodano cleanup geometrii/materialów FDM.**  
   `FdmInstances` dispose’uje zasoby przy przełączaniu geometrii i materiału, co ogranicza degradację podczas dłuższej sesji.

7. **Ciężkie subtree viewportowe zostały ustabilizowane.**  
   `ViewportBar`, `ViewportCanvasArea`, `WorkspaceControlStrip`, `FemViewportScene` i `FemGeometry` zostały opakowane w memoizację, żeby nie odświeżały się od każdego parent re-render.

8. **Ograniczono część globalnego churnu UI.**  
   `useActiveStageLayout()` przestał czytać cały store, a kilka callbacków/effectów w `RunControlRoom` i `ViewportPanels` dostało węższe zależności.

9. **Analyze nie spamuje już pustych endpointów `eigen/*`.**  
   Frontend sprawdza listę artefaktów przed fetchowaniem `spectrum`, `dispersion` i `branches`, więc nie generuje już niepotrzebnych `404`.

10. **`useSessionStream` został ustabilizowany pod dev/StrictMode.**  
    Początkowe łączenie WebSocketu jest minimalnie odroczone, żeby nie tworzyć i nie zamykać natychmiast socketu w czasie remountów developerskich.

11. **Tick transportowy został odchudzony.**  
    Odświeżanie `elapsed/stepsPerSec` podczas pracy solvera nie chodzi już co `200 ms`, tylko co `1000 ms`, co zmniejsza globalny churn UI bez realnej utraty użyteczności.

### Zmitigowane, ale nie zlikwidowane architektonicznie

1. **`useControlRoom()` nadal istnieje w kilku miejscach shella.**  
   Najcięższe subtree viewportowe zostały od niego odseparowane memoizacją i przeniesieniem szybkich danych do transportu, ale sam legacy facade nadal istnieje i warto go docelowo wycofać z `ControlRoomShell`, `SettingsPanel` i inspektorów.

2. **FEM nadal renderuje kilka `FemGeometry` dla wielu warstw.**  
   Memoizacja ogranicza zbędne przebudowy, ale docelowy refactor „jedna baza topologii + lekkie overlaye” nadal byłby lepszy.

### Dodatkowy incydent dev-runtime wykryty podczas walidacji

W czasie walidacji pojawił się osobny problem, który **nie jest tym samym co regresja wydajności viewportu**:

- aktywny Next dev-serwer pracował na porcie `3000`,
- równolegle został uruchomiony pełny `next build`, który czyści `apps/web/.next`,
- działający proces dev-serwera pozostał podpięty do niekompletnego stanu chunków,
- skutkiem było `500` na `/` oraz błędy klienta typu:
  - `Cannot find module './778.js'`
  - `originalFactory is undefined`

To jest awaria stanu bundlera / HMR, a nie błąd logiki React komponentów.  
Naprawa była operacyjna:

- zatrzymanie starego procesu `dev-server.mjs`,
- czysty restart control-room,
- ponowna kompilacja dev bundle.

Wniosek praktyczny: nie należy czyścić `apps/web/.next` ani odpalać pełnego `next build` równolegle z działającym live control-room, jeśli chcemy zachować zdrowy stan developerskiego runtime.

### Czego jeszcze nie potwierdza sam build

Ten pakiet zmian usuwa główne źródła render-churn i błędów lifecycle, ale pełne potwierdzenie wymaga jeszcze uruchomienia workspace w przeglądarce i sprawdzenia:

- płynności orbit/pan/zoom dla małej sceny FDM,
- płynności zmiany zakładek `Model / Study / Results`,
- braku błędu R3F `addEventListener` na `null`,
- braku sztucznych `404` dla `eigen/spectrum`, `eigen/dispersion`, `eigen/branches`.
