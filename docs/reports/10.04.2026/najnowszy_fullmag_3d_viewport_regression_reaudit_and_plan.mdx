# Fullmag 3D viewport — re-audyt po ostatnim pushu

## Cel

Sprawdzić, czy ostatnie zmiany faktycznie naprawiły regresję wydajności viewportu 3D, oraz czy implementacja jest już poprawna architektonicznie. Zakres audytu obejmuje:

- host viewportu i przepływ stanu w Control Room,
- FDM `MagnetizationView3D` + `FdmInstances`,
- FEM `ScientificViewportShell` + `FemMeshView3D` + `FemGeometry` + `FemArrows`,
- pomocnicze gizma i insety (`ViewCube`, `HslSphere`),
- granicę między zmianą topologii a zmianą pola.

Data audytu: bieżący `master` repo po najnowszym pushu.

---

## Werdykt

**Nie, to nie jest jeszcze zaimplementowane dobrze end-to-end.**

Masz kilka realnych popraw:
- FEM dostał porządny wspólny shell viewportu z profilami jakości, `frameloop="demand"` i rozdzieleniem jakości interakcyjnej od capture.
- W `ControlRoomContext` pojawił się sensowny split na `femMeshBase` i `femFieldData`.
- FDM nadal zachowuje jeden zamontowany canvas, więc nie traci kontekstu WebGL przy przełączaniu widoków.

Ale jednocześnie **pozostały trzy krytyczne błędy architektoniczne**, które nadal mogą zabijać FPS nawet dla małych scen:

1. **Host viewportu nadal subskrybuje zbyt szeroki stan**  
   `ViewportCanvasArea` nadal bierze legacy `useControlRoom()`, czyli z powrotem scala `transport + viewport + command + model`. To podważa sens split-contextów.

2. **FEM nadal przebudowuje kosztowną topologię za szeroko**  
   `FemGeometry` deklaruje „topology memo”, ale memo zależy od **całego `meshData`**, podczas gdy `ControlRoomContext` i tak składa nowy `femMeshData` po każdej aktualizacji pola. W efekcie przy zmianie `fieldData` bardzo łatwo przebudować topologię, normalsy, wireframe i points.

3. **FDM nadal ma ciężki własny pipeline renderu poza wspólnym shell-em**  
   FDM nie przeszedł na `ScientificViewportShell`. Nadal ma osobny `Canvas`, własne GL ustawienia, oddzielne insety i kosztowne pełne przebudowy instancji w `FdmInstances`.

W skrócie: **wydajność poprawiła się miejscami, ale rdzeń regresji nie został jeszcze usunięty**.

To jest poważny problem architektoniczny.

FEM przeszedł na wspólny shell z profilami jakości i sensownym `preserveDrawingBuffer` policy, ale FDM nadal ma własny, odrębny pipeline.

## Dlaczego nadal możesz widzieć „lag nawet dla małej sceny”

Bo obecny system nadal ma **nakładanie się kilku kosztów naraz**:

1. Host viewportu renderuje szerzej niż powinien (`useControlRoom()`).
2. FDM ma własny cięższy canvas path niż FEM.
3. FDM ma stale żywy inset `HslSphere`.
4. FEM „topology memo” nie jest naprawdę topologiczne.
5. FEM nadal buduje zasoby dla kilku trybów naraz.
6. FDM/FEM nadal robią ciężkie pełne przebiegi po danych przy update.

To daje efekt:
- CPU zajęty Reactem + aktualizacją buforów,
- GPU zajęte przez niepotrzebne zasoby i dodatkowe canvasy,
- `demand` nie daje tyle, ile powinien, bo zbyt często coś robi `invalidate()` albo stale renderuje obok.

## Etap 0 — szybki hotfix regresji (P0, 1 dzień)

### 0.1. Wyłączyć `preserveDrawingBuffer` w interaktywnym FDM
**Cel:** natychmiast odzyskać FPS.

Zmiana:
- w `MagnetizationView3D.tsx` zastąpić stałe `preserveDrawingBuffer: true`
- policy:
  - interaktywny viewport: `false`
  - snapshot/capture: tymczasowy offscreen lub tymczasowy render profile

Docelowo:

```tsx
gl={{
  antialias: settings.quality !== "low",
  preserveDrawingBuffer: captureRequested,
```

albo lepiej:
- przejść całym FDM na `ScientificViewportShell`.

Najprostszy test A/B:
- ukryć `HslSphere` za feature flagą lub ustawieniem,
- zostawić tylko `ViewCube`.

Jeśli FPS wróci natychmiast, masz twarde potwierdzenie kosztu inset canvas.

## Etap 1 — prawdziwe rozdzielenie topologii od pola w FEM

### 1.1. Zmienić API `FemGeometry`

Aktualnie:
```tsx
<FemGeometry meshData={meshData} field={field} ... />
```

Docelowo:
```tsx
<FemGeometry
  meshBase={meshBase}
  fieldData={fieldData}
  activeMask={activeMask}
  quantityDomain={quantityDomain}
  field={field}
  renderMode={renderMode}
  ...
/>
```

Gdzie:
- `meshBase` ma stabilną referencję dopóki nie zmienia się topologia,
- `fieldData` ma własną referencję i zmienia się per tick,
- `activeMask` i `quantityDomain` są osobno.

### 1.2. Memo topologii ma zależeć tylko od topologii

Docelowe dependencies:
```tsx
[
  meshBase,
  customBoundaryFaces,
  clipAxis,
  clipEnabled,
  clipPos,
  displayBoundaryFaceIndices,
  displayElementIndices,
  globalCenter,
  shrinkFactor,
]
```

Nie:
```tsx
meshData
```

### 1.3. `invalidate()` usunąć z `useMemo`
`useMemo` ma być czyste.
`invalidate()` przenieść do dedykowanego `useEffect`.

## Etap 2 — lazy build zasobów FEM per render mode

### 2.1. Budować tylko to, co potrzebne
Dzisiaj budujesz:
- normals,
- edge geom,
- tetra wire,
- points geom,

zawsze.

Powinno być:

```tsx
const needSurface = renderMode === "surface" || renderMode === "surface+edges";
const needEdges = renderMode === "surface+edges";

const needPoints = renderMode === "points";
```

I dopiero na tej podstawie:
- `computeVertexNormals()` tylko gdy `needSurface`
- `WireframeGeometry` tylko gdy `needEdges`
- `pointsGeometry` tylko gdy `needPoints`
- `tetraEdgesGeometry` tylko gdy `needVolumeWire`

### 2.2. Kolor update też rozdzielić
Jeśli render mode nie używa np. punktów, nie wolno aktualizować `pointsColorAttr`.

## Etap 3 — unifikacja FDM na `ScientificViewportShell`

To jest mój najmocniejszy rekomendowany ruch.

### Dlaczego
Bo dziś FEM i FDM żyją na dwóch różnych filozofiach viewportu:
- FEM: profile jakości, render policy, shared shell
- FDM: własny Canvas, własne GL options, własne overlaye i insety

To utrudnia:
- utrzymanie,
- diagnostykę,
- porównanie zachowań,
- wspólne hotfixy wydajności.

### Docelowy ruch
Przenieść FDM na:

```tsx
<ScientificViewportShell
  qualityProfile={runtimeQualityProfile}
  renderPolicy={{ mode: "demand", hidden: !viewportVisible, interactionActive }}
  ...
>
  <FdmScene />
</ScientificViewportShell>
```

Korzyści:
- jedna polityka `preserveDrawingBuffer`,
- jedna polityka DPR,
- jedna polityka hidden/paused/demand,
- jedna warstwa bridge do gizm.

## Etap 4 — ograniczenie kosztów insetów i gizm

### 4.1. `HslSphere`
Zmienić z:
- osobnego stale aktywnego canvasu z `useFrame`

na jedno z:
- statyczny canvas renderowany tylko po zmianie kamery,
- `<Canvas frameloop="demand">` + ręczne `invalidate`,
- sprite/tekstura zamiast pełnego drugiego renderera,
- domyślnie OFF, włączane przez user toggle.

### 4.2. `ViewCube`
Przejść z polling loop do event-driven sync:
- subskrypcja `controls change`,
- update tylko po realnej zmianie kamery,
- bez własnego `requestAnimationFrame`.

### 5.1. FDM `FdmInstances`
Rozdzielić:
- zmiany topologii/isolate bounds,
- zmiany koloru,
- zmiany opacity,
- zmiany threshold/sampling.

Nie każda zmiana powinna powodować pełny rebuild macierzy.

#### Priorytety
1. jeśli zmienia się tylko opacity → tylko materiał
2. jeśli zmienia się tylko color mode → tylko colors
3. jeśli zmienia się tylko isolate bounds → tylko subset widocznych instancji
4. maxMagnitude nie powinno być liczone pełnym skanem przy każdym update, jeśli istnieje upstream stats lub można użyć próbkowania

### 5.2. FEM `FemArrows`
Rozdzielić:
- sampling node set (rzadko),
- payload pozycji (rzadko),
- payload kolor/skala (częściej),
- upload instancing data (najczęściej, ale minimalny).

Docelowo:
- `sampledNodes` zależy tylko od topologii + density + masks
- `positions` zależą tylko od topologii + center
- `colors/scales` zależą od `fieldData`
- jeśli to za mało: worker / off-main-thread preprocessing

#### P1. `useCurrentLiveStream` pobiera bootstrap przy każdym reconnect/remouncie

To nie musi być złe raz, ale przy niestabilnym drzewie komponentów bardzo szybko zamienia się w lawinę żądań.

1. **Natychmiast usunąć `useCurrentAnalyzeArtifacts()` z `RunSidebar`**

2. **Zostawić jednego właściciela danych Analyze**

- wszystko jest ciężkie nawet dla małych danych.

1. **Analyze ma jednego właściciela danych**
   - `useCurrentAnalyzeArtifacts()` został usunięty z sidebara

   - nowy helper: `apps/web/lib/workspace/navigation-debug.ts`
   - zapisuje zdarzenia do `window.__FULLMAG_DEBUG_EVENTS__`
   - loguje m.in.:
     - root redirect z `/`
     - detekcję live session
     - wejście do workspace stage
     - `router.push()` z control room
     - auto scene push i jego odroczenia

3. **Ograniczono bootstrap storm w live stream**
   - `useSessionStream.ts` ma teraz:
     - cache krótkiego życia dla bootstrap payload
     - reuse inflight bootstrap
     - skip pełnego `fetchBootstrap()` przy świeżym stanie po reconnect

4. **Przycięto auto scene push w rozruchu**
   - `ControlRoomContext.tsx` nie robi już automatycznego `updateSceneDocument()` podczas najwcześniejszej fazy hydratacji / bootstrappingu
   - dodano krótki startup gate oraz logowanie `scene-sync`

### Jak czytać nową instrumentację

W devtools console i w runtime można teraz sprawdzić:

```js
window.__FULLMAG_DEBUG_EVENTS__
```

Najważniejsze scope:

- `root-route`
- `live-intent`
- `workspace-entry`
- `run-control-room`
- `live-stream`
- `scene-sync`

### Wniosek architektoniczny

Tak: problem rzeczywiście częściowo polega na tym, że frontend próbował mieć za dużo „aktywnych” subsystemów naraz.

To nie jest jeszcze tak, że cały Analyze viewport renderował się zawsze, ale:

- dane Analyze były eager-loadowane z sidebara,
- live session bootstrap był uruchamiany za szeroko,
- scene sync wchodził zbyt wcześnie,

Czyli problem nie sprowadza się do „jednego widoku”, tylko do **braku ostrego ownershipu danych i lifecycle per ekran / per stage**.
