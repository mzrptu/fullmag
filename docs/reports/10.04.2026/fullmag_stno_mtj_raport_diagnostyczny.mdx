# Raport diagnostyczny: Fullmag a STNO/MTJ vortex

## Kontekst

Cel inspekcji był bardzo konkretny:

- **układ**: nanokropka / nanopillar o średnicy **400 nm**
- **warstwa aktywna**: **CoFeB(1 nm) / NiFe(8 nm)**
- **zjawisko**: **vortex STNO**
- **pobudzenie**:
  - prąd elektryczny stały (DC),
  - prąd elektryczny zmienny w czasie (AC / envelope),
  - interesują nas zarówno **mod gyrotropowy vortexu**, jak i **fale spinowe**
- **pytanie praktyczne**:
  1. czy obecna fizyka / backend Fullmag już to potrafi,
  2. czego jeszcze brakuje,
  3. czy **GUI** jest gotowe do takiej analizy.

Inspekcja objęła pięć warstw repozytorium:

1. **publiczny Python DSL**
2. **IR / planner**
3. **engine / runner**
4. **dokumentację i capability matrix**
5. **frontend / GUI / analyze workspace**

---

## Werdykt w jednym akapicie

**Krótki werdykt techniczny**:

- **backend fizyczny** jest **bliżej gotowości niż sugeruje README i część dokumentacji**;
- **publiczny workflow użytkownika** (Python DSL + GUI + analiza wyników) **nie jest jeszcze domknięty** dla pełnego przypadku **CPP MTJ vortex STNO**;
- **pole Oersteda** i **czasowe pobudzenia prądowe** są już bardzo wyraźnie obecne;
- **STT (w tym Slonczewski dla MTJ)** istnieje w warstwie IR / planner / runner, ale **nie jest normalnie wystawione w publicznym `Problem(...)`**;
- **GUI** ma dziś sens głównie jako ogólny builder studiów i workbench do **eigenmodes / dispersion**, a **nie** jako kompletne środowisko do **time-domain STNO**.

W praktyce oznacza to:

> **Tak — da się dojść do symulacji vortex STNO w Fullmag, ale jeszcze nie przez w pełni gotowy, oficjalny i klikalny workflow.**  
> **Nie — nie nazwałbym obecnego stanu „gotowym modułem STNO/MTJ” bez kilku brakujących ekspozycji i narzędzi diagnostycznych.**

## Co repozytorium ma już dziś

## 1) Publiczny Python DSL: więcej niż sugerują stare notki

W publicznym DSL znalazłem kilka bardzo istotnych elementów:

### 1.1. Pole Oersteda od nanopillara

W `packages/fullmag-py/src/fullmag/model/energy.py` jest:

- `OerstedCylinder`
- obwiednie czasowe:
  - `Constant`
  - `Sinusoidal`
  - `Pulse`

Co więcej, sam komentarz przy `OerstedCylinder` wprost odnosi się do **STNO / MTJ pillar**. To jest mocny sygnał, że autor repo **świadomie celował** w ten przypadek fizyczny.

To oznacza, że dla Twojego zastosowania:

- **DC current** → można modelować **statyczny bias Oersteda**
- **AC current** → można modelować **rezonansowe pobudzenie** modu gyrotropowego i modów fal spinowych
- **pulse** → można robić ringdown i ekstrakcję częstotliwości z odpowiedzi impulsowej

### 1.2. Anizotropia, DMI, thermal-noise są w DSL

W tym samym obszarze kodu są również:

- `UniaxialAnisotropy`
- `CubicAnisotropy`
- `InterfacialDMI`
- `BulkDMI`
- `ThermalNoise`

To nie jest jeszcze dowód, że każda z tych rzeczy działa end-to-end w każdym backendzie, ale jest to ważne z punktu widzenia architektury:

- **CoFeB(1 nm)** często wymaga sensownego potraktowania **anizotropii interfejsowej / efektywnej**
- dla oscylatora STNO docelowo chcesz móc badać:
  - próg autooscylacji,
  - amplitudę,
  - selekcję modu,
  - **linewidth**,
  - wpływ **temperatury**

Bez thermal/noise nie ma wiarygodnej ścieżki do linewidth.

### 1.3. Materiał i geometria mają wystarczającą bazę pod thin-film disk

W `Material` są jawnie dostępne:

- `Ms`
- `A`
- `alpha`
- stałe anisotropii
- nawet pola przestrzennie zmienne:
  - `Ms_field`
  - `A_field`
  - `alpha_field`
  - `Ku_field` itd.

To jest bardzo ważne dla Twojego przypadku, bo daje **dwie ścieżki modelowania**:

#### Ścieżka A — model efektywny
Potraktować `CoFeB(1)/NiFe(8)` jako **jedną efektywną warstwę swobodną** o grubości ~9 nm i skutecznych parametrach `Ms, A, alpha, Ku`.

#### Ścieżka B — model jawny po `z`
Zrobić jedną bryłę i nadać jej **przestrzennie zmienne parametry po osi `z`**:
- komórka 0..1 nm → CoFeB
- komórki 1..9 nm → NiFe

To jest dzisiaj najbezpieczniejsza droga, jeśli chcesz odwzorować **exchange-coupled composite free layer** bez budowania skomplikowanej semantyki stosu MTJ.

## 2) IR / planner / runner: tu widać prawdziwy potencjał STNO

Najmocniejszy wniosek z całej inspekcji jest taki:

> **Warstwa IR / planner / runner wygląda na bardziej rozwiniętą fizycznie niż publiczny Python API i niż oficjalne „public-executable” notki.**

### 2.1. STT już siedzi w `ProblemIR`

W `crates/fullmag-ir/src/lib.rs` są pola typu:

- `current_density`
- `stt_degree`
- `stt_beta`
- `stt_spin_polarization`
- `stt_lambda`
- `stt_epsilon_prime`
- `temperature`

To jest dokładnie ten zestaw, którego oczekujesz od implementacji:

- **Zhang–Li**
- **Slonczewski**
- temperatura / thermal path

Dla **MTJ vortex oscillatora** najważniejsze jest tutaj:

- `stt_spin_polarization`
- `stt_lambda`
- `stt_epsilon_prime`
- `current_density`

czyli sygnały, że **CPP-like STT path** jest przewidziany.

### 2.2. Planner przenosi te pola dalej

W `crates/fullmag-plan/src/fdm.rs` i `crates/fullmag-plan/src/fem.rs` widać, że plan zawiera i przenosi:

- `current_modules`
- parametry STT
- `temperature`
- parametry `OerstedCylinder`

To oznacza, że nie jest to „martwe pole w IR”, tylko coś, co miało być użyte w backendzie wykonawczym.

### 2.3. CPU reference i native FDM mają wyraźne ślady STT

W runnerach znalazłem kilka istotnych faktów:

- `cpu_reference.rs` buduje konfiguracje:
  - `ZhangLiSttConfig`
  - `SlonczewskiSttConfig`
  - `SotConfig`
- `native_fdm.rs` mapuje:
  - `current_density`
  - `stt_degree`
  - `stt_beta`
  - `stt_spin_polarization`
  - `stt_lambda`
  - `stt_epsilon_prime`
  - oraz `OerstedCylinder`

To już nie wygląda jak szkic. To wygląda jak **realnie przygotowany backend path**, tylko jeszcze **słabo wyprowadzony do użytkownika końcowego**.

### 2.4. Anteny FEM i analiza źródła fal spinowych działają jako osobny tor

Publiczne `current_modules` i `AntennaFieldSource` są z kolei powiązane z realnym runnerem FEM dla pola antenowego.

To jest ważne, bo pokazuje, że Fullmag umie już rozwiązać **prąd → pole RF → pobudzenie fal spinowych** w dedykowanym torze.

Dla Twojego zastosowania oznacza to:

- jeśli chcesz badać **spin-wave excitation** niezależnie od STT, repo ma już rozsądny punkt zaczepienia,
- ale architektonicznie jest tu pewien chaos, bo:
  - **OerstedCylinder** siedzi jako **energy term**
  - **RF antenna** siedzi jako **current_module**
  - **STT** siedzi jako **ukryte pola `ProblemIR`**

To są trzy różne ścieżki opisu „prądu”, które powinny docelowo zostać zunifikowane.

Czyli:

> **low-level backend umie lub prawie umie, ale użytkownik nie ma normalnej, oficjalnej klamki do zadania CPP-STT w skrypcie.**

To nie jest drobny detal. To jest główna luka, która dziś blokuje „normalny” workflow STNO.

## 5) Czy obecna fizyka wystarcza do Twojego dokładnego układu?

## 5.1. Dla vortexu 400 nm — tak, jako time-domain micromagnetics

Nanokropka 400 nm z warstwą permalloy-like / soft free layer to klasyczny przypadek, w którym:

- relaksacja do vortexu jest naturalna,
- mod gyrotropowy najlepiej wyciągać z:
  - impulsu,
  - słabego pobudzenia AC,
  - albo ringdown po krótkim kopnięciu.

Na to Fullmag ma już wystarczającą bazę:

- FDM
- relax
- time evolution
- OerstedCylinder
- zapisy `mx, my, mz`
- snapshoty `m`

### 5.2. Dla **MTJ CPP auto-oscillation** — jeszcze nie jako publiczne „gotowe”

Jeżeli przez „modelowanie STNO (MTJ)” rozumiesz dokładnie:

- free layer + reference layer + barrier
- spin polarization z reference layer
- CPP Slonczewski torque
- Oersted bias
- autooscylację
- częstotliwość, orbita, linewidth i sygnał mikrofalowy

to odpowiedź jest:

> **backend wygląda na częściowo gotowy, ale produktowo i publicznie jeszcze nie.**

Brakuje przynajmniej:

1. **publicznego API do STT**
2. **GUI do STT**
3. **diagnostyki STNO**
4. **jawnego modelu stacku MTJ**
5. **spójnej dokumentacji konwencji znaków i parametrów**

### 5.3. Co z CoFeB(1)/NiFe(8)?

Dla tej konkretnej warstwy aktywnej rekomenduję **trzystopniową strategię modelowania**:

#### Poziom 1 — model efektywny
Najpierw uruchom:
- jeden cylinder
- wysokość 9 nm
- efektywne `Ms, A, alpha, Ku`

To pozwoli szybko odpowiedzieć:
- czy vortex się stabilizuje,
- jaka jest częstotliwość gyrotropowa,
- jak wygląda odpowiedź na Oersteda.

#### Poziom 2 — z-dependent material fields
Potem dopiero przejdź na:
- jedną bryłę
- pola materiałowe zmienne po `z`

To pozwoli odróżnić:
- cienki CoFeB z możliwą większą anizotropią,
- gruby NiFe jako główną część vortexową.

#### Poziom 3 — pełny STT + temperatura
Dopiero po zgraniu etapu 1–2:
- włącz STT,
- zrób sweep `J`,
- dopiero potem `T > 0` i linewidth.

To ograniczy liczbę miejsc, w których można „zgubić fizykę” albo stabilność numeryczną.

## 6) Co jest słabe lub niebezpieczne w obecnym stanie

## 6.1. Rozszczepiona semantyka prądu

Aktualnie „prąd” jest reprezentowany trzema różnymi mechanizmami:

1. **`OerstedCylinder`** jako `energy term`
2. **`AntennaFieldSource`** jako `current_module`
3. **STT** jako ukryte pola `ProblemIR`

To jest zła sytuacja z punktu widzenia:
- UX,
- GUI,
- walidacji,
- spójności dokumentacji.

Dopóki to nie zostanie spięte jedną publiczną semantyką, GUI będzie zawsze „niepełne”.

## 6.2. Brak first-class obserwabli STNO

Dzisiaj masz głównie:
- `mx`, `my`, `mz`
- `time`
- `solver_dt`
- `max_h_eff`
- `max_dm_dt`
- pola `m`, `H_eff`, itd.

To wystarcza do:
- ręcznego FFT,
- ręcznego oglądania vortexu,
- eksportu snapshotów

ale nie wystarcza do sensownego modułu STNO, gdzie chcesz od razu mieć:

- PSD
- peak frequency
- linewidth
- orbit radius
- pozycję rdzenia vortexu `(Xc, Yc)`
- topological charge / polarity diagnostics
- fazę chwilową
- amplitude envelope
- ewentualnie `R(t)` / `V(t)` z modelu TMR

## 6.3. GUI nie ma domenowych paneli STNO

W UI jest:

- ogólny builder studiów,
- `study pipeline`,
- analyze workbench nastawiony na **eigenmodes / dispersion**

Nie znalazłem natomiast pełnego, dedykowanego wsparcia dla:

- bias current density
- STT polarization vector
- `lambda`, `epsilon_prime`, `beta`
- Oersted pillar geometry editor
- thermal seed / temperature
- DC sweep + AC overlay
- FFT / PSD / linewidth panel
- vortex core orbit plot
- sygnał mikrofalowy / MTJ voltage

## 6.4. Obecne eigenmodes nie są głównym narzędziem dla vortexu

To jest bardzo ważny punkt.

Dokumenty eigen/FEM mówią wprost, że aktualny solver:

- jest jeszcze MVP / reference-style,
- zachowuje tylko część operatora,
- jest wiarygodniejszy dla stanów **jednorodnych lub prawie jednorodnych**

Vortex jest z definicji:
- **silnie niejednorodny**
- ma topologię i przesunięty rdzeń
- łatwo w nim o problemy przy uproszczonej linearyzacji

Wniosek:

> **Dla vortex STNO nie traktowałbym obecnego eigen-solvera jako głównego narzędzia walidacyjnego.**  
> Podstawą powinien być **time-domain** + FFT + tracking rdzenia.
