# Fullmag — raport diagnostyczny modułu tekstur magnetycznych (vortex / skyrmion / transform / mapping)

> Stan audytu: analiza publicznego `master` repozytorium `MateuszZelent/fullmag` na dzień 2026-04-10.  
> Zakres: **frontend + authoring + runtime/backend Python** dla `preset_texture`, ze szczególnym naciskiem na:  
> `vortex`, `bloch_skyrmion`, `neel_skyrmion`, translację/rotację/skalę/pivot tekstury względem obiektu oraz round-trip przez SceneDocument / builder.

---

## 1. Werdykt

Najważniejszy wniosek jest prosty:

**To nie wygląda na pojedynczy zły wzór vorteksu albo skyrmionu.**  
Problem jest znacznie głębszy: moduł został wdrożony na **kilku niespójnych kontraktach danych naraz**. W efekcie:

1. część ścieżek używa `params`,
2. część używa `preset_params`,
3. część używa `rotation_quat`,
4. część używa `rotationQuat`,
5. część authoringu w ogóle **gubi** preset, mapping i transform podczas round-tripu,
6. runtime ma **niejednoznaczną semantykę skali** tekstury,
7. część obiecanych projekcji mappingu jest dziś tylko atrapą.

W praktyce oznacza to, że nawet jeśli sam evaluator presetu jest sensowny matematycznie, to do niego bardzo często **nie docierają poprawne parametry i poprawne współrzędne lokalne**.

### Krótko: co zrobiliśmy źle

Największe błędy są cztery:

- **Błąd kontraktu pól:** `params` vs `preset_params`.
- **Błąd round-tripu authoringowego:** builder/adapters tracą preset, mapping i transform.
- **Błąd kontraktu transformacji:** `rotation_quat` vs `rotationQuat`.
- **Błąd semantyki współrzędnych:** jednocześnie używamy parametrów presetu w metrach i „teksturo-podobnej” przestrzeni lokalnej `[-0.5, 0.5]`, co bez bardzo jasnej konwersji rozwala skalę vorteksu/skyrmionu.

To wystarcza, żeby wynik „kompletnie nie przypominał tego co powinien”.

---

## 2. Co działa, a co nie

## 2.1. Co wydaje się **sensowne**

Sam rdzeń wzorów w `preset_eval.py` nie wygląda na główny problem:

- `vortex` używa kąta azymutalnego `phi = atan2(py, px)`, in-plane cyrkulacji i gaussowskiego rdzenia `mz`,
- `bloch_skyrmion` i `neel_skyrmion` są rozróżnione przez inną helicity (`π/2` vs `0`),
- plane switching `xy/xz/yz` jest zaimplementowany,
- wyniki są normalizowane.

To znaczy: **sam evaluator nie wygląda na „absurdalny”**.  
Diagnoza wskazuje raczej na **błędy w przesyłaniu parametrów i transformacji**, a nie na samą analityczną definicję pola.

## 2.2. Co jest realnie uszkodzone

Realnie uszkodzony jest cały tor:

`frontend state -> SceneDocument -> Rust authoring -> builder -> runtime initial_state -> evaluator`

i to w kilku miejscach jednocześnie.

---

## 3. Docelowa architektura vs stan rzeczywisty

## 3.1. Co plan zakładał

Plan wdrożenia tekstur był sensowny:

- Python DSL miał generować `preset_texture`,
- runtime miał próbkuwać teksturę dopiero tuż przed solverem,
- Rust authoring miał być **pass-through**,
- frontend miał operować na SceneDocument i edytować `texture_transform` oraz `preset_params`,
- `TextureTransformGizmo` miał sterować wyłącznie teksturą, nie geometrią,
- round-trip JSON miał zachowywać `kind="preset_texture" + preset_params`.

To jest dobra architektura.

## 3.2. Co jest dziś w kodzie

Dziś w kodzie masz **co najmniej trzy konkurencyjne schematy** dla tego samego bytu:

### Schemat A — Python DSL / bezpośredni IR
```python
{
  "kind": "preset_texture",
  "preset_kind": "...",
  "params": {...},
  "mapping": {...},
  "texture_transform": {...}
}
```

### Schemat B — SceneDocument / Rust authoring / UI asset
```json
{
  "kind": "preset_texture",
  "preset_kind": "...",
  "preset_params": {...},
  "mapping": {...},
  "texture_transform": {...}
}
```

### Schemat C — frontendowy pomocniczy stan edytora
```ts
{
  preset_kind: "...",
  params: {...}
}
```

To jest dokładnie typ architektury, który prawie zawsze kończy się regresją:
- jedne ścieżki zapisują,
- drugie czytają pod inną nazwą,
- trzecie robią własny lokalny mapping.

## 4. Diagnostyka szczegółowa

## 4.1. Krytyczny błąd nr 1 — `params` vs `preset_params`

To jest **najbardziej niebezpieczny błąd kontraktu**.

### Co robi Python DSL

`PresetTexture.to_ir()` produkuje pole:
```python
"params": dict(self.params)
```

### Co robi runtime sampler

`prepare_initial_magnetization(...)` dla `kind == "preset_texture"` czyta:
```python
params = dict(spec.get("params", {}))
```

czyli runtime oczekuje **`params`**.

### Co robi SceneDocument

`scene_document.py` zapisuje i odczytuje:
```python
"preset_params": ...
```

czyli authoring/scene/UI używa **`preset_params`**.

### Dlaczego to jest katastrofalne

Jeżeli ścieżka przechodzi przez SceneDocument / magnetization asset / builder i nigdzie po drodze nie ma jawnego mapowania
`preset_params -> params`,
to runtime sampler dostaje pusty słownik parametrów.

Skutki:
- dla `vortex` może wejść fallback `core_radius = 1e-9`,
- dla `skyrmion` evaluator oczekuje `radius` i `wall_width` jako wymagane; ich brak jest błędem logicznym,
- nawet jeśli coś nie crashuje, to profil będzie zły, bo użyje nie tych wartości, których oczekiwał użytkownik.

### Wniosek

Masz tu **nie jeden format IR, tylko dwa**.  
To trzeba natychmiast zunifikować.

---

## 4.2. Krytyczny błąd nr 2 — Rust authoring **gubi** preset texture podczas round-tripu

To jest drugi bardzo ciężki problem.

### Co jest dobre

W `scene.rs` `MagnetizationAsset` ma już sensowne pola:
- `mapping`
- `texture_transform`
- `preset_kind`
- `preset_params`
- `preset_version`
- `ui_label`

czyli sam model SceneDocument jest przygotowany.

### Co jest złe

`ScriptBuilderMagnetizationState` w `builder.rs` nadal ma tylko stare pola:
- `kind`
- `value`
- `seed`
- `source_path`
- `source_format`
- `dataset`
- `sample_index`

i **nie ma**:
- `mapping`
- `texture_transform`
- `preset_kind`
- `preset_params`
- `preset_version`
- `ui_label`

To oznacza, że builder state **nie umie być nośnikiem preset texture** w pełnym sensie.

### Jeszcze gorzej: adaptery to kasują ręcznie

W `adapters.rs` funkcja konwersji builder -> asset ustawia:
- `mapping: default()`
- `texture_transform: default()`
- `preset_kind: None`
- `preset_params: None`
- `preset_version: None`
- `ui_label: None`

czyli zamiast „przepuścić” preset texture dalej, **czyści go do zera**.

A funkcja konwersji asset -> builder także odczytuje tylko legacy fields i nie zwraca presetowych danych z powrotem.

### Konsekwencja

Każdy round-trip typu:
- scene -> builder,
- builder -> scene,
- UI mutation -> adapter -> builder -> scene,

może spowodować:
- utratę rodzaju presetu,
- utratę parametrów,
- utratę mappingu,
- utratę transformacji tekstury.

To jest **wprost sprzeczne** z założeniem planu, że Rust authoring ma być pass-through.

### Dodatkowy sygnał ostrzegawczy

W `adapters.rs` istnieje poprawny helper `magnetization_asset_for_preset(...)`, który tworzy asset z:
- `kind = "preset_texture"`
- `preset_kind = Some(...)`
- `preset_params = Some(...)`
- `mapping`
- `texture_transform`

ale jest on odseparowany od głównej ścieżki i nie naprawia round-tripu.

### Wniosek

Authoring nie jest jeszcze „preserve-all-data”.  
On jest dziś **„truncate-to-legacy-fields”**.

---

## 4.3. Krytyczny błąd nr 3 — `rotation_quat` vs `rotationQuat`

To jest bardzo konkretna, bardzo niebezpieczna niespójność frontendowa.

### W runtime / scene / session

W warstwie scene/session używasz snake_case:
```ts
rotation_quat
```

### W helperze frontendowym i gizmo

`apps/web/lib/textureTransform.ts` definiuje transform jako:
```ts
rotationQuat
```

`TextureTransformGizmo.tsx` także pracuje na:
```ts
transform.rotationQuat
```

### Dlaczego to jest problem

Jeżeli dowolny kod poda do gizmo obiekt sesyjny / scene-documentowy z polem `rotation_quat`,
to:
- gizmo nie zobaczy rotacji,
- może utworzyć quaternion z `undefined`,
- albo zadziała tylko przez niejawne/ad-hoc adaptery,
- albo zapisze z powrotem inny kształt danych niż oczekuje backend.

To jest klasyczny przypadek błędu „UI niby coś edytuje, ale backend tego nie widzi”.

### Wniosek

Dla transformacji tekstury musisz mieć **jeden canonical shape** w całej aplikacji.  
Dzisiaj go nie masz.

---

## 4.4. Krytyczny błąd nr 4 — round-trip obiektu zachowuje tylko translację

Tekstura „względem obiektu” wymaga poprawnej informacji o transformacji obiektu.

### Co robi SceneDocument builder

`build_scene_document_from_builder(...)` buduje `object.transform` tak:
- `translation` z geometrii,
- `rotation_quat = identity`,
- `scale = [1,1,1]`,
- `pivot = [0,0,0]`.

Czyli nawet jeśli obiekt był logicznie obracany / skalowany w UI, ta informacja nie przechodzi tutaj dalej jako prawdziwy object transform.

### Co robi runtime sampler

`prepare_initial_magnetization(...)` używa `object_transform` przy `mapping.space == "object"`.

### Co robi `problem.py`

`_object_transform_for_magnet_geometry(...)` składa object transform praktycznie tylko z translacji `Translate(...)` oraz identity rotation/scale/pivot.

### Konsekwencja

„Object space” w praktyce oznacza dziś głównie:
- przesunięcie,
- ale **nie pełną transformację obiektu**.

Jeżeli użytkownik oczekuje:
- rotacji tekstury razem z obiektem,
- skali względem obiektu,
- poprawnego pivotu obiektowego,

to obecna ścieżka tego nie dowozi.

### Wniosek

Semantyka „texture relative to object” jest dziś **niepełna**.  
Masz tylko ułamek tego, co użytkownik rozumie przez „względem obiektu”.

## 4.5. Krytyczny błąd nr 5 — błędna matematyka `inverse transform` dla pivot + scale

To jest błąd stricte numeryczny.

### Co robi runtime

W `initial_state.py` inverse transform działa w przybliżeniu tak:

1. odejmij `translation` i `pivot`,
2. zastosuj odwrotną rotację,
3. **dodaj pivot z powrotem**,
4. dopiero potem podziel przez `scale`.

### Dlaczego to jest podejrzane

Jeżeli forward transform jest:
```text
world = translation + pivot + R * ( S * (local - pivot) )
```

to poprawna odwrotność ma postać:
```text
local = pivot + S^-1 * R^-1 * (world - translation - pivot)
```

Kluczowa różnica:
- `pivot` powinien wrócić **po inverse scale**,
- a nie przed inverse scale.

### Co to psuje

Jeśli:
- `pivot != 0`
- i `scale != 1`

to tekstura będzie:
- przesunięta nie tam gdzie trzeba,
- rozciągnięta nie wokół właściwego punktu,
- przy rotacji wokół pivotu będzie zachowywać się niefizycznie.

### Objaw użytkowy

„Przesuwam / obracam / skaluję teksturę i wychodzi dziwnie”  
— dokładnie taki objaw bardzo dobrze pasuje do tej klasy błędu.

### Wniosek

Nawet po naprawie schematu danych transformacja nadal będzie dawała błędy, dopóki inverse math nie zostanie poprawiona.

---

## 4.6. Krytyczny błąd nr 6 — niejednoznaczna semantyka skali tekstury

To jest być może **najbardziej podstępny** błąd całego modułu.

### Co mówi frontend

`magnetizationPresetCatalog.ts` opisuje parametry takie jak:
- `core_radius`
- `radius`
- `wall_width`
- `center_offset`

w **jednostkach SI**, zwykle `m`.

Czyli UI komunikuje użytkownikowi:
> To są rozmiary fizyczne, np. 35e-9 m.

### Co robi runtime

W `initial_state.py` po inverse transform stosujesz clamp/repeat/mirror w przestrzeni:
```python
[-0.5, 0.5]
```

To bardzo mocno przypomina znormalizowaną **unit-box texture space**.

### Co robi helper frontendowy

`fitTextureToBounds(...)` ustawia:
- `translation = center(bounds)`
- `scale = full object extents`

To sugeruje, że texture transform jest używane jak klasyczne dopasowanie tekstury do bounding boxa.

### Gdzie powstaje konflikt

Jeżeli:
- lokalne współrzędne po inverse transform są **dzielone przez scale**,
- a `scale` jest rozmiarem obiektu,
- a parametry presetu `radius`, `wall_width`, `core_radius` pozostają w **metrach**,

to porównujesz:
- współrzędne **znormalizowane do rozmiaru obiektu**
z
- promieniami zapisanymi w **metrycznych jednostkach świata**.

To są dwie różne semantyki naraz.

### Konsekwencja fizyczna

Dla obiektu rzędu setek nanometrów:
- punkty po dopasowaniu do bounds będą rzędu `0.1 ... 0.5`,
- a promień skyrmionu będzie rzędu `35e-9`.

Wtedy profil radialny będzie kompletnie nie ten:
- skyrmion może stać się prawie jednolity,
- rdzeń vortexu może się zapadać do jednego niemal niewidocznego punktu,
- wall width może być efektywnie zerowe.

### Wniosek

Musisz wybrać **jedną** z dwóch semantyk:

### Semantyka A — fizyczna (zalecana)
- sample points są w metrach,
- texture transform działa też w metrach,
- `radius`, `wall_width`, `core_radius` są w metrach,
- clamp/repeat operują na fizycznej przestrzeni lub są całkiem wyłączone dla presetów topologicznych.

### Semantyka B — znormalizowana
- lokalne współrzędne są w unit-box space,
- wtedy parametry presetu też muszą być w tej samej przestrzeni,
- UI nie może wtedy opisywać ich jako „m”.

Dziś w kodzie występują ślady **obu naraz**, co jest błędem architektonicznym.

## 4.7. Błąd nr 7 — projekcje mappingu są głównie atrapą

`initial_state.py` realnie implementuje tylko:
- `planar_xy`
- `planar_xz`
- `planar_yz`

Natomiast:
- `box`
- `triplanar`
- `cylindrical`
- `spherical`

są dziś komentowane jako przypadki, które „na razie zostawiają współrzędne bez zmian”.

To oznacza, że jeśli frontend pokazuje lub sugeruje te tryby, to użytkownik dostaje obietnicę funkcji, która realnie jest **no-opem**.

### Wniosek

Albo:
- to implementujesz,
albo:
- ukrywasz z UI,
albo:
- oznaczasz jako `preview only / not implemented`.

Dziś jest najgorsza opcja: pole istnieje, ale semantyka nie jest domknięta.

## 4.8. Błąd nr 8 — frontend sam ma dwa różne modele stanu presetu

W `apps/web/lib/session/types.ts` współistnieją:

### `ScriptBuilderMagnetizationEntry`
z polami:
- `preset_kind`
- `preset_params`
- `texture_transform`

i równolegle:

### `MagnetizationPresetState`
z polami:
- `preset_kind`
- `params`

To oznacza, że już w samym froncie masz:
- model assetowy z `preset_params`
- model edytora z `params`

czyli problem `params` vs `preset_params` nie jest tylko między Pythonem a Rustem.  
On jest obecny **już w samym TypeScript**.

### Wniosek

To nie jest przypadkowy bug.  
To jest systemowy brak jednego canonical modelu danych.

## 4.9. Błąd nr 9 — `MagneticTextureLibraryPanel` nadal jest zbyt „starterowy”

Plan wprost mówił, że panel biblioteki ma być podpięty pod mutation API SceneDocument, a nie pod ogólny callback.

Aktualny panel nadal jest bardzo cienki:
- `selectedKind`
- `onCreatePreset(kind)`
- `onSelectKind(kind)`

czyli sam panel nie narzuca jeszcze:
- spójnego tworzenia assetu,
- spójnego zapisu `preset_params`,
- spójnego zapisu `texture_transform`,
- walidacji typu presetu.

To zwiększa ryzyko, że każda ścieżka nadrzędna robi własne mapowanie „po swojemu”.

### Wniosek

Frontend integracyjnie jest nadal **za luźny** jak na tak delikatny kontrakt danych.

## 4.10. Dodatkowy sygnał ostrzegawczy — niespójności katalogu presetów względem backendu

Nawet poza vortex/skyrmion widać, że cały moduł nie jest jeszcze w pełni zdyscyplinowany.

### Najczytelniejszy przykład: `two_domain`

W katalogu frontendowym `two_domain` używa nazw:
- `m_left`
- `m_right`
- `wall_width`
- `wall_center`

Natomiast Pythonowy factory/evaluator `two_domain` oczekuje:
- `left`
- `right`
- `wall`
- `normal_axis`

To jest jawny mismatch nazw parametrów.

To ważny sygnał, bo pokazuje, że problem nie dotyczy wyłącznie jednego presetowego kształtu — **cały system presetów nie ma jeszcze jednego, wspólnego schematu parametrów**.

## 5. Dlaczego wynik „nie przypomina tego co powinno”

Teraz można złożyć objawy w całość.

## 5.1. Dlaczego rotacja wydaje się nie działać
Bo część frontendu używa `rotationQuat`, a runtime/scene używają `rotation_quat`.

## 5.2. Dlaczego przesuwanie względem obiektu jest dziwne
Bo `object space` nie niesie pełnego transformu obiektu — głównie translację.

## 5.3. Dlaczego skala rozjeżdża teksturę
Bo mieszasz:
- metryczne promienie skyrmionu/vortexu
z
- unit-box style coordinate normalization.

## 5.4. Dlaczego po zapisaniu / przeładowaniu / zmianie zaznaczenia wszystko może się „psuć”
Bo adaptery authoringowe gubią:
- `preset_kind`
- `preset_params`
- `mapping`
- `texture_transform`

## 5.5. Dlaczego część presetów może działać „trochę”, a część „wcale”
Bo:
- vortex ma sensowne fallbacki,
- skyrmion wymaga obowiązkowych parametrów,
- część presetów ma zgodne nazwy parametrów,
- część nie.

## P1 — naprawa semantyki i funkcjonalności

### P1.1. Zdecydować: metryczna czy znormalizowana przestrzeń tekstury
To jest decyzja architektoniczna.

Dla presetów magnetycznych typu vortex/skyrmion rekomenduję **przestrzeń metryczną**:
- promienie w metrach,
- offsety w metrach,
- transform tekstury w metrach,
- object-local tylko zmienia frame odniesienia, nie jednostki.

### P1.2. `fitTextureToBounds(...)`
Jeżeli zostajesz przy metrycznej semantyce presetów topologicznych, to helper „fit to bounds” nie może po prostu narzucać unit-box style scaling bez jednoczesnej zmiany semantyki parametrów.

Masz trzy opcje:
1. usunąć to dla presetów topologicznych,
2. zmienić nazwę i jasno opisać, że to przechodzi do normalized mode,
3. automatycznie przeliczać również preset params.

### P1.3. Projekcje mappingu
Albo implementujesz:
- `box`
- `triplanar`
- `cylindrical`
- `spherical`

albo ukrywasz je w UI do czasu pełnej implementacji.

### P1.4. Zachować pełny object transform
Nie tylko translation.  
Jeżeli użytkownik edytuje obiekt w scenie, texture/object relation musi to widzieć.

## P2 — dopracowanie frontendu

### P2.1. Library panel nie może być tylko generatorem callbacków
Panel powinien:
- tworzyć canonical preset asset,
- podstawiać domyślne parametry,
- synchronizować `preset_kind`, `preset_params`, `texture_transform`,
- walidować zgodność z katalogiem.

### P2.2. Jedna ścieżka edycji
Frontend nie może mieć osobnego lokalnego `MagnetizationPresetState.params` bez jawnego, testowanego mapera do canonical assetu.

### P2.3. Preview proxy i preset schema
`previewProxy` oraz schema parametrów muszą być zgodne z backendem.  
Jeśli preset `two_domain` ma inne parametry w UI i inne w evaluatorze, preview i solver nigdy nie będą spójne.

## 9. Moja końcowa ocena

Mój końcowy osąd jest taki:

### Sama idea modułu jest dobra
Architektura z:
- analytic preset,
- runtime sampling,
- scene asset,
- texture transform,
- mapping space,

jest sensowna i rozwojowa.

### Ale implementacja jest dziś niespójna systemowo
Największe błędy nie siedzą w samym wzorze skyrmionu, tylko w tym, że:
- frontend,
- authoring,
- scene document,
- runtime

**nie mówią jeszcze tym samym językiem**.

### Najbardziej prawdopodobna główna przyczyna obecnego objawu
Jeśli mam wskazać jedną najważniejszą klasę przyczyny, to jest nią:

> **utrata lub deformacja informacji o presecie i jego transformacji zanim evaluator dostanie poprawne lokalne współrzędne.**

Innymi słowy:
- nie tyle „zły skyrmion”,
- tylko „skyrmion policzony na złych danych wejściowych”.

## 10. Najkrótsza rekomendacja naprawcza

Gdybym miał to naprawiać od razu, zrobiłbym tylko te 5 rzeczy, w tej kolejności:

1. **Zunifikować schema:** `preset_params` / `texture_transform` / `mapping`.
2. **Naprawić round-trip Rust authoring:** builder + adapters muszą być lossless.
3. **Ujednolicić `rotation_quat` w całym stacku.**
4. **Naprawić inverse transform math.**
5. **Zdefiniować jedną semantykę przestrzeni tekstury** i dostosować do niej `fitTextureToBounds`.

Dopiero po tym wracałbym do polishu UI.
