# Raport diagnostyczny: vortex texture, texture transform i konflikt gizmo/kamera

**Data:** 2026-04-10  
**Repozytorium:** `MateuszZelent/fullmag`  
**Zakres:**
- problem 1 — błędna tekstura magnetyczna dla domyślnego presetu `vortex`
- problem 2 — brak poprawnego fizycznego przesuwania/obracania środka tekstury
- problem 3 — konflikt sterowania kamerą i gizmo translacji/rotacji/skali

---

## 1. Status końcowy

### Wniosek główny
Obecna implementacja nie jest jeszcze spójna end-to-end.

Najważniejsze źródła błędu są **trzy**:

1. **Transformacja tekstury istnieje w modelu danych i GUI, ale nie jest stosowana w evaluatorze presetów.**
   To oznacza, że translacja/rotacja/skala/pivot tekstury nie przeliczają fizycznie orientacji magnetyzacji w węzłach.

2. **Gizmo tekstury działa w układzie współrzędnych sceny 3D, a nie w kanonicznym układzie symulacji/obiektu.**
   W preview FDM osie sceny są jawnie przemieszane (`scene X = sim X`, `scene Y = sim Z`, `scene Z = sim Y`), natomiast gizmo tekstury zapisuje translację i kwaternion bez konwersji bazy.

3. **Warstwa interakcji viewportu nie ma twardego mechanizmu arbitrażu wejścia między manipulatorem i kamerą.**
   Część gizmo/overlay już dziś może nakładać ruch kamery na ruch obiektu.

### Konsekwencja praktyczna
To, co użytkownik interpretuje jako „źle narysowany vortex”, najprawdopodobniej nie jest jednym błędem, tylko **sumą dwóch klas problemów**:

- **błąd fizyczny / samplingowy** — preset jest liczony w złym układzie punktów lub bez zastosowania `texture_transform`,
- **błąd UI / basis mapping** — gizmo i preview operują na współrzędnych sceny, podczas gdy model tekstury powinien być przechowywany w kanonicznym układzie obiektu/symulacji.

## 2. Ograniczenia diagnozy

Nie miałem w tej rozmowie dostępnego pliku obrazu ze screenem jako osobnego załącznika do bezpośredniej analizy piksel-po-pikselu. Dlatego diagnoza opiera się na:

- aktualnym stanie publicznego kodu repozytorium,
- opisanych przez Ciebie objawach,
- powiązaniach między warstwą GUI, stanem sesji, reprezentacją `texture_transform` i Pythonowym evaluatorem presetów.

To wystarcza do wskazania bardzo mocnych kandydatów na root-cause i do przygotowania konkretnego planu naprawy.

## 3. Materiał dowodowy — co dokładnie w kodzie już dziś wskazuje na błąd

### 3.1. UI i model sesji zakładają, że tekstura ma pełny transform
W warstwie frontendu magnetyzacja typu `preset_texture` ma:

- `mapping`
- `texture_transform`
- `preset_kind`
- `preset_params`

Dodatkowo UI udostępnia użytkownikowi:

- `translation`
- `rotation_quat`
- `scale`
- `pivot`
- `fit to object`
- tryby gizmo `translate / rotate / scale`

To znaczy, że interfejs użytkownika **obiecuje** pełną transformację tekstury.

### 3.2. Akcja `assignMagneticPreset(...)` automatycznie dopasowuje teksturę do obiektu
Przy pierwszym przypisaniu niejednorodnego presetu wykonywane jest `fitTextureToObject(...)`, które ustawia:

- `translation = center(bounds)`
- `scale = extent(bounds)`

czyli domyślny preset tekstury ma być osadzony w obiekcie, a nie w globalnym `(0,0,0)`.

### 3.3. Pythonowy evaluator presetów ignoruje `mapping` i `texture_transform`
Aktualny evaluator:

```python

def evaluate_preset_texture(
    preset_kind: str,
    params: Mapping[str, object],
    points: Iterable[Sequence[float]],
) -> EvaluatedTexture:
```

przyjmuje tylko:

- `preset_kind`
- `params`
- `points`

Nie przyjmuje:

- `mapping`
- `texture_transform`
- `object/world frame`

To oznacza, że preset jest liczony na „surowych” punktach bez uwzględniania translacji, rotacji, skali i pivotu tekstury.

### 3.4. Sam vortex liczony jest wokół zera układu lokalnego funkcji
Implementacja:

```python
px, py, pz = _plane_coords(point, str(params.get("plane", "xy")))
phi = math.atan2(py, px)
r = math.hypot(px, py)
```

To znaczy:

- środek vortexu jest implicitnie w `(0,0)` wybranej płaszczyzny,
- nie ma żadnego `center`,
- nie ma zastosowania `translation`,
- nie ma inverse-rotation,
- nie ma inverse-scale,
- nie ma pivotu.

### 3.5. Preview FDM działa w innej bazie osi niż kanoniczne `(x,y,z)`
W preview 3D znajduje się jawny komentarz i implementacja mapowania osi:

- `scene-X = sim-X`
- `scene-Y = sim-Z`
- `scene-Z = sim-Y`

Wektor magnetyzacji również jest mapowany jako `(mx, mz, my)`.

To samo dotyczy położeń komórek/voxelów.

### 3.6. Overlay geometrii i anten już dziś wykonują ręczne mapowanie scena -> fizyka
W `MagnetizationView3D.tsx` translacja overlayu geometrii i anten podczas `onDragEnd` jest konwertowana jawnie:

- `physicalDx = p.x * cellX`
- `physicalDz = p.y * cellZ`
- `physicalDy = p.z * cellY`
- a dalej przekazywana jako `(dx, dy, dz)` po zamianie osi

To jest mocny dowód, że preview operuje w innej bazie niż warstwa fizyczna.

### 3.7. Gizmo tekstury nie wykonuje takiej konwersji
`TextureTransformGizmo.tsx` przy commit/live-change zapisuje bezpośrednio:

- `translation = [group.position.x, group.position.y, group.position.z]`
- `rotation = group.quaternion`
- `scale = group.scale`

bez żadnej konwersji scena -> symulacja/obiekt.

To jest **bardzo prawdopodobny root-cause błędu współrzędnych** dla przesuwania/obracania tekstury.

### 3.8. Dodatkowo warstwa transformów ma niespójne nazewnictwo kwaternionu
W przejrzanych plikach:

- model sesji i utilsy używają `rotation_quat`,
- `TextureTransformGizmo.tsx` odwołuje się do `rotationQuat`.

To jest osobny, krytyczny problem kontraktu danych. Nawet jeśli lokalnie część tego została już naprawiona, publiczny stan repo pokazuje, że ta warstwa była niespójna.

## 4. Diagnoza problemu 1 — dlaczego domyślny vortex wygląda źle

### 4.1. Co powinno się dziać
Jeżeli użytkownik wybiera preset `vortex`, a system automatycznie robi `fitTextureToObject`, to dla każdego węzła/voxela powinno następować:

1. wybór punktu bazowego w odpowiedniej przestrzeni (`object` albo `world`),
2. projekcja zgodna z `mapping.projection`,
3. zastosowanie **odwrotnej transformacji tekstury**,
4. dopiero potem obliczenie analitycznej magnetyzacji vortex/skyrmion/domain wall,
5. normalizacja wektora magnetyzacji.

### 4.2. Co dzieje się dziś
Najprawdopodobniej:

- `fitTextureToObject` wpisuje poprawny środek i skalę do `texture_transform`,
- ale evaluator presetów tego **nie używa**,
- więc vortex pozostaje zakotwiczony w `(0,0)` surowych współrzędnych próbkowania,
- a nie w środku obiektu.

Jeżeli obiekt nie jest wycentrowany w układzie globalnym albo preview korzysta z innej bazy osi, wynik wygląda jak „zepsuty vortex”.

### 4.3. Najmocniejsza hipoteza root-cause
**Domyślny vortex jest błędny, bo domyślne „fit to object” zapisuje transform tekstury, ale evaluator fizyczny go ignoruje.**

To jest bardzo mocna hipoteza, bo idealnie tłumaczy obserwację:

- UI pokazuje, że tekstura ma centrum i rozmiar,
- użytkownik oczekuje vortexu osadzonego w nanokropce,
- ale pole magnetyzacji jest liczone bez tego osadzenia.

### 4.4. Druga hipoteza uzupełniająca
**Nawet po naprawie samego evaluatora nadal trzeba skorygować mapowanie bazy sceny 3D do bazy fizycznej.**

W przeciwnym razie:

- drag X/Y/Z w gizmo nie będzie odpowiadał fizycznemu X/Y/Z,
- obrót będzie zapisany w złej bazie,
- pivot i skala również będą interpretowane niezgodnie z osią symulacji.

## 5. Diagnoza problemu 2 — przesuwanie/rotacja środka tekstury nie przelicza magnetyzacji fizycznie

### 5.1. Wymaganie funkcjonalne
Masz rację co do oczekiwanego zachowania.

Dla tekstury magnetycznej typu analitycznego obowiązuje reguła:

> Transformacja tekstury nie może być tylko wizualnym przesunięciem proxy/gizmo.  
> Musi zmieniać punkt próbkowania tekstury dla **każdego węzła** i przez to zmieniać wynikową orientację magnetyzacji.

Czyli po przesunięciu tekstury o `Δx` rdzeń vortexu rzeczywiście ma się przesunąć w nanokropce; po rotacji o `θ` wzór pola ma się obrócić; po zmianie skali ma się zmienić rozkład w przestrzeni.

### 5.2. Poprawny model matematyczny
Dla kanonicznego punktu próbkowania `x_c` należy policzyć punkt w przestrzeni tekstury `x_t` przez **odwrotną transformację**:

```text
x_t = p + S^{-1} R^{-1} (x_c - t - p)
```

Gdzie:

- `t` — translacja tekstury,
- `R` — macierz rotacji z kwaternionu,
- `S = diag(sx, sy, sz)` — skala,
- `p` — pivot.

Dopiero `x_t` trafia do funkcji typu `_vortex`, `_skyrmion`, `_domain_wall`, `_helical`, itd.

### 5.3. Co to oznacza implementacyjnie
Dla każdego węzła:

1. pobierasz punkt bazowy `x_base`,
2. w zależności od `mapping.space` wybierasz przestrzeń `object` albo `world`,
3. w zależności od `mapping.projection` wybierasz odpowiednią projekcję (`object_local`, `planar_xy`, `planar_xz`, `planar_yz`),
4. liczysz `x_t` wzorem inverse-transform,
5. liczysz `m(x_t)` z presetu,
6. normalizujesz.

### 5.4. Wniosek
Aktualny system **nie spełnia** jeszcze tej semantyki w evaluatorze presetów.

Zatem drugi problem nie jest tylko bugiem UI. To jest brakujący kawałek modelu fizycznego.

## 6. Diagnoza problemu 3 — konflikt kamera vs gizmo

### 6.1. Co już jest zrobione
W `MagnetizationView3D.tsx` istnieje tryb interakcji inspirowany 3ds Max:

- `camera`
- `move`
- `rotate`
- `scale`

Dla tekstury, jeśli tryb nie jest `camera`, `TrackballControls` są wyłączane przez:

```tsx
cameraEnabled={cameraActive && viewportVisible}
```

oraz sam `TextureTransformGizmo` jest renderowany tylko dla `!cameraActive`.

To znaczy, że dla **samego teksturowego gizmo** była już próba rozwiązania problemu.

### 6.2. Gdzie nadal jest dziura
Overlay geometrii i overlay anten używają `PivotControls`, ale nie widać tam pełnego handshake typu:

- `onDragStart -> disable camera`
- `onDragEnd -> enable camera`
- `pointer capture / interaction lock`

To oznacza, że część manipulatorów może nadal współpracować z aktywną kamerą.

### 6.3. Co to daje jako objaw
Dokładnie to, co opisujesz:

- klikasz strzałkę translacji,
- manipulator zaczyna się ruszać,
- ale jednocześnie reaguje kamera,
- więc użytkownik nie ma pewności, czy przesunął obiekt, czy viewport.

### 6.4. Wniosek
Potrzebny jest **jeden wspólny mechanizm blokady interakcji viewportu**, a nie lokalne warunki rozsiane po komponentach.

## 8. Plan refaktoryzacji i naprawy

Poniżej jest plan w kolejności, która minimalizuje ryzyko i pozwala szybko uzyskać poprawne zachowanie dla STNO/MTJ i innych tekstur.

#### A1. Ustalić, że kanoniczny model transformacji tekstury jest przechowywany w osi symulacji/obiektu
**Rekomendacja:**
- przechowywać `texture_transform` w bazie **kanonicznej fizyki**, nie w bazie sceny WebGL,

#### A2. Ustalić jedno nazewnictwo pól
**Rekomendacja:**

  - `translation`

#### A3. Naprawić wszystkie odwołania do `rotationQuat`
**Pliki do zmiany:**
- `apps/web/components/preview/TextureTransformGizmo.tsx`
- `apps/web/components/preview/MagnetizationView3D.tsx` (jeżeli gdziekolwiek zakłada camelCase)
- każdy helper preview pracujący na transformie tekstury

#### A4. Dodać test kontraktu typu / round-trip
**Cel:**
- każdy transform przechodzi `session -> preview adapter -> session` bez utraty informacji.

### Etap B — wydzielić wspólną warstwę transformacji scena <-> symulacja

#### B1. Dodać nowy moduł adaptera osi
**Nowy plik rekomendowany:**
- `apps/web/lib/preview/coordinateFrames.ts`

**Funkcje:**

```ts
export type Vec3 = [number, number, number];
export type Quat = [number, number, number, number];

export function simVecToScene(v: Vec3): Vec3;
export function sceneVecToSim(v: Vec3): Vec3;
export function simQuatToScene(q: Quat): Quat;
export function sceneQuatToSim(q: Quat): Quat;
export function simScaleToScene(v: Vec3): Vec3;
export function sceneScaleToSim(v: Vec3): Vec3;
```

#### B2. Zdefiniować bazę permutacji osi jawnie
Dla obecnego FDM preview:

```text
scene = B * sim
B = [[1,0,0],
     [0,0,1],
     [0,1,0]]
```

Ponieważ `B^{-1} = B^T = B`, konwersja wektora jest po prostu zamianą `y <-> z`.

#### B3. Zaimplementować konwersję kwaternionu przez koniugację macierzy rotacji

```ts
R_scene = B * R_sim * B^T
R_sim   = B^T * R_scene * B
```

**Nie wolno** przepisywać kwaternionu 1:1 między sceną i fizyką.

#### B4. Dodać testy losowe round-trip dla 100–1000 transformacji
**Warunek akceptacji:**
- błąd translacji/skali/pivotu < `1e-9`
- błąd macierzy rotacji po round-trip < `1e-9`

## 12. Przykładowa implementacja — inverse transform w Pythonie

```python
# packages/fullmag-py/src/fullmag/init/texture_sampling.py
from __future__ import annotations

import math
from typing import Mapping, Sequence

Vec3 = tuple[float, float, float]
Quat = tuple[float, float, float, float]


def _quat_conj(q: Sequence[float]) -> Quat:
    return (-float(q[0]), -float(q[1]), -float(q[2]), float(q[3]))


def _quat_mul(a: Sequence[float], b: Sequence[float]) -> Quat:
    ax, ay, az, aw = map(float, a)
    bx, by, bz, bw = map(float, b)
    return (
        aw * bx + ax * bw + ay * bz - az * by,
        aw * by - ax * bz + ay * bw + az * bx,
        aw * bz + ax * by - ay * bx + az * bw,
        aw * bw - ax * bx - ay * by - az * bz,
    )


def _rotate_vec(q: Sequence[float], v: Sequence[float]) -> Vec3:
    pure = (float(v[0]), float(v[1]), float(v[2]), 0.0)
    qr = _quat_mul(_quat_mul(q, pure), _quat_conj(q))
    return (qr[0], qr[1], qr[2])


def inverse_apply_texture_transform(
    point: Sequence[float],
    transform: Mapping[str, object] | None,
) -> Vec3:
    if not transform:
        return (float(point[0]), float(point[1]), float(point[2]))

    t = transform.get("translation", (0.0, 0.0, 0.0))
    q = transform.get("rotation_quat", (0.0, 0.0, 0.0, 1.0))
    s = transform.get("scale", (1.0, 1.0, 1.0))
    p = transform.get("pivot", (0.0, 0.0, 0.0))

    x = float(point[0]) - float(t[0]) - float(p[0])
    y = float(point[1]) - float(t[1]) - float(p[1])
    z = float(point[2]) - float(t[2]) - float(p[2])

    xr, yr, zr = _rotate_vec(_quat_conj(q), (x, y, z))

    sx = max(float(s[0]), 1e-30)
    sy = max(float(s[1]), 1e-30)
    sz = max(float(s[2]), 1e-30)

    return (
        float(p[0]) + xr / sx,
        float(p[1]) + yr / sy,
        float(p[2]) + zr / sz,
    )
```

To jest właśnie mechanika, której dziś brakuje pomiędzy punktem symulacji a funkcją presetu.

## 15. Ostateczny werdykt

### Problem z vortexem
Tak — w kodzie widać bardzo mocne przesłanki, że macie realny problem współrzędnych i samplingu tekstury. Najmocniejszy root-cause to brak zastosowania `texture_transform` w evaluatorze presetów oraz brak adaptera scena <-> symulacja dla gizmo tekstury.

### Problem z przesuwaniem środka tekstury
Tak — obecnie system nie implementuje jeszcze poprawnej fizycznie semantyki „przesuwam teksturę, więc przeliczam magnetyzację w każdym węźle”. To trzeba dopiero domknąć przez inverse-transform w samplerze tekstury.

### Problem z nakładaniem się kamery i gizmo
Tak — obecna architektura viewportu nie ma jeszcze jednego, wspólnego locka interakcji dla wszystkich manipulatorów. To trzeba uporządkować na poziomie preview state machine.
