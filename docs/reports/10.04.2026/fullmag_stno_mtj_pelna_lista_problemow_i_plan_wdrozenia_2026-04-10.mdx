# Fullmag STNO/MTJ — pełna lista problemów i bardzo szczegółowy plan wdrożenia

## 1. Cel dokumentu

Ten dokument zbiera **wszystkie wykryte problemy** związane z próbą zbudowania **produkcyjnego** workflow STNO/MTJ dla przypadku:

- vortex w nanokropce o średnicy 400 nm,
- stos materiałowy uproszczony do modelu swobodnej warstwy typu CoFeB(1 nm) / NiFe(8 nm),
- pobudzenie przez:
  - spin-transfer torque (STT),
  - pole Oersteda od prądu stałego lub zmiennego w czasie,
  - temperaturę / fluktuacje termiczne,
- analiza:
  - sygnału czasowego,
  - PSD/FFT,
  - częstotliwości piku,
  - linewidth,
  - trajektorii rdzenia vortexu,
  - promienia orbity,
  - fazy.

Dokument ma dwa zadania jednocześnie:

1. **Audyt stanu obecnego** repozytorium.
2. **Plan wdrożeniowy plik-po-pliku**, bardzo szczegółowy, tak aby można go było zamienić na serię PR-ów.

## 2. Werdykt wykonawczy

### 2.1. Werdykt krótki

**Nie, repo nie jest jeszcze w stanie „produkcyjny model STNO end-to-end”.**

Najbliżej tego celu jest dziś ścieżka:

- **Python DSL / IR** — w dużej części przygotowana,
- **native FDM / CUDA** — najbliżej pełnej fizyki STNO,
- **frontend** — nadal niepełny,
- **CPU fallback** — nie ma pełnej parzystości z backendem produkcyjnym,
- **FEM path** — ma Oersteda i temperaturę, ale brak potwierdzonego pełnego STT end-to-end.

### 2.2. Co już działa lub jest bardzo blisko

Na podstawie publicznie sprawdzalnych warstw repo da się potwierdzić, że:

- `Problem` ma już pola `spin_torque` i `temperature`.
- Python DSL ma klasy:
  - `SlonczewskiSTT`,
  - `ZhangLiSTT`,
  - `OerstedCylinder`,
  - `Sinusoidal`,
  - `Pulse`,
  - `ThermalNoise`,
  - `UniaxialAnisotropy`,
  - `CubicAnisotropy`.
- IR wspiera:
  - top-level STT,
  - top-level `temperature`,
  - `EnergyTermIR::OerstedCylinder`,
  - `TimeDependenceIR`,
  - `RfDriveIR`,
  - capability flags: `supports_stt`, `supports_thermal`, `supports_oersted`.
- `native_fdm` runner ma opis planu obejmujący STT, temperaturę i Oersteda, w tym parametry czasowe pobudzenia.
- `native_fem` runner przenosi Oersteda i temperaturę.
- przykład `examples/stno_vortex_mtj_workflow.py` już istnieje i serializuje STT, temperaturę oraz Oersteda.

### 2.3. Dlaczego mimo tego nie daję statusu „produkcyjny”

Bo nadal występują blokery strukturalne:

1. **Rozjazd publicznego API i builder/UI**  
   `Problem` ma już STT i temperaturę, ale builder manifest nie publikuje ich jako pełnoprawnych pól round-trip dla UI.

2. **Brak pełnej parzystości backendów**  
   Najbliżej pełnej fizyki STNO jest `native_fdm`, ale:
   - CPU reference ma STT + thermal, lecz brak w nim potwierdzonego Oersteda,
   - native FEM ma Oersteda + thermal, ale brak potwierdzonego STT.

3. **Frontend nie ma pełnego UI STNO**  
   Nie ma potwierdzonego kompletnego authoringu i analyze workbench dla:
   - STT,
   - temperatury,
   - Oersted cylinder,
   - waveformów,
   - PSD / linewidth / orbit / core tracking.

4. **Postprocessing jest na razie badawczy, nie produkcyjny**  
   `linewidth_lorentzian` jest w praktyce estymatorem half-maximum, a nie rzetelnym dopasowaniem Lorenza.  
   `track_vortex_core` jest heurystyką, nie stabilnym trackerem produkcyjnym.

5. **Brak twardej walidacji, benchmarków i testów regresyjnych STNO**  
   Widoczny zestaw testów Python nie pokrywa STNO, Oersteda, temperatury, linewidth ani trackingu vortexu.

## 3. Matryca gotowości — stan realny

## 3.1. Warstwa Python DSL

**Stan:** częściowo gotowa, ale niespójna.

Plusy:
- są klasy STT,
- jest OerstedCylinder z time dependence,
- jest temperatura i ThermalNoise,
- przykład STNO istnieje.

Minusy:
- alias `EnergyTerm` jest nieaktualny względem realnych klas fizycznych,
- publiczny export surface jest niespójny,
- builder manifest nie wystawia wszystkiego, co DSL już potrafi.

## 3.2. IR / Planner

**Stan:** dobra baza kontraktowa, ale z oznakami dryfu architektonicznego.

Plusy:
- IR ma pola STT / thermal / Oersted / waveformy,
- capability matrix przewiduje STT, Oersteda, thermal.

Minusy:
- komentarze i część plannerowego opisu nadal sugerują „phase 1” i węższy zakres wykonawczy niż to, co widać już w runnerach,
- Python DSL nie mapuje jeszcze całego zakresu dostępnego w IR.

## 3.3. Native FDM

**Stan:** najbliżej produkcyjnego STNO.

Plusy:
- runner przenosi STT, thermal i Oersteda,
- capability matrix deklaruje szeroki zestaw termów.

Minusy:
- brak potwierdzonego pełnego przepływu thermal seed,
- trzeba zbudować walidację fizyczną i benchmarki STNO,
- trzeba domknąć artefakty i analitykę.

## 3.4. CPU reference FDM

**Stan:** niepełna ścieżka referencyjna.

Plusy:
- ma STT,
- ma temperaturę / thermal_dt.

Minusy:
- brak potwierdzonego Oersteda,
- capability commentary mówi, że `H_ant` jest zawsze zerowe,
- brak pełnej parzystości z native FDM.

## 3.5. Native FEM

**Stan:** niepełny dla STNO.

Plusy:
- ma Oersteda,
- ma temperaturę.

Minusy:
- brak potwierdzonego przepływu STT w wrapperze,
- brak podstaw, by traktować FEM jako kompletny backend STNO dla MTJ vortex oscillator.

## 3.6. Frontend

**Stan:** niepełny.

Plusy:
- ma ogólny builder etapów i analyze dla eigen/dispersion.

Minusy:
- brak pełnego UI authoring dla STNO,
- brak analyze workbench pod PSD/FFT/linewidth/core tracking,
- brak sensownego round-trip builder manifest ↔ UI dla STT/thermal/Oersted.

## 3.7. Dokumentacja i testy

**Stan:** nieprodukcyjny.

Plusy:
- jest przykład workflow STNO.

Minusy:
- brak noty fizycznej STNO/vortex/Oersted,
- brak szerokiego pakietu testów,
- brak benchmarków jakościowych i ilościowych.

# Problem F01 — alias `EnergyTerm` w Python DSL jest nieaktualny względem realnych klas fizycznych

**Krytyczność:** wysoka  
**Obszar:** Python DSL / typowanie / builder / ergonomia API

## Stan obecny

W `problem.py` alias `EnergyTerm` nie odzwierciedla pełnego zestawu klas, które faktycznie istnieją w `energy.py` i są używane w przykładzie STNO. To oznacza, że DSL i typowanie są rozjechane z realną fizyką modelu.

## Skutek

- typowanie jest mylące,
- edytory i narzędzia statyczne pokazują niepełny kontrakt,
- builder manifest i walidacja mogą nie nadążać za realnym zakresem API,
- przykład STNO wygląda jakby używał fizyki „spoza oficjalnego typu”.

## Potwierdzone symptomy

Widoczne są klasy:
- `OerstedCylinder`,
- `UniaxialAnisotropy`,
- `CubicAnisotropy`,
- `ThermalNoise`,

a alias `EnergyTerm` nie jest z nimi w pełni zgodny.

## Pliki do zmiany

1. `packages/fullmag-py/src/fullmag/model/problem.py`
2. ewentualnie `packages/fullmag-py/src/fullmag/model/energy.py`
3. testy:
   - `packages/fullmag-py/tests/test_api.py`
   - nowy: `packages/fullmag-py/tests/test_energy_terms_contract.py`

## Wymagane zmiany

### A. Ujednolicić alias `EnergyTerm`

Docelowo alias powinien obejmować wszystkie publiczne, wspierane przez DSL termy energii. Minimalna wersja:

```python
EnergyTerm = (
    Exchange
    | Demag
    | Zeeman
    | InterfacialDMI
    | BulkDMI
    | UniaxialAnisotropy
    | CubicAnisotropy
    | OerstedCylinder
    | Magnetoelastic
    | ThermalNoise
)
```

### B. Zdecydować, czy `ThermalNoise` ma pozostać energy termem

To jest ważna decyzja architektoniczna.

Są dwie opcje:

#### Opcja 1 — `ThermalNoise` pozostaje energy termem
Wtedy:
- trzeba konsekwentnie włączyć go do `EnergyTerm`,
- jawnie opisać relację `Problem.temperature` ↔ `ThermalNoise`,
- zdefiniować precedence rules.

#### Opcja 2 — `ThermalNoise` przestaje być energy termem
Wtedy:
- trzeba usunąć go z `energy.py` jako term lub oznaczyć deprecated,
- wprowadzić osobny model `ThermalConfig`,
- przenieść go do top-level `Problem`.

**Rekomendacja:** dla produkcyjnego STNO wybrać **Opcję 2**.  
Top-level temperatura + seed + opcjonalna polityka generatora jest architektonicznie czystsza niż „term energii”, który w praktyce jest fluktuacją stochastyczną, a nie deterministycznym wkładem energetycznym.

### C. Dodać walidację spójności

Jeżeli przez okres przejściowy zostaje:
- `Problem.temperature`
- oraz `ThermalNoise(...)`

to trzeba w `Problem.__post_init__` lub w validatorze zablokować niejednoznaczne przypadki:

- `temperature is not None` i jednocześnie `ThermalNoise` w `energy_terms` z inną temperaturą,
- dwa różne seed source,
- `temperature < 0`.

## Kryteria akceptacji

- type checker widzi pełny kontrakt,
- wszystkie przykłady STNO przechodzą bez hacków typowych,
- dokumentacja i auto-complete odzwierciedlają realne API.

## Testy

1. Konstrukcja `Problem(..., energy_terms=[OerstedCylinder(...)])` nie powoduje niezgodności typowej.
2. Konstrukcja z `UniaxialAnisotropy` i `CubicAnisotropy` przechodzi.
3. Konfiguracje konfliktowe thermal są odrzucane z czytelnym błędem.

## Szacowany zakres

- kod: 40–120 LOC,
- testy: 80–180 LOC.

# Problem F02 — builder manifest nie publikuje `spin_torque` i `temperature` jako pełnych pól problemu

**Krytyczność:** krytyczna  
**Obszar:** Python DSL ↔ frontend round-trip

## Stan obecny

`Problem` ma już pola `spin_torque` i `temperature`, ale manifest budowany dla buildera nie wystawia ich jako pełnoprawnej części modelu, którą UI może bezstratnie edytować i zapisać.

## Skutek

- UI nie ma gwarantowanego round-tripu dla STT i temperatury,
- backend potrafi więcej niż builder/frontend,
- pojawia się „ukryta fizyka” dostępna tylko z kodu.

## Pliki do zmiany

1. `packages/fullmag-py/src/fullmag/model/problem.py`
2. frontend konsumujący manifest:
   - `apps/web/lib/study-builder/types.ts`
   - `apps/web/lib/study-builder/materialize.ts`
   - komponenty formularzy problemu / physics editor (nowe lub istniejące)

## Wymagane zmiany

### A. Rozszerzyć `build_problem_builder_manifest`

Do serializacji manifestu trzeba dopisać:

- `spin_torque`,
- `temperature`,
- docelowo `thermal_seed`,
- opcjonalnie `analysis` / `postprocessing` jeśli chcemy round-trip dla STNO workbench.

Przykładowy kierunek:

```python
"problem": {
    ...
    "spin_torque": _to_builder_value(problem.spin_torque),
    "temperature": problem.temperature,
    "thermal_seed": problem.thermal_seed,
    ...
}
```

### B. Dodać schema metadata dla UI

Sam surowy payload nie wystarczy. UI potrzebuje semantyki:

- discriminator:
  - `slonczewski`
  - `zhang_li`
- jednostki:
  - `A/m^2`,
  - `K`,
  - bezwymiarowe współczynniki STT,
- zakresy walidacyjne.

Przykład struktury:

```python
"schema": {
  "spin_torque": {
    "kind": "union",
    "discriminator": "kind",
    "variants": [
      {"kind": "slonczewski", "fields": {...}},
      {"kind": "zhang_li", "fields": {...}}
    ]
  },
  "temperature": {
    "kind": "float",
    "units": "K",
    "min": 0.0
  }
}
```

### C. Włączyć round-trip tests

Potrzebny test typu:

1. zbuduj `Problem` ze STT i temperaturą,
2. wygeneruj manifest,
3. przemapuj manifest z powrotem do `Problem`,
4. porównaj semantycznie wynik.

## Kryteria akceptacji

- UI może stworzyć, otworzyć i ponownie zapisać problem STNO bez utraty STT/temperature,
- eksport/import projektu zachowuje te pola,
- builder manifest jest stabilny w testach snapshot.

## Testy

- snapshot manifestu,
- round-trip manifest ↔ problem,
- walidacja wszystkich wariantów STT.

## Szacowany zakres

- backend manifest: 80–180 LOC,
- frontend schema/forms: 250–700 LOC,
- testy: 150–350 LOC.

# Problem F04 — Python DSL nie wystawia `PiecewiseLinear`, mimo że IR już to wspiera

**Krytyczność:** wysoka  
**Obszar:** waveformy / pobudzenie czasowe

## Stan obecny

IR przewiduje `PiecewiseLinear`, ale Python DSL dla `TimeDependence` ma tylko:
- `Constant`,
- `Sinusoidal`,
- `Pulse`.

## Skutek

- Python public API jest uboższe niż backend kontraktowy,
- nie można wygodnie modelować ramp, modulacji i ręcznie zadanych obwiedni,
- użytkownik STNO nie ma sensownego narzędzia do:
  - sweepu amplitudy,
  - kick-startu,
  - wygaszania,
  - niestandardowych impulsów.

## Pliki do zmiany

1. `packages/fullmag-py/src/fullmag/model/energy.py`
2. `packages/fullmag-py/src/fullmag/model/antenna.py` lub równoważna warstwa waveformów current modules
3. `packages/fullmag-py/src/fullmag/model/__init__.py`
4. `packages/fullmag-py/src/fullmag/__init__.py`
5. testy:
   - nowy `packages/fullmag-py/tests/test_time_dependence.py`

## Wymagane zmiany

### A. Dodać klasę `PiecewiseLinear`

Przykładowy szkic:

```python
@dataclass(frozen=True)
class PiecewiseLinear:
    points: tuple[tuple[float, float], ...]

    def __post_init__(self) -> None:
        if len(self.points) < 2:
            raise ValueError("PiecewiseLinear requires at least 2 points")
        ts = [p[0] for p in self.points]
        if any(t2 <= t1 for t1, t2 in zip(ts, ts[1:])):
            raise ValueError("PiecewiseLinear times must be strictly increasing")

    def to_ir(self) -> dict[str, object]:
        return {
            "kind": "piecewise_linear",
            "points": [[float(t), float(v)] for t, v in self.points],
        }
```

### B. Rozszerzyć alias `TimeDependence`

```python
TimeDependence = Constant | Sinusoidal | Pulse | PiecewiseLinear
```

### C. Wyrównać to w OerstedCylinder i RF drive

To samo powinno być możliwe dla:
- `OerstedCylinder.time_dependence`,
- `RfDrive.waveform`,
- docelowo innych time envelope.

## Kryteria akceptacji

- Python DSL może wygenerować każdy waveform obsługiwany przez IR,
- frontend może taki waveform wyedytować.

## Testy

1. `PiecewiseLinear` z poprawnymi punktami serializuje się do IR.
2. Błędna monotoniczność czasu zgłasza wyjątek.
3. `OerstedCylinder(time_dependence=PiecewiseLinear(...))` przechodzi round-trip.

## Szacowany zakres

- kod: 80–160 LOC,
- testy: 100–180 LOC.

# Problem F09 — `linewidth_lorentzian` nie jest produkcyjnym fitterem linewidth

**Krytyczność:** wysoka  
**Obszar:** analiza STNO / metrologia spektralna

## Stan obecny

Funkcja nazwana `linewidth_lorentzian` jest w praktyce prostym estymatorem half-maximum width, a nie rzetelnym dopasowaniem funkcji Lorenza.

## Dlaczego to jest poważny problem

W STNO linewidth bywa jednym z głównych wyników fizycznych.  
Jeżeli narzędzie:
- nie dopasowuje tła,
- nie liczy niepewności,
- nie oddziela sąsiednich pików,
- nie uwzględnia jakości PSD,

to wynik może być mylący.

## Pliki do zmiany

1. `packages/fullmag-py/src/fullmag/analysis/spectrum.py`
2. nowy:
   - `packages/fullmag-py/src/fullmag/analysis/fitting.py`
3. testy:
   - `packages/fullmag-py/tests/test_stno_analysis.py`

## Wymagane zmiany

### A. Rozdzielić estymator i prawdziwy fitter

Dziś jedna funkcja robi coś, co nie odpowiada nazwie.  
Docelowo:

- `linewidth_halfmax(...)` — szybki estymator,
- `fit_lorentzian_linewidth(...)` — właściwy fitter.

### B. Zdefiniować model fitu

Minimalny model:

\[
S(f) = B_0 + B_1 f + \frac{A}{1 + \left(\frac{f-f_0}{\Gamma}\right)^2}
\]

gdzie:
- \(f_0\) — peak frequency,
- \(\Gamma\) — half-width,
- linewidth FWHM = \(2\Gamma\).

### C. Dodać preprocessing

- detrending sygnału czasowego,
- windowing,
- Welch PSD,
- wybór okna fitu,
- opcjonalne odjęcie tła.

### D. Zwracać pełny wynik

Nie samą liczbę, tylko np.:

```python
@dataclass(frozen=True)
class LinewidthFitResult:
    f0_hz: float
    fwhm_hz: float
    q_factor: float
    amplitude: float
    background_offset: float
    background_slope: float
    fit_r2: float | None
    covariance: np.ndarray | None
    fit_window_hz: tuple[float, float]
    method: str
```

### E. Zachować starą funkcję jako deprecated alias

To zmniejszy ryzyko łamania API.

## Kryteria akceptacji

- dla syntetycznego piku Lorenza błąd linewidth < 5%,
- testy rozróżniają half-max estimate od fitu,
- UI nie opisuje estymatora jako „Lorentzian fit”, jeśli nim nie jest.

## Szacowany zakres

- kod: 200–450 LOC,
- testy: 150–300 LOC.

**Obszar:** pobudzenia / RF / rozszerzalność

Dla części scenariuszy STNO to nie blokuje v1, ale ogranicza dalszy rozwój:
- mikrostrip,
- nanocontact niestandardowy,
- złożone profile przestrzenne prądu.

## Pliki do zmiany

1. `packages/fullmag-py/src/fullmag/model/antenna.py`
2. IR i runner — zależnie od decyzji zakresowej
3. frontend waveform + geometry editors

Na v1 nie traktowałbym tego jako blocker.  
Ale trzeba jawnie opisać w roadmapie:
- jakie profile przestrzenne prądu są wspierane,
- które są planned, a które out-of-scope.

## Kryteria akceptacji

- brak niejawnych obietnic w docs/UI,
- capability matrix odzwierciedla realny zakres.

# Problem F18 — brak pierwszoklasowych obserwabli STNO i artefaktów analitycznych

**Krytyczność:** krytyczna  
**Obszar:** artifacts / postprocessing / UX

## Stan obecny

Postprocessing utilities istnieją w Pythonie, ale nie widać kompletnego, produktowego systemu artefaktów STNO.

## Co powinno powstawać w runie STNO

Minimalny pakiet artefaktów:

1. `m_avg.csv` lub `m_avg.parquet`
2. `signal_probes.parquet`
3. `core_track.parquet`
4. `psd.npz` lub `psd.parquet`
5. `linewidth.json`
6. `stno_summary.json`

## Brakujące cechy

- standaryzowany format artefaktów,
- metadata z jednostkami,
- versioning schematu artefaktów,
- możliwość porównania sweepów.

## Pliki do zmiany

Prawdopodobne miejsca:
1. `crates/fullmag-runner/src/*artifacts*`
2. `crates/fullmag-runner/src/*quantities*`
3. `packages/fullmag-py/src/fullmag/analysis/*`
4. frontend analyze components

## Wymagane zmiany

### A. Zdefiniować artefakty pierwszej klasy

Na poziomie specyfikacji:
- nazwa,
- schema,
- units,
- provenance.

### B. Dodać kontrolę w `Problem.outputs` lub osobnym `analysis_config`

Użytkownik powinien móc zaznaczyć:
- save average magnetization,
- save probe traces,
- save core track,
- compute PSD,
- fit linewidth.

### C. Zaimplementować generację artefaktów

Najpierw minimalnie w Python postprocess po runie,
docelowo część w runnerze.

## Kryteria akceptacji

- każdy run STNO może automatycznie produkować zestandaryzowane artefakty,
- frontend umie je odczytać bez ad-hoc parserów.

## Szacowany zakres

- spec + runner + frontend: 800–2500 LOC.

# Problem F19 — semantyka temperatury jest niejednoznaczna (`Problem.temperature` vs `ThermalNoise`)

**Krytyczność:** wysoka  
**Obszar:** model fizyczny / spójność API

## Stan obecny

Repo ma:
- top-level `Problem.temperature`,
- oraz publiczny byt `ThermalNoise`.

To jest konstrukcyjnie podejrzane, bo użytkownik ma dwa sposoby opisu tej samej klasy zjawisk.

## Skutek

- ryzyko konfliktów,
- niejasność, który parametr jest źródłem prawdy,
- trudne UI i trudne testy.

## Rekomendacja architektoniczna

### Preferowany model produkcyjny

Zostawić top-level:

```python
Problem(
    ...
    temperature=300.0,
    thermal_seed=12345,
)
```

a `ThermalNoise`:
- oznaczyć jako deprecated,
- lub usunąć z publicznego DSL po okresie przejściowym.

## Pliki do zmiany

1. `packages/fullmag-py/src/fullmag/model/problem.py`
2. `packages/fullmag-py/src/fullmag/model/energy.py`
3. `packages/fullmag-py/src/fullmag/__init__.py`
4. docs / examples / tests

## Kryteria akceptacji

- istnieje tylko jeden kanoniczny sposób ustawiania temperatury i seedu,
- użytkownik nie może wejść w stan konfliktowy.

# Problem F20 — brak pełnej, jawnej polityki reproducibility dla thermal seed

**Krytyczność:** krytyczna dla badań ilościowych  
**Obszar:** reprodukowalność / stochastic simulations

## Stan obecny

IR przewiduje `ThermalSeedConfig`, engine ma pojęcie `thermal_seed`, ale w sprawdzonych publicznych ścieżkach nie widać kompletnego, jawnego, produkcyjnego surfacingu seed policy end-to-end.

## Skutek

- trudniej odtworzyć wyniki,
- porównania linewidth przy różnych commitach mogą być pozornie niestabilne,
- testy regresyjne stochastyczne są trudne.

## Pliki do zmiany

1. `packages/fullmag-py/src/fullmag/model/problem.py`
2. `crates/fullmag-ir/src/lib.rs`
3. `crates/fullmag-plan/src/fdm.rs`
4. `crates/fullmag-plan/src/fem.rs` jeśli dotyczy
5. `crates/fullmag-runner/src/cpu_reference.rs`
6. `crates/fullmag-runner/src/native_fdm.rs`
7. `crates/fullmag-runner/src/native_fem.rs`
8. `crates/fullmag-engine/src/fdm_problem.rs`

## Wymagane zmiany

### A. Publiczny API

Dodać do `Problem`:

```python
thermal_seed: int | None = None
```

lub bardziej elastycznie:

```python
thermal: ThermalConfig | None = None
```

### B. Jasna polityka seedowania

- `None` = nondeterministic/random seed,
- `int` = deterministic,
- opcjonalnie `seed_policy`:
  - `"fixed"`
  - `"from_time"`
  - `"from_run_id"`

### C. Propagacja przez wszystkie warstwy

`Problem` → IR → plan → runner → engine/native backend

### D. Ująć seed w metadanych artefaktów

Każdy run STNO powinien zapisać seed w `summary.json`.

## Kryteria akceptacji

- dwa runy z tym samym seedem są bitowo lub statystycznie zgodne w akceptowalnym zakresie,
- run bez seedu jest jawnie oznaczony jako nondeterministic.

## Szacowany zakres

- 150–500 LOC backend,
- 50–150 LOC docs/tests.

# Problem F21 — capability matrix i preview quantities są niespójne między backendami

**Krytyczność:** średnia do wysokiej  
**Obszar:** UX / debug / analyze

- CPU nie wystawia części pól,

## Pliki do zmiany

2. frontend capability gating
3. docs runtime matrix

## Wymagane zmiany

### A. Jawny capability contract dla observables

Nie tylko physics terms, ale także:
- preview fields,
- snapshot fields,
- derived artifacts.

### B. Frontend gating

UI ma ukrywać / wyszarzać funkcje niewspierane przez backend.

### C. Dążyć do minimalnej wspólnej bazy

Dla STNO minimalna wspólna baza powinna obejmować:
- average magnetization trace,
- external field,
- effective field summary,
- core-track if available.

## Kryteria akceptacji

- użytkownik nie może uruchomić panelu, którego backend nie wspiera,
- capability matrix obejmuje także analitykę i artefakty.

# Problem F22 — model MTJ jest nadal uproszczony i semantycznie niepełny jako „stack device”

**Krytyczność:** średnia strategicznie  
**Obszar:** model urządzenia / produktowość

## Stan obecny

Obecny DSL opisuje fizykę wolnej warstwy, STT i Oersteda, ale nie widać pełnej semantyki urządzenia MTJ jako stosu:
- free layer,
- reference layer / polarizer,
- spacer / barrier,
- geometria kontaktu,
- stack-resolved current path.

## Dlaczego to istotne

Na v1 to nie musi blokować, jeśli uczciwie mówimy:
> „To jest mikromagnetyczny model swobodnej warstwy z efektywnym STT i Oerstedem, a nie pełny TCAD stack model MTJ.”

Ale dla dokumentacji i marketingu to rozróżnienie jest krytyczne.

## Rekomendacja

Na v1:
- zostawić model efektywny,
- jasno to opisać w docs.

Na v2:
- rozważyć `MtjDevice` / `NanocontactDevice` jako wysokopoziomowe byty.

# Problem F24 — brak benchmarków ilościowych STNO i testów regresji fizycznej

**Krytyczność:** krytyczna  
**Obszar:** physics validation

## Stan obecny

Nie widać benchmark suite odpowiadającego na pytania:
- jaki jest zakres częstotliwości gyro-mode dla referencyjnego dysku,
- jak zależy od prądu,
- jak zależy od temperatury,
- jak zależy od amplitudy Oersteda,
- jaki jest expected linewidth trend.

## Co trzeba zrobić

### Minimalny benchmark set

1. **Vortex relax benchmark**
   - sprawdza, że stan relaksuje do vortexu.

2. **Gyrotropic mode frequency benchmark**
   - ma expected range dla małego pobudzenia.

3. **Oersted AC resonance benchmark**
   - pik PSD w pobliżu trybu gyrotropowego.

4. **STT threshold trend benchmark**
   - wzrost amplitudy oscylacji po przekroczeniu progu.

5. **Thermal broadening benchmark**
   - linewidth rośnie jakościowo z temperaturą.

## Pliki do dodania

1. `benchmarks/stno_vortex/*`
2. `tests/physics/stno/*`
3. docs z expected envelope

## Kryteria akceptacji

- każdy benchmark ma tolerancję i seed policy,
- benchmarki są uruchamiane co najmniej na nightly / release CI.

## 7. Zalecana architektura docelowa v1

Żeby nie ugrzęznąć, rekomenduję **bardzo konkretny scope v1**:

### 7.1. Co ogłaszamy jako wspierane produkcyjnie

**STNO v1 = native FDM backend**

Zakres:
- vortex free-layer disk,
- STT:
  - Slonczewski,
  - opcjonalnie Zhang–Li, jeśli ma testy,
- OerstedCylinder:
  - constant,
  - sinusoidal,
  - pulse,
  - piecewise-linear,
- temperatura:
  - top-level `temperature`,
  - `thermal_seed`,
- analiza:
  - time trace,
  - PSD,
  - peak frequency,
  - Lorentzian linewidth fit,
  - core trajectory,
  - orbit radius,
  - sweep compare.

### 7.2. Co oznaczamy jako eksperymentalne

- native FEM dla STNO,
- `ThermalNoise` jako osobny byt, jeśli zostaje,
- zaawansowane waveformy beyond PWL,
- zaawansowane current modules.

### 7.3. Co wycofujemy lub zamrażamy

- nazwy sugerujące większą dokładność niż realna implementacja, np. obecne `linewidth_lorentzian`, jeśli nie robi fitu.

### 9.2. `energy.py` — `PiecewiseLinear`

```python
@dataclass(frozen=True)
class PiecewiseLinear:
    points: tuple[tuple[float, float], ...]

    def __post_init__(self) -> None:
        if len(self.points) < 2:
            raise ValueError("expected at least two points")
        ts = [t for t, _ in self.points]
        if any(t2 <= t1 for t1, t2 in zip(ts, ts[1:])):
            raise ValueError("times must be strictly increasing")

    def to_ir(self) -> dict[str, object]:
        return {
            "kind": "piecewise_linear",
            "points": [[float(t), float(v)] for t, v in self.points],
        }
```

## 10. Minimalny zestaw równań, które powinny trafić do docs i kodu opisowego

### 10.1. LLG z efektywnym polem i torque

\[
\frac{d\mathbf{m}}{dt}
=
-\gamma \mu_0 \mathbf{m} \times \mathbf{H}_{\mathrm{eff}}
+
\alpha \mathbf{m} \times \frac{d\mathbf{m}}{dt}
+
\mathbf{T}_{\mathrm{STT}}
+
\mathbf{T}_{\mathrm{thermal}}
\]

### 10.2. Slonczewski STT — forma efektywna

Praktycznie w dokumentacji trzeba jasno opisać:
- znaczenie `current_density`,
- `spin_polarization`,
- `degree`,
- `lambda_asymmetry`,
- `epsilon_prime`,
- konwencję znaku prądu i orientacji polaryzacji.

### 10.3. Zhang–Li STT

\[
\mathbf{T}_{\mathrm{ZL}}
\sim
(\mathbf{u}\cdot\nabla)\mathbf{m}
-
\beta \mathbf{m}\times (\mathbf{u}\cdot\nabla)\mathbf{m}
\]

### 10.4. Pole Oersteda dla cylindra

Dla nieskończonego przewodnika cylindrycznego:

\[
H_\phi(r) \propto
\begin{cases}
r, & r < R \\
1/r, & r \ge R
\end{cases}
\]

z odpowiednim kierunkiem azymutalnym zależnym od osi przewodnika i znaku prądu.

### 10.5. Termiczny poziom szumu

W dokumentacji i kodzie trzeba spójnie opisać:
- zależność od \(T\),
- kroku czasowego,
- objętości komórki,
- tłumienia.

To dziś największe ryzyko percepcyjne.

### R4 — UI bez capability gating
Może wystawiać funkcje, których backend nie umie wykonać.

### R5 — brak benchmarków ilościowych
Bez nich linewidth i częstotliwości będą wyglądały „ładnie”, ale nie będzie wiadomo, czy są wiarygodne.

## 15. Końcowa rekomendacja strategiczna

Moja mocna rekomendacja jest taka:

### A. Formalnie ogłoś STNO v1 jako **native FDM-first**
Nie próbuj teraz sprzedawać tego jako pełne „FDM/FEM production STNO”.

### B. Uczyń `Problem + manifest + frontend` pierwszoklasowym interfejsem
Dziś to jest największy rozjazd między potencjałem backendu a realnym produktem.

### C. Potraktuj analizę STNO jako osobny produkt
PSD, linewidth, core track, sweep compare to nie są „drobiazgi”.
To jest połowa wartości funkcji STNO.

### D. Wprowadź benchmarki zanim padnie słowo „produkcyjny”
Bez benchmarków będzie to nadal funkcja badawcza / eksperymentalna.
