# Fullmag Study / Stages Refactor Plan

Data: 2026-04-07
Status: draft architecture + UX blueprint
Cel: przebudowac `Study` / `Stages` w drzewie modelu tak, aby dzialaly jak prawdziwy COMSOL-like study builder, a nie jak jeden wspolny panel zlepiajacy kilka roznych odpowiedzialnosci.

---

## 1. Executive Summary

Obecny `Study` jest zle rozwarstwiony:

- drzewo udaje semantyczne nody, ale wiele z nich nie ma wlasnego inspectora,
- ribbon nie jest naprawde sterowany selekcja w drzewie,
- `StudyPanel` miesza authoring, backend config, runtime summary i stage sequence,
- `study_pipeline` i `stages` sa rownoleglymi modelami tej samej rzeczy,
- Python i GUI nie maja jednego kanonicznego kontraktu stage.

Docelowo musimy przyjac prosty model:

1. `Stage` to pojedynczy user-facing etap badania.
2. `Execution step` to backendowy krok wykonywania po materializacji stage do prymitywow.
3. Drzewo po lewej pokazuje prawdziwa strukture `Study`.
4. Ribbon u gory pokazuje komendy dla aktualnie wybranego noda.
5. Inspector po prawej jest node-specific.
6. Dolny panel pokazuje progress, log, queue, artifacts i telemetry.
7. GUI i Python konfiguruja dokladnie ten sam model danych.

W praktyce oznacza to:

- usuwamy monolityczny `StudyPanel` jako glowny model pracy,
- usuwamy duplikacje `Stage Builder` vs `Stage Sequence`,
- przenosimy `Backend Configuration` do `Study Defaults` albo `Stage Solver Override`,
- budujemy jeden kanoniczny `StudyDocumentV2`,
- dopiero z niego kompilujemy legacy `flat stages` dla backendu.

## 2. Diagnoza Obecnego Stanu

Najwazniejsze problemy widoczne dzis w repo:

### 2.1. Kazdy study node pokazuje prawie to samo

W `apps/web/components/panels/SettingsPanel.tsx` wszystkie nody `study-*` sa routowane do tego samego `StudyPanel`.

Efekt:

- klik w `Study`,
- klik w `Stage Builder`,
- klik w `Stage Sequence`,
- klik w `study-stage-0`,
- klik w `Integrator`,

konczy sie bardzo podobnym lub identycznym panelem. To jest anty-COMSOL. W COMSOL selection = inny ribbon + inny settings view.

### 2.2. `StudyPanel` miesza 4 rozne role

Obecny `apps/web/components/panels/settings/StudyPanel.tsx` scala w jednym miejscu:

- stage authoring,
- backend configuration,
- runtime settings summary,
- stage sequence cards.

To nie sa jedna i ta sama rzecz.

### 2.3. Drzewo ma fake nody zamiast prawdziwych bytow

W `apps/web/components/panels/ModelTree.tsx` mamy obecnie:

- `Study Setup`,
- `Stage Builder`,
- `Stage Sequence`,
- `Integrator`,
- `Time Stepping`,
- `Convergence`.

To nie jest dobra struktura authoringu. To jest mieszanka:

- struktury modelu,
- skrutow do paneli,
- oraz agregatow runtime.

### 2.4. Dwa source-of-truth dla tej samej sekwencji

Obecnie wspolistnieja:

- `study.stages` - backendowe flat primitive stages,
- `study.study_pipeline` - authoring pipeline.

To ma sens jako warstwa kompatybilnosci, ale nie ma sensu jako docelowy model UI.

### 2.5. Kontrakt stage jest plaski, stringly-typed i niepelny

`ScriptBuilderStageState` w `apps/web/lib/session/types.ts` i `crates/fullmag-authoring/src/builder.rs` ma pola typu:

- `kind: string`,
- `integrator: string`,
- `fixed_timestep: string`,
- `until_seconds: string`,
- `relax_algorithm: string`,
- itd.

Brakuje:

- pelnego adaptive timestep config,
- spoleczonego modelu solver overrides,
- jawnych input/output policies,
- normalnej typizacji dla stage kinds,
- spoleczonego kontraktu GUI <-> Python <-> Rust.

### 2.6. Python nadal mysli globalnym `solver()`, nie stage-first

W `packages/fullmag-py/src/fullmag/world.py`:

- `solver()` ustawia globalny stan,
- `run()`, `relax()`, `eigenmodes()` buduja Problem na podstawie tego globalnego stanu,
- stage override jest czesciowy.

To nie odpowiada temu, jak uzytkownik mysli o sekwencji etapow w drzewie.

### 2.7. Materializacja dopisuje domysly, zamiast pracowac na jawnych danych

W `apps/web/lib/study-builder/materialize.ts` pipeline materialization tworzy stage z twardymi defaultami.

To oznacza, ze:

- nie wszystko jest jawnie skonfigurowane,
- preview sekwencji niekoniecznie pokazuje pelny prawdziwy kontrakt,
- round-trip Python <-> GUI jest slaby.

## 3. Decyzje Produktowe

To powinny byc decyzje zamrozone przed implementacja.

### 3.1. `Study` authoring zyje w `Model Builder`

To jest najwazniejsza decyzja.

`Model Builder`:

- geometra,
- materialy,
- physics,
- mesh,
- study defaults,
- study stages.

`Study` workspace:

- uruchamianie,
- monitorowanie,
- progress,
- queue,
- logs,
- quick run actions.

Czyli:

- authoring w drzewie modelu,
- execution/monitoring w workspace `Study`.

### 3.2. `Stage` jest user-facing bytem, nie backendowym artefaktem

Definicja:

- `Stage` = jeden etap, ktory uzytkownik dodaje i widzi w drzewie.
- `Execution step` = jeden backendowy krok po kompilacji.

Przyklady:

- `Relax` = primitive stage,
- `Run` = primitive stage,
- `Eigensolve` = primitive stage,
- `Hysteresis Loop` = macro stage,
- `Field Sweep` = macro stage.

### 3.3. Drzewo jest lista sekwencji

Lewy panel juz sam w sobie powinien byc `Stage Sequence`.

Nie potrzebujemy osobnego carda `Stage Sequence` jako glownego authoring surface. To tylko duplikuje tree.

### 3.4. Ribbon jest context-sensitive

Klikam:

- `Study > Stages` -> ribbon `Study` pokazuje przyciski dodawania stage,
- `Stage 2 - Relax` -> ribbon `Study` pokazuje komendy dla relax stage,
- `Stage 3 - Eigensolve > Spectrum` -> ribbon pokazuje akcje i presety dla spectrum/output settings.

### 3.5. Kazdy node ma wlasny inspector

Nie ma juz jednego wielkiego `StudyPanel`.

Ma byc registry:

- `node_kind -> inspector component`,
- `node_kind -> ribbon groups`,
- `node_kind -> validation summary`,
- `node_kind -> status badges`.

### 3.6. `Backend Configuration` nie moze byc wszedzie

To trafia tylko do:

- `Study > Defaults > Runtime & Backend`,
- opcjonalnie `Stage N > Solver Override`.

Nie powinno sie pokazywac jako stale globalne cardy niezaleznie od wybranego study noda.

```text
Tree           = what exists
Ribbon         = what you can do now
Inspector      = what selected node means and how to edit it
Bottom panel   = what happened / what is running
Viewport       = what you see
```

To musi byc zachowane konsekwentnie.

## 5. Kanoniczny Model Danych

### 5.1. Docelowy podzial

Potrzebujemy 3 warstw:

1. `StudyDocumentV2`
   - user-facing authoring source of truth

2. `CompiledExecutionPlan`
   - wynik materializacji do backendowych primitive steps

3. `RuntimeExecutionState`
   - zywy stan wykonywania, progress, telemetry, status per stage

### 5.2. `StudyDocumentV2`

```ts
type StudyDocumentV2 = {
  version: "study.v2";
  defaults: StudyDefaults;
  stages: StudyStageNode[];
};

type StudyDefaults = {
  runtime: RuntimeDefaults;
  solver: SolverDefaults;
  outputs: OutputDefaults;
  validation_mode: "strict" | "warn";
};

type RuntimeDefaults = {
  backend: "auto" | "fdm" | "fem";
  device: "auto" | "cpu" | "gpu" | "cuda:0";
  precision: "auto" | "single" | "double";
  execution_mode: "strict" | "allow_fallback";
};

type SolverDefaults = {
  gamma: number | null;
  integrator: "auto" | "heun" | "rk4" | "rk23" | "rk45" | "abm3";
  timestep_mode: "auto" | "fixed" | "adaptive";
  fixed_dt: number | null;
  adaptive: AdaptiveTimestepConfig | null;
  precession_mode: "auto" | "on" | "off";
};

type AdaptiveTimestepConfig = {
  atol: number;
  rtol: number;
  dt_initial: number | null;
  dt_min: number;
  dt_max: number | null;
  safety: number;
  growth_limit: number;
  shrink_limit: number;
  max_spin_rotation: number | null;
  norm_tolerance: number | null;
};
```

### 5.3. Wspolne pola stage

Kazdy stage powinien miec ten sam wspolny naglowek:

```ts
type StageCommon = {
  id: string;
  kind: StudyStageKind;
  name: string;
  enabled: boolean;
  notes: string | null;
  source: "ui" | "python" | "imported";
  solver_override: Partial<SolverDefaults> | null;
  runtime_override: Partial<RuntimeDefaults> | null;
  outputs_override: Partial<OutputDefaults> | null;
  state_input: StageStateInput | null;
  state_output: StageStateOutput | null;
};
```

### 5.4. Stage kinds v1

Pierwsza fala powinna objac:

- `run`
- `relax`
- `eigensolve`
- `hysteresis_loop`

Druga fala:

- `field_sweep`
- `current_sweep`
- `load_state`
- `save_state`
- `export`
- `parameter_sweep`

### 5.5. Primitive stages

```ts
type RunStage = StageCommon & {
  kind: "run";
  config: {
    until_seconds: number;
    sample_every: number | null;
    autosave_state_every: number | null;
  };
};

type RelaxStage = StageCommon & {
  kind: "relax";
  config: {
    algorithm: "llg_overdamped" | "projected_gradient_bb" | "nonlinear_cg" | "tangent_plane_implicit";
    torque_tolerance: number;
    energy_tolerance: number | null;
    max_steps: number;
    relax_alpha: number | null;
  };
};

type EigensolveStage = StageCommon & {
  kind: "eigensolve";
  config: {
    count: number;
    target: "lowest" | "nearest";
    target_frequency: number | null;
    include_demag: boolean;
    equilibrium_source: "provided" | "relax" | "artifact";
    equilibrium_artifact: string | null;
    normalization: "unit_l2" | "unit_max_amplitude";
    damping_policy: "ignore" | "include";
    k_vector: [number, number, number] | null;
    spin_wave_bc: string | Record<string, unknown>;
  };
};
```

### 5.6. Macro stages

```ts
type HysteresisLoopStage = StageCommon & {
  kind: "hysteresis_loop";
  config: {
    quantity: "b_ext" | "current";
    axis: [number, number, number];
    start: number;
    stop: number;
    steps: number;
    spacing: "linear" | "custom";
    custom_values: number[] | null;
    settle_stage: "relax" | "run";
    settle_relax: RelaxStage["config"] | null;
    settle_run: RunStage["config"] | null;
    save_point_state: boolean;
  };
};
```

To jest bardzo wazne:

- `Hysteresis Loop` ma byc jednym user-facing stage,
- ale moze materializowac sie do wielu backendowych krokow.

### 5.7. Macierz modulow per stage

| Stage kind | Moduly w ribbonie | Sekcje inspectora |
|---|---|---|
| `run` | Time, Solver, Outputs | Overview, Solver, Time Range, Outputs |
| `relax` | Relax, Solver, Outputs | Overview, Solver, Stop Criteria, Outputs |
| `eigensolve` | Spectrum, Equilibrium, Advanced | Overview, Solver, Equilibrium, Operator & Spectrum, Outputs |
| `hysteresis_loop` | Sweep, Per Point Action, Preview | Overview, Sweep Definition, Settle Stage, Outputs, Materialized Preview |

### 5.8. Najwazniejsza zasada kontraktu

Nie trzymamy juz osobno:

- `study_pipeline`,
- `flat stages`,
- luźnych solver cards.

Trzymamy:

- `StudyDocumentV2` jako authoring,
- `CompiledExecutionPlan` jako output kompilacji.

`flat stages` pozostaje tylko kompatybilnym backend artifactem.

## 6. Python = GUI, 1:1

### 6.1. Zasada

Kazdy stage kind musi miec:

1. ten sam zestaw pol w Pythonie,
2. ten sam zestaw pol w GUI,
3. ta sama walidacje,
4. te same defaulty,
5. ten sam serializer do scene document / script export.

### 6.2. Docelowe API Python

```python
study = fm.study(
    defaults=fm.StudyDefaults(
        backend="fem",
        precision="double",
        solver=fm.SolverDefaults(
            integrator="rk45",
            timestep_mode="adaptive",
            adaptive=fm.AdaptiveTimestep(
                atol=1e-6,
                rtol=1e-3,
                dt_initial=1e-15,
                dt_min=1e-18,
                dt_max=1e-10,
            ),
        ),
    )
)

study.add_relax(
    name="Relax equilibrium",
    algorithm="projected_gradient_bb",
    torque_tolerance=1e-6,
    max_steps=100_000,
)

study.add_run(
    name="Transient ringdown",
    until_seconds=5e-9,
)

study.add_eigensolve(
    name="Linear modes near equilibrium",
    count=20,
    target="lowest",
    include_demag=True,
    equilibrium_source="relax",
)

study.add_hysteresis_loop(
    name="Major loop Bz",
    quantity="b_ext",
    axis=(0, 0, 1),
    start=-0.25,
    stop=0.25,
    steps=101,
    settle_stage="relax",
)

fm.execute(study)
```

### 6.3. Kompatybilnosc wsteczna

Stare API nadal moze istniec jako sugar:

```python
fm.relax(...)
fm.run(...)
fm.eigenmodes(...)
```

ale wewnetrznie powinno budowac:

- jedno-stage albo multi-stage `StudyDocumentV2`,
- nie globalny, rozproszony stan `_state + _captured_stages`.

### 6.4. Jedno zrodlo prawdy dla schema

Rekomendacja:

- schema authoring ma zyc w `crates/fullmag-authoring` jako `StudyDocumentV2`,
- Rust eksportuje maszynowy schema / JSON contract,
- TypeScript bierze z niego typy i form metadata,
- Python ma cienkie dataclasses / buildery mapowane 1:1.

Nie odwrotnie.

Powod:

- Rust juz jest warstwa integracyjna dla authoring + runtime,
- frontend i Python sa klientami tego kontraktu.

```text
   |- Stage 1 - Relax
   |- Stage 2 - Run
   `- Stage 3 - Eigensolve
```

Zastosowanie:

- znajdz rownowage,
- policz transient,
- policz mody dla stanu rownowagi.

### 10.2. Relax -> Hysteresis Loop

```text
Study
`- Stages
   |- Stage 1 - Relax
   `- Stage 2 - Hysteresis Loop
      |- Sweep Definition
      |- Settle Stage
      `- Materialized Preview
```

Zastosowanie:

- major loop / minor loop,
- pomiar M(H),
- zapisywanie stanu w punktach.

### 10.3. Relax -> Eigensolve -> Run

```text
Study
`- Stages
   |- Stage 1 - Relax
   |- Stage 2 - Eigensolve
   `- Stage 3 - Run
```

To jest wazne, bo drzewo ma byc neutralne wobec kolejnosci. Uzytkownik ustawia etapy swobodnie, a walidacja ma tylko sygnalizowac sens lub ryzyko.

## 11. Materializacja I Walidacja

### 11.1. Regula

UI authoruje `StudyDocumentV2`.

Backend wykonuje `CompiledExecutionPlan`.

### 11.2. Materialization Preview

Kazdy study document musi miec panel:

- `Validate Study`,
- `Preview Materialization`.

Preview pokazuje:

- stage -> execution steps map,
- warnings,
- implicit defaults,
- backend capability issues,
- niejednoznaczne zaleznosci.

### 11.3. Przyklady walidacji

- `Eigensolve` bez poprzedniego `Relax` i z `equilibrium_source="relax"` -> error.
- `Run` bez `until_seconds` -> error.
- `Relax` z `max_steps <= 0` -> error.
- `Hysteresis Loop` z `steps < 2` -> error.
- adaptive timestep na integratorze bez adaptacji -> error.
- `dt_fixed` i `adaptive` jednoczesnie -> error.

### 11.4. Capability coverage

`Capability Coverage` nie powinno byc ogolnym cardem w kazdym view.

Powinno byc:

- pod `Study > Defaults > Runtime & Backend`,
- oraz jako lokalne warning badges na stage, gdy stage wymaga czegos nieobslugiwanego.

### 12.1. Bottom panel powinien miec osobne progress bars

```text
Bottom Runtime Strip

Study Progress     [##--------] 1 / 4 stages
Current Stage      Relax equilibrium
Stage Progress     [#######---] 73%
Solver Detail      step 1980 / 5000, dt = 1e-15, max|dm/dt| = ...
```

### 12.2. Dlaczego to zalezy od nowego modelu stage

Bez prawdziwych stage node'ow nie da sie dobrze pokazac:

- ktory stage jest aktywny,
- ile zostalo do konca sekwencji,
- jaki stage wygenerowal jakie artifacts,
- co jest meshingiem, a co study execution.

Nowy model stage daje:

- jasny `overall progress`,
- jasny `current stage`,
- jawne `stage execution map`,
- jawne `mesh -> solve -> outputs` rozdzielenie.

## 15. Kryteria Akceptacji

Refaktor mozna uznac za udany dopiero gdy:

1. Klikniecie roznych study node'ow pokazuje rozne, sensowne inspectory.
2. Klikniecie `Study > Stages` zmienia ribbon na `Study` i pokazuje stage modules.
3. Dodanie stage z ribbonu od razu dodaje nowy node do drzewa z auto numeracja.
4. `Run`, `Relax`, `Eigensolve` i `Hysteresis Loop` maja jawne, kompletne parametry.
5. Python i GUI round-tripuja ten sam kontrakt bez gubienia pol.
6. `Backend Configuration` istnieje tylko tam, gdzie semantycznie ma sens.
7. Uzytkownik moze zbudowac cala sekwencje badawcza bez edycji surowego Pythona.
8. Dolny panel umie pokazac progress osobno dla meshu i osobno dla stage.
9. `flat stages` sa widoczne tylko jako preview / compiled artifact.

## 16. Najwazniejsza Konkluzja

Dzisiejszy problem nie polega na tym, ze brakuje kilku przyciskow.

Problem polega na tym, ze:

- drzewo,
- ribbon,
- inspector,
- authoring model,
- oraz backend materialization

nie opisuja tego samego obiektu.

Dopiero po wprowadzeniu jednego `StudyDocumentV2` i prawdziwych, node-specific stage views `Study` zacznie dzialac jak COMSOL, a nie jak panel z kilkoma sekcjami obok siebie.
