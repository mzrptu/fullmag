## 1. Zasady Prowadzenia Refaktoru

### 1.1. Zasada glowna

Najpierw zamrazamy kontrakt danych i semantyke `Study`, potem przebudowujemy tree/ribbon/inspector, a dopiero na koncu usuwamy stare panele i warstwy kompatybilnosci.

### 1.2. Zasada source-of-truth

Docelowym authoring source-of-truth ma byc `StudyDocumentV2`.

Do czasu konca migracji wolno utrzymywac:

- legacy `flat stages`,
- legacy `study_pipeline`,
- legacy `StudyPanel`,

ale tylko jako warstwy przejsciowe, nie jako miejsce dalszego rozwoju.

### 1.3. Zasada kompatybilnosci

W trakcie rollout:

- stare skrypty Python musza dalej sie otwierac,
- stare sceny musza dalej sie importowac,
- backend nadal moze dostawac compiled `flat stages`,
- GUI nie moze wymagac big-bang migracji danych.

### 1.4. Zasada odbioru

Kazdy etap jest zamkniety dopiero wtedy, gdy:

- kod jest scalony,
- przechodzi swoja metode weryfikacji,
- nie zostawia ukrytej duplikacji source-of-truth,
- nastepny etap moze na nim bezpiecznie polegac.

## 3. Etap 1 - Freeze Scope I Nazewnictwa

### Cel

Zamrozic slownik i granice systemu przed pisaniem kodu, zeby frontend, Rust i Python nie rozjechaly sie w nazewnictwie.

### Zakres

- ustalic finalne nazwy bytow,
- opisac relacje `Stage` vs `Execution step`,
- ustalic zakres v1 i v2,
- zamrozic acceptance criteria.

### Dotykane pliki / moduly

- `docs/reports/07042026/fullmag_study_stages_comsol_refactor_plan_2026-04-07.mdx`
- nowy dokument wdrozeniowy
- ewentualnie `docs/adr/*` jesli zapadnie decyzja o ADR

### Szczegolowy opis prac

1. Ustalic finalne nazwy:
   - `StudyDocumentV2`
   - `StudyDefaults`
   - `StudyStageNode`
   - `CompiledExecutionPlan`
   - `RuntimeExecutionState`
2. Ustalic stage kinds v1:
   - `run`
   - `relax`
   - `eigensolve`
   - `hysteresis_loop`
3. Ustalic, ze:
   - `Study` authoring zyje w `Model Builder`,
   - `Study` workspace sluzy glownie do runtime i monitoringu.
4. Zamrozic decyzje o tym, ze `flat stages` sa tylko compiled artifact.

### Ryzyka / uwagi

- bez tego etapu kolejne PR-y beda produkowaly rozne wersje tego samego modelu,
- szczegolnie ryzykowny jest rozjazd nazewnictwa `eigenmodes` vs `eigensolve`.

### Metoda weryfikacji

- manual doc review przez wlasciciela architektury,
- sprawdzenie, czy wszystkie nowe taski i PR-y uzywaja tych samych nazw,
- szybki grep po nowych diffach pod katem starych nazw, jezeli byly juz edyty.

### Kryterium zamkniecia

- istnieje jeden zatwierdzony slownik nazw,
- nie ma otwartych sporow typu `study_pipeline` vs `study_document`,
- wszystkie kolejne etapy odwoluja sie do tego samego modelu.

## 4. Etap 2 - Inwentaryzacja Zachowan I Fixture Baseline

### Cel

Zlapac obecne zachowanie systemu przed migracja, zeby po drodze nie zgubic semantyki, nawet jesli sama architektura sie zmieni.

### Zakres

- zapisac fixture current-state,
- zlapac aktualne round-tripy,
- zlapac obecne drzewo i obecna serializacje `stages`,
- przygotowac zestaw scen testowych.

### Dotykane pliki / moduly

- nowe fixture docs lub test fixtures
- `apps/web/lib/session/*`
- `packages/fullmag-py/src/fullmag/runtime/script_builder.py`
- `packages/fullmag-py/src/fullmag/runtime/scene_document.py`

### Szczegolowy opis prac

1. Przygotowac 4 wzorcowe scenariusze:
   - tylko `Run`,
   - tylko `Relax`,
   - `Relax -> Run`,
   - `Relax -> Eigensolve`.
2. Dla kazdego zapisac:
   - JSON scene export,
   - legacy `flat stages`,
   - export Python script,
   - screenshot lub opis obecnego tree.
3. Doliczyc piaty scenariusz docelowy:
   - `Relax -> Hysteresis Loop`,
   nawet jesli jeszcze nie jest obslugiwany end-to-end.
4. Zapisac baseline problemow:
   - identyczny `StudyPanel` dla wielu node'ow,
   - `Stage Sequence` dublowane,
   - globalny `Backend Configuration`.

### Ryzyka / uwagi

- bez baseline trudno bedzie odroznic swiadoma zmiane od regresji,
- fixture powinny byc male i czytelne, nie produkcyjne.

### Metoda weryfikacji

- manual: otwarcie fixture i porownanie exportow,
- technical: round-trip `scene -> builder -> scene` nie moze gubic pol juz istniejacych,
- jesli pojawia sie testy, powinny porownywac serialized outputs.

### Kryterium zamkniecia

- istnieje zestaw referencyjnych fixture,
- da sie na nich odtworzyc obecne zachowanie i przyszle porownania,
- kolejny etap moze bezpiecznie ruszac kontrakt danych.

### Zakres

- `apps/web/lib/study-builder/types.ts`

### Metoda weryfikacji

- testy fixture `StudyDocumentV2 -> compiled steps`,
- porownanie wyniku dla prostych przypadkow z legacy materialization,
- manual kontrola preview dla `Hysteresis Loop`.

### Kryterium zamkniecia

- `StudyDocumentV2` zawsze daje deterministyczny compiled plan,
- node map jest pelna i stabilna,
- macro stage ma jawny materialized preview.

## 10. Etap 8 - Validation Engine V2

### Cel

Dodac prawdziwa walidacje study authoringu, a nie tylko ciche domysly i fallbacki.

### Zakres

- structural validation,
- semantic validation,
- backend capability validation,
- warning/error/info diagnostics.

### Dotykane pliki / moduly

- `apps/web/lib/study-builder/validate.ts`
- `apps/web/lib/workspace/capability-contract.ts`
- `apps/web/lib/workspace/capability-audit.ts`
- `crates/fullmag-authoring/src/validation.rs`

### Szczegolowy opis prac

1. Dodac walidacje strukturalna:
   - wymagane pola,
   - poprawne enumy,
   - poprawne zakresy liczb.
2. Dodac walidacje semantyczna:
   - `eigensolve` z `equilibrium_source="relax"` wymaga poprzedniego relax stage,
   - `run` wymaga `until_seconds`,
   - `adaptive` nie moze wspolistniec z `fixed_dt`.
3. Dodac capability validation:
   - backend nie obsluguje danego operatora,
   - backend nie obsluguje danego stage kind,
   - fallback bedzie konieczny.
4. Wprowadzic severity:
   - `info`
   - `warning`
   - `error`

### Ryzyka / uwagi

- nie wolno mieszac validation error z runtime telemetry,
- validation ma byc authoring-time guidance.

### Metoda weryfikacji

- testy jednostkowe dla typowych bledow,
- manual w GUI: wymuszenie blednych konfiguracji i sprawdzenie komunikatow,
- porownanie z capability profile backendu.

### Kryterium zamkniecia

- nie da sie zbudowac ewidentnie blednej sekwencji bez jawnej diagnozy,
- capability coverage jest pokazywane w odpowiednich miejscach,
- validation wynik nie zalezy od przypadkowych defaultow UI.
