# FDM/FEM exchange consistency audit — 2026-03-24

## Scope

This audit checks three things against the current local implementation:

1. consistency between code and product/spec documentation,
2. consistency between code and physics notes,
3. cross-backend consistency of the exchange term in FDM and FEM under mesh refinement.

Artifacts generated by the new study:

- `.fullmag/studies/exchange_density_fdm_fem.csv`
- `.fullmag/studies/exchange_density_fdm_fem.svg`

Related source added during this audit:

- `crates/fullmag-engine/src/studies.rs`
- `crates/fullmag-engine/tests/exchange_density_study.rs`
- `crates/fullmag-engine/examples/exchange_density_study.rs`

## Executive verdict

### What is consistent

The **exchange term itself is now reasonably consistent** across:

- FDM implementation,
- FEM implementation,
- the core exchange physics notes,
- and the executable public slice.

This is true for the current narrow baseline:

- homogeneous material,
- free-surface Neumann exchange BC,
- FDM 6-point stencil,
- FEM P1 tetrahedra with lumped magnetic mass recovery.

The new refinement study shows that FDM and FEM move toward each other as density increases:

| Level | FDM cells/axis | FEM divisions/axis | Relative exchange-energy gap | Relative center-`H_ex` gap |
|------|----------------|--------------------|------------------------------|----------------------------|
| `5c/2d` | 5 | 2 | `0.1082` | `0.3996` |
| `9c/4d` | 9 | 4 | `0.0157` | `0.1619` |
| `13c/6d` | 13 | 6 | `0.0003` | `0.0813` |
| `17c/8d` | 17 | 8 | `0.0042` | `0.0477` |

This is not a proof of full asymptotic equivalence, but it is a strong sign that the two exchange realizations are semantically aligned and numerically compatible for smooth states.

### What is not fully consistent

The project is **not fully documentation-consistent yet**.

Main drifts:

1. `docs/specs/capability-matrix-v0.md` is stale for FEM demag.
2. `docs/specs/capability-matrix-v0.md` is stale for `backend="auto"` resolution.
3. `docs/specs/problem-ir-v0.md` still contains duplicated sections and partially stale wording around `DynamicsIR` / `SamplingIR` vs `StudyIR`.
4. demag implementation is still weaker than the stronger foundation docs `0420` / `0430` already describe.

So the exchange story is mostly coherent, but the broader magnetostatics story is still only partially aligned.

## 1. Exchange physics vs code

## 1.1 FDM exchange

Physics notes:

- `docs/physics/0200-llg-exchange-reference-engine.md`
- `docs/physics/0400-fdm-exchange-demag-zeeman.md`

Current implementation:

- `crates/fullmag-engine/src/lib.rs`

Observed implementation:

- exchange field uses the expected prefactor
  - `2 A / (mu0 Ms)`
- spatial operator is the expected 6-point second-difference stencil,
- free boundaries are implemented by plateau extension / clamped neighbor selection,
- this corresponds to the frozen homogeneous Neumann exchange BC.

Verdict:

- **consistent** with the current exchange physics notes.

## 1.2 FEM exchange

Physics note:

- `docs/physics/0410-fem-exchange-demag-zeeman-mfem-gpu.md`

Current implementation:

- `crates/fullmag-engine/src/fem.rs`

Observed implementation:

- elementwise P1 tetrahedral stiffness matrices are assembled,
- only magnetic elements contribute,
- nodal exchange field is recovered by dividing by lumped magnetic node volumes,
- this matches the documented bootstrap path
  - weak form `M_{mu0 Ms} h_ex = -K_A m`
  - approximated with a lumped-mass inverse.

Verdict:

- **consistent in the bootstrap/reference sense** with the FEM exchange note.
- It is not yet the final MFEM/libCEED/hypre GPU realization, but the semantics match.

## 2. FDM vs FEM exchange consistency study

## Method

A new shared study was added in `crates/fullmag-engine/src/studies.rs`.

It builds the same smooth analytic magnetization field on:

- a structured FDM cube,
- a structured tetrahedral FEM cube,

and compares, for increasing density:

- total exchange energy density,
- exchange field at the cube center.

Geometry used:

- cube of side `40 nm`

Density ladder:

- FDM: `5`, `9`, `13`, `17` cells per axis
- FEM: `2`, `4`, `6`, `8` tetrahedral subdivisions per axis

Generated outputs:

- CSV: `.fullmag/studies/exchange_density_fdm_fem.csv`
- SVG: `.fullmag/studies/exchange_density_fdm_fem.svg`

## Result

The backend gap shrinks with refinement.

Most useful signals:

- relative exchange-energy gap falls from `10.82%` to `0.42%`,
- relative center exchange-field gap falls from `39.96%` to `4.77%`.

Interpretation:

- coarse meshes still differ materially, as expected,
- exchange energy converges faster than pointwise center-field agreement,
- the current FEM and FDM exchange operators are sufficiently coherent to support comparative refinement studies.

## Regression test added

- `cargo +nightly test -p fullmag-engine --test exchange_density_study -- --nocapture`

The test currently asserts:

- the relative energy gap shrinks from the coarsest to the finest level,
- the relative center-field gap shrinks from the coarsest to the finest level,
- both finest-level gaps stay below a broad bootstrap acceptance threshold.

This is intentionally a **trend test**, not a publication-grade convergence proof.

## 3. Documentation drift found during the audit

## 3.1 Capability matrix is stale for FEM demag

Current doc state in `docs/specs/capability-matrix-v0.md`:

- `Demag` says FDM executable, FEM planned,
- FEM demag outputs say planned for FEM and executable only for FDM.

Current code state:

- `crates/fullmag-runner/src/fem_reference.rs` already handles:
  - `enable_demag`,
  - `H_demag`,
  - `E_demag`.

Verdict:

- **doc drift**. The matrix still understates current FEM bootstrap execution.

## 3.2 Capability matrix is stale for `backend="auto"`

Current doc state:

- `backend="auto"` resolves to `fdm` during bootstrap planning.

Current code state:

- `crates/fullmag-plan/src/lib.rs` has `resolve_auto_backend(problem)`,
- auto can resolve to FEM when only a FEM-executable path exists.

Verdict:

- **doc drift**.

## 3.3 `problem-ir-v0.md` is partially stale and duplicated

Current doc state:

- duplicated `GeometryIR` and `InitialMagnetizationIR` sections,
- wording still emphasizes top-level `DynamicsIR` / `SamplingIR` even though the implemented model already carries `StudyIR`.

Current code state:

- `crates/fullmag-ir/src/lib.rs` already uses `StudyIR`.

Verdict:

- **spec cleanup needed**.

## 4. Physics-document drift still open outside exchange

## 4.1 Demag remains the main unresolved seam

Current docs:

- `docs/physics/0420-fdm-dipolar-demag-foundations.md`
- `docs/physics/0430-fem-dipolar-demag-mfem-gpu-foundations.md`

These describe stronger, more final target models:

- FDM: cell-averaged tensor / Newell-like convolution semantics,
- FEM: better open-boundary magnetostatics path beyond the bootstrap air-box/Robin reference.

Current code:

- FDM still uses the spectral projection implementation in `crates/fullmag-engine/src/lib.rs`,
- FEM demag is still the bootstrap dense/Robin reference path in `crates/fullmag-engine/src/fem.rs`.

Verdict:

- exchange is aligned,
- demag is still only **bootstrap-aligned**, not yet fully aligned with the stronger foundation docs.

## 5. Conclusion

### Current honest status

The current project state is:

- **exchange-consistent across docs, physics, FDM, and FEM**, within the current bootstrap assumptions,
- **good enough to run meaningful FDM/FEM exchange comparison studies**,
- **not yet fully documentation-consistent for demag and status reporting**.

### Recommended next documentation fixes

1. update `docs/specs/capability-matrix-v0.md` for FEM demag and `backend="auto"`,
2. clean and deduplicate `docs/specs/problem-ir-v0.md`,
3. explicitly separate in docs:
   - bootstrap demag implementations already in code,
   - stronger target demag operators still planned.

### Recommended next numerical step

After this exchange audit, the next serious numerical milestone should be:

- either a stronger FDM demag operator alignment with `0420`,
- or a more rigorous cross-backend convergence study extended to demag-enabled cases.
